# Why an ordinary restart looked like a ten-minute coordination failure

Instar has a watchdog for the small repeating task that decides which machine is currently in charge. The task runs every two minutes. If it truly stops running for long enough, the watchdog restarts its timer and reports that the multi-machine coordination layer briefly degraded.

The watchdog stored “when the task last ran” as a number. At startup that number was `0`, because the first task run had not happened yet. The watchdog accidentally interpreted `0` as a very old timestamp. Its own first check happens after about one minute, so an ordinary server boot could immediately produce a message claiming that the task had failed to advance for more than ten minutes and had been recovered. Nothing had stalled: the server was simply younger than its first scheduled task run.

The fix makes the missing state explicit. There are now three possible answers: `uninitialized` means no usable cadence evidence exists, `healthy` means positive cadence evidence is recent, and `stale` means that evidence is older than the allowed limit. Before the first task run, successfully installing the timer is itself a real baseline. That keeps normal startup quiet, but it also means a timer that disappears before its first callback cannot hide forever: after the full stale window, the aged arm event proves callback #1 was missed and authorizes one recovery. A bare absent, zero, or invalid timestamp with no arm evidence stays `uninitialized` and does nothing.

This matters beyond one watchdog. Monitoring code often uses a timestamp of zero to mean “not seen yet.” If absence is silently treated as failure, every restart can look like an incident. The shared classifier gives future cadence monitors the same typed three-state boundary instead of asking each author to remember the rule. The standards registry now states the rule under P20: missing evidence is unknown, not proof of the alarming state.

The safety posture is deliberately narrow. Genuine stale observations still re-arm the timer and produce the same degradation report. A young but slow live tick still keeps its existing protection. The change does not alter lease ownership, epochs, machine roles, acquisition, renewal, or any cross-machine data. It only prevents a recovery actuator from firing before it has evidence that something previously alive became stale.

Tests cover the helper’s exact boundary, the coordinator integration, and the real timer lifecycle. One end-to-end path advances through the watchdog’s first 60-second callback and proves it stays silent, then supplies a genuinely stale prior sample and proves recovery still happens. A second path deliberately cancels the main interval before callback #1, advances beyond the stale ceiling, and proves exactly one recovery/report occurs before normal ticking resumes. That catches both dangerous directions: crying wolf at boot and accidentally blinding the real watchdog.
