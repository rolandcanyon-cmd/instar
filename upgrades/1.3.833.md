## What Changed

The instar-dev workflow gains a **duplicate-build guard** (ACT-592): before an agent building instar spends a session implementing a spec, a deterministic check asks "is someone already building this?" — scanning a local sibling-session in-flight ledger (write-first-then-scan, PID-reuse-proof liveness, terminal cleanup), open PRs (bounded two-stage `gh` scan), and recently-merged commits — plus a calibrated fuzzy match that survives renames. On a `likely-duplicate`/`verify` verdict, a repo-local PreToolUse hook blocks the FIRST implementation write until the author records a disposition (`{decision, reason, acknowledgedEvidenceIds[]}`); the precommit gate carries a presence-only backstop and pre-push a `warnings[]`-only advisory. Everything is fail-open-total (any error → non-blocking, visible `check-errored` stub) and signal-not-authority (no layer ever gates on the verdict's meaning; the author is the authority). Kill switch `INSTAR_DUP_BUILD_CHECK=off`. Born from the 2026-07-12 incident where a full session rebuilt work already merged under a different tracking id — and live-verified: run on its own spec, the guard correctly surfaced the literal incident commits as evidence.

## What to Tell Your User

None — internal development-workflow tooling with no user-facing surface. This guard runs only inside the instar repo's own development gates (a repo-local hook + the commit/push gates); it never touches agent behavior, messaging, sessions, or any fleet-installed file. Nothing changes for any instar user or agent operator.

## Summary of New Capabilities

None user-facing. Internally: the instar development workflow now structurally prevents the "two parallel builds of the same feature under different tracking ids" failure — the first implementation write literally cannot happen until a detected overlap has been confronted and dispositioned, with a full audit trail (`logs/dup-build-check.jsonl`).
