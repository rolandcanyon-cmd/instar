# Side-Effects Review — multi-transport mesh bind/predicate fix (1.3.630 bind-inert)

**Spec:** rides the already-converged+approved `docs/specs/multi-transport-mesh-comms.md` (Layer 0.5 bind + advertiser, Decision 17). This is an implementation→spec **bug fix** — no design change, no re-convergence.

**What shipped wrong in 1.3.630 (caught in LIVE-VERIFY on the real Mini+Laptop pair, NOT by unit tests — they mocked config):**
- **BUG 1 (predicate):** the advertiser (`server.ts advertiseSelfMeshEndpointsNow`) and the 0.0.0.0 bind (`AgentServer.ts`) gated on `config.multiMachine?.enabled` — a near-dead field a REAL multi-machine agent never sets (multi-machine is activated by `identityManager.hasIdentity()`). So both shipped **inert**: meshEndpoints empty, server stayed on 127.0.0.1.
- **BUG 2 (bind host-default):** even forcing the flag true, the server stayed 127.0.0.1 because `loadConfig` (Config.ts) ALWAYS defaults `host` to `'127.0.0.1'` when unset, so `this.config.host || … || meshBindDefault` could never reach `meshBindDefault('0.0.0.0')`.

**The fix:**
- Gate advertiser + bind on the canonical signal `identityManager.hasIdentity()` (passed to AgentServer as `meshBindActive`, computed in server.ts from `coordinator.managers.identityManager.hasIdentity() && meshTransport.enabled !== false`).
- Extract bind-host resolution into a pure, exported, unit-tested `resolveMeshBindHost()` (`MeshUrlAdvertiser.ts`): a LOOPBACK `configHost` is treated as NON-explicit (so the mesh default applies); an explicit NON-loopback host still wins; `meshTransport.bindHost` is the loopback escape hatch.

1. **Over-block (rejects legitimate inputs it shouldn't):** None. A single-machine agent (no identity) still binds 127.0.0.1 exactly as before — `meshBindActive=false`. A user who explicitly set a non-loopback `host` still gets it (explicitHost wins). The only behavioral change is the intended one: a genuinely multi-machine agent with mesh enabled now binds 0.0.0.0 (as the approved spec specified).
2. **Under-block (failure modes still missed):** A user who EXPLICITLY sets `host: '127.0.0.1'` on a mesh agent intending loopback is now overridden to 0.0.0.0 (a loopback host is indistinguishable from `loadConfig`'s default). Mitigation: `meshTransport.bindHost: '127.0.0.1'` (escape hatch) or `meshTransport.enabled: false`. This is the deliberate, documented tradeoff — without it the feature is dead-on-arrival behind the host default. Single-machine and explicit-non-loopback cases are unaffected.
3. **Level-of-abstraction fit:** Correct layer. Bind-host resolution is a pure function next to the other mesh helpers (`computeSelfMeshEndpoints`, etc.); AgentServer just calls it. The multi-machine predicate is now sourced from the SAME place the rest of the codebase decides multi-machine-ness (`hasIdentity()` / `coordinator.enabled`), removing the divergent `multiMachine.enabled` predicate that caused the bug.
4. **Signal vs authority compliance:** N/A as a decision-gate — this is a bind-host computation + a feature-activation predicate, not a message/action gate. It holds no blocking authority over agent behavior; it only chooses a listen interface. The 0.0.0.0 exposure keeps every existing auth (machine-auth on /api/*, dashboard PIN, Bearer) and is strictly LESS reachable than the always-on public Cloudflare tunnel the agent already runs.
5. **Interactions:** `resolveMeshBindHost` is the single source of the bind host; no other listen path competes (the second `host` in AgentServer is a localhost self-fetch inside `driveDeliveredFollowMeEnrollments`, correctly 127.0.0.1, untouched). The advertiser predicate change aligns with the existing `coordinator.enabled` gate already used at AgentServer construction (`coordinator: coordinator.enabled ? … : undefined`). No double-fire, no shadowing.
6. **External surfaces:** YES — intended. On a multi-machine agent with mesh enabled, the HTTP server now listens on 0.0.0.0 (LAN + Tailscale interfaces) instead of 127.0.0.1. This is the entire point (peers could not reach the advertised TS/LAN endpoints on a loopback bind). Exposure is bounded: all routes keep their existing auth; it is strictly less than the public Cloudflare tunnel already running; single-machine agents are never newly exposed (no identity ⇒ 127.0.0.1).
7. **Multi-machine posture (Cross-Machine Coherence):** This IS the multi-machine feature. Machine-local by design per machine (each machine resolves its own bind host + advertises its own endpoints from its own identity); the advertised endpoints propagate to peers via the existing authenticated machine-registry sync (same path as `lastKnownUrl`), with the accept-ack responder-identity as the spoof defense (unchanged from the shipped design). `/pool` not showing endpoints is a known VIEW limitation (`listPoolMachines` returns only `{machineId,nickname,lastKnownUrl}`), not a sync gap — tracked, not a regression.
8. **Rollback cost:** Cheap. `meshTransport.enabled: false` → back to 127.0.0.1 (the gate's explicit opt-out). No data migration, no state repair. A bad release backs out by version pin. The pure function is covered by 7 regression unit tests (loopback-default→0.0.0.0, undefined→0.0.0.0, localhost/::1→0.0.0.0, single-machine→127, explicit-non-loopback wins, bindHost escape hatch, explicit-outranks-bindHost).

**Diagnostic nit (second-pass, non-blocking):** Config.ts's "binding non-loopback without authToken" WARNING keys on `config.host` (which stays `127.0.0.1` for a defaulted mesh agent), so it won't fire even though the server now actually binds `0.0.0.0`. This is diagnostic-only and not exposure: route auth is always enforced and `init` always mints `authToken = randomUUID()`, so "multi-machine agent without a token" is not a realizable state. Tracked as a follow-up to make that warning reflect the real bind host. <!-- tracked: topic-27515 mesh-authtoken-warning-followup -->

## Second-pass review

**Independent reviewer verdict: CONCUR.** Verified: (1) SECURITY — the bind change affects reachability only, never authorization; all routes keep route-layer auth; `init` always mints an authToken so a tokenless multi-machine agent is not realizable; a single-machine agent (no identity) stays 127.0.0.1, never newly exposed (both sides test-covered). (2) CORRECTNESS — `hasIdentity()` IS the canonical signal (`coordinator.start()` itself decides single-vs-multi on `!hasIdentity()`); `coordinator.managers` is a getter that always returns the instance regardless of `.enabled`, and `hasIdentity()` is a no-throw `fs.existsSync`, so the construction-site call cannot be undefined/throw; coordinator is created+started before server construction. (3) the loopback set `{127.0.0.1, localhost, ::1}` matches `loadConfig`'s exactly; explicit non-loopback (incl. a deliberate 0.0.0.0) is preserved — no single-machine regression. (4) 7 new tests cover both boundary sides + escape hatch + precedence. Raised one non-blocking diagnostic nit (the authToken warning above), recorded.
