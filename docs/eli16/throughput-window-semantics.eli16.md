# Throughput window semantics

The blocker-lifecycle API has two useful views of completed work, but they answer different time questions. The summary answers “how many completions happened in the requested rolling number of hours?” and defaults to 24 hours. The trend answers “how many completions happened across the requested rolling number of days, and how are the complete UTC-day buckets changing?” and defaults to seven days. Before this change, both surfaces named the factor `deliverable-completion` but neither result carried its window beside the count. A summary count of zero and a trend count of ten therefore looked like contradictory measurements even when all ten completions were simply older than the summary’s 24-hour window.

This change does not alter, copy, or reinterpret any completion event. It adds an explicit `window` object to the existing deliverable-completion factor on both reads. Summary says `rolling-hours` and gives the exact requested hours. Trend says `rolling-days`, gives the exact requested days, identifies UTC daily buckets, and says the current day is partial. Pool reads validate and preserve the same labels from peers. The ledger, delivery feed, counts, daily buckets, and observe-only authority posture stay unchanged.
