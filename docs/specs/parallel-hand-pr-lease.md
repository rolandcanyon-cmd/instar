# Parallel-Hand PR Lease — one hand owns a branch's pushes at a time

**Status:** round-3-reviewed draft (authored 2026-06-16; rewritten after convergence rounds 1, 2, and 3)
**Author:** Echo (instar-dev)
**Tracking:** P0/P5 dev-cycle smoothness (Justin 2026-06-15: "make the dev cycle smooth & streamlined… immediate priority"). Closes the **push-competition** cause of the 2026-06-15 PR #1183 merge thrash (the worktree-collision cause is a tracked follow-up, §9 — this spec does NOT claim to close it). <!-- tracked: topic-12476-worktree-lease -->
**Related:** [[auto-merge-is-the-merge-path]] (the treadmill cause — strict=false + native auto-merge — is already shipped: #1185, #1191). ResumeQueue lock/auto-heal lessons (`src/monitoring/ResumeQueue.ts`, FD1/FD2/FD4/FD5 from the 2026-06-15 stale-host-lock incident) are the inherited foundation, §3.3.

## ELI16 — When two of my own sessions work the same branch, they stomp each other; this makes one take a turn-based "lease" so the other stands down

The instar-dev agent can have more than one session running at once (multi-session autonomy, or an autonomous run plus the interactive session you're chatting in). On 2026-06-15 two of those sessions both decided to fix the *same* pull request (#1183) at the *same* time. Each force-pushed its own commit over the other's, and every push restarted the ~15-minute CI from scratch — the PR that should have merged in minutes took ~2 hours of the two hands fighting.

This feature gives each branch a **lease**: before a session pushes commits to a branch, it must hold that branch's lease. If another *live* session of mine already holds it, the second session **stands down** instead of pushing a competing commit. The lease expires on a timer and auto-heals if the holding session dies, so it can never wedge a branch permanently. It is enforced in CODE at the exact place the competing pushes actually happen — a pre-tool hook that runs *before* a session's `git push` command in its terminal (not in a checklist a session has to remember) — keys ownership on the stable conversation **topic** (so a session that restarts mid-work still recognizes its own lease), and only coordinates my *own* cooperating sessions — it is never authority over another person or agent, and a human action always wins. It does NOT solve the separate problem of two sessions editing the same files on disk before either pushes — that is a tracked follow-up. <!-- tracked: topic-12476-worktree-lease -->

## 1. Problem statement

### 1.1 The observed failure (2026-06-15, PR #1183)
Two sessions of the same agent ("two echo hands") independently drove PR #1183. Two distinct mechanisms thrashed it:
- **(a) Competing commit pushes** — the head moved 5+ times as each hand force-pushed its own rebuild/rebase over the other's (2b22354b4 → 223f1a04 → 448bfaf8d → 4738cdf5d → f52965bde → f649d1a97). Every push **reset CI** (~15 min), so the two hands actively *prevented* convergence. **This spec closes (a).**
- **(b) Shared-worktree edit bleed** — both edited the same worktree (`.worktrees/playwright-profile-registry`), so uncommitted edits from one appeared under the other. **This spec does NOT close (b)** — see §2 / §9.

Net cost of both: ~2 hours for a PR that auto-merged in seconds once a single hand was left to drive it.

### 1.2 Why the existing machinery does not cover (a)
- **strict=false + native auto-merge** (shipped) removes the *treadmill* (main moving under a green PR), nothing about two hands pushing competing commits.
- **ParallelActivityIndex / ParallelWorkSentinel** (`src/core/ParallelActivityIndex.ts`) is read-only *observability* (derives per-topic focus/tags from TopicIntent refs); holds no lock, gates nothing.
- **The green-pr-automerge `[HOLD: …]` marker** stops the *watcher* merging, not a second *session* pushing.
- **Threadline single-negotiator lease** is the right *shape* (one voice per conversation) but scoped to A2A conversations, not git.

No primitive says **"one session owns this branch's pushes right now."** That is the gap.

## 2. Goal & non-goals

**Goal:** prevent two of the agent's own concurrent sessions from pushing competing commits to the same branch, via a per-branch ownership lease enforced at the git-push code chokepoint, that the non-holder honors (stands down), with TTL + dead-holder auto-heal + an absolute max-hold ceiling so it can never wedge a branch.

**Non-goals (stated honestly):**
- **Does NOT solve the shared-worktree edit collision (§1.1b).** Two same-machine hands sharing one worktree can still cross-contaminate *uncommitted* edits before either reaches the push gate. The structural mitigation is the existing `instar worktree create` per-agent-isolated-worktree convention; a worktree-level lease is tracked in §9.
- **Does NOT prevent a cross-MACHINE collision in v1.** The collision unit (a shared GitHub branch/PR) is inherently cross-machine, but v1's lease state is machine-local. If two hands on DIFFERENT machines drive the same branch, v1 does not prevent the thrash — the same recurs. v1 makes `holderMachineId` load-bearing ONLY for the safety rule "never falsely declare a foreign-machine holder dead" (§3.5); a fully replicated cross-machine lease is §9. This residual risk is stated, not hidden.
- **NOT a security/authority boundary, and NOT a complete push interceptor.** It coordinates *cooperative* same-agent hands; never blocks a different principal; a human/operator action always wins (Signal-vs-Authority, §7). The PreToolUse Bash hook catches a *literal* `git push` in the command (the demonstrated #1183 shape, incl. `cd && push`, env-prefix, `git -C`, multiline); a push hidden in a called **script body**, a **git alias**, or **deliberate string obfuscation** is an **accepted residual evasion** (closing them would mean forbidding the agent from scripting git — unwarranted for a cooperative guard). instar's own internal/scripted pushes are caught by the SafeGitExecutor defense-in-depth layer (§3.4).
- **NOT a replacement** for `[HOLD:]` (operator hold) or the MergeRunner watcher.

## 3. Design

### 3.1 Lease unit — the branch/ref (the actual push surface), canonicalized
**Per branch/ref**, keyed `branch:<canonical-ref>`. Rationale (codex#3 R1): the git push surface IS the branch — pushes happen *before* a PR exists, so a PR-number key cannot guard a pre-PR collision. PR metadata (`prNumber`) is attached as **secondary** once the PR is created (for the `GET /pr-leases` read + the MergeRunner consult), never as the primary key.

**Canonical key derivation — delegate to git's OWN ref resolution, do NOT reimplement push semantics (codex#2 R2/R3/R4 — load-bearing).** Reimplementing `push.default`/`remote.pushDefault`/`branch.<x>.pushRemote`/`remote.<name>.push` by hand is a guaranteed mis-key source. Resolution is two-tier to keep the hot path local (codex#2 R4):
- **Fast path (no network):** an EXPLICIT refspec in the command (`origin foo`, `HEAD:foo`, `HEAD:refs/heads/foo`) is parsed directly; a no-refspec push resolves the destination locally via `git rev-parse --symbolic-full-name @{push}` (reads config, no remote contact). This covers the overwhelming common case with a sub-ms local `git` call.
- **Fallback (only when the fast path is ambiguous):** `git push --dry-run --porcelain <args…>` (git's own resolver; prints the exact `<local>:<remote-ref>` lines it WOULD push) — this DOES contact the remote, so it is reserved for the genuinely-ambiguous case and is timeout+fail-open bounded (below).
Determinism: same underlying ref → same key; distinct refs → distinct keys, because git itself resolves it. Edge handling:
- **Bounded timeout (round-4 LOW):** `git push --dry-run` contacts the remote, so it can HANG (not throw) on a slow/unreachable remote, an auth prompt, or a credential-helper stall — and a hang is NOT caught by the §3.4 own-crash try/catch (which only catches thrown errors), so a hung resolver would block the push (the very lockout class the own-crash guard prevents). The resolver invocation therefore carries a hard wall-clock timeout (a few seconds); on timeout → treat as **no derivable key → fail-OPEN** (same disposition as a resolver error). The resolver also runs with `GIT_TERMINAL_PROMPT=0` / non-interactive credential settings so it can never block on an auth prompt.
- Dry-run resolver errors/ambiguous, detached HEAD with no destination, or a push to a tag/non-`heads` ref → **no lease key derivable → fail-OPEN** (do not gate; not the branch-collision surface). Logged.
- A ref deletion (`--delete` / `:refs/heads/foo`) → no commit content → not gated.
Two pushes that target the same underlying `refs/heads/<name>` derive the same key regardless of surface form (`foo`, `HEAD:foo`, `HEAD:refs/heads/foo`, no-refspec-with-upstream). The dry-run resolver is the canonicalization (§6 tests assert it over `git -C`, `cd && push`, env-prefix, `command git push`, multiline, force, and no-refspec-with-various-push.default variants).

### 3.2 State model
`state/pr-hand-leases.json` (machine-local v1), **one file**, single-writer CAS, one record per branch:
```
{
  "branch:echo/playwright-profile-registry": {
    "holderTopicId": 12476,          // STABLE logical owner (survives respawn) — the identity
    "holderSessionId": "<tmux name | uuid>",  // liveness-probe handle ONLY, never the identity
    "holderMachineId": "m_…",        // load-bearing for the never-falsely-dead rule (§3.5)
    "prNumber": 1183,                // secondary metadata, set when the PR exists (may be null)
    "intent": "build|rework|merge",  // closed enum, default "build"
    "acquiredAt": <epoch ms>,
    "renewedAt": <epoch ms>,
    "ttlMs": 1800000,                // 30m dead-holder backstop; renewed by heartbeat
    "maxHoldMs": 5400000             // 90m absolute ceiling (§3.9) — independent of renewal
  }
}
```
- **Identity = `holderTopicId`** (B2). The topic is the stable "hand" across the constant respawns instar performs (CompactionSentinel, context-wedge, `/sessions/refresh`, resume-queue revival). A hand whose `holderTopicId` == its own topic ALWAYS treats the lease as "mine" (renew + proceed, never yield), regardless of session id.
- **Renewal interval = TTL/3 (10m).** Concurrent leases are bounded by `autonomousSessions.maxConcurrent` (default 5), so the single-file CAS + fsync cost is negligible (M5).
- Tombstones (§3.7) are pruned opportunistically on every CAS write (zero extra I/O).

### 3.3 Concurrency control — one process-wide lock, atomic CAS takeover
**One** process-wide lock guards `pr-hand-leases.json` (matching ResumeQueue's single `lockPath` — NOT per-branch `.lock` sidecars, which would grow unbounded; M3). The lock + persist discipline is **inherited from `src/monitoring/ResumeQueue.ts`**: `'wx'`/`O_EXCL` first-writer-wins with the EEXIST loser re-evaluating and never blind-overwriting (FD4), temp-file + atomic `rename` + fsync `persist()`, and the FD1/FD2/FD5 host-classification (§3.5). Crash between lock-acquire and write leaves a stale process-lock that the same dead-holder discipline (§3.5) heals. **NEW (not inherited — small, sound addition under that lock):** ResumeQueue mutates in-memory then persists the whole state; this store adds a per-RECORD compare-and-swap (write a record only if the on-disk record still matches the observed `(holderTopicId, acquiredAt)`), needed because one file holds many branch records. Because the read-compare-write runs inside the process-wide lock it is genuinely atomic; the new surface is why the two-healers-race and two-past-ceiling-hands unit tests (§6) are load-bearing.

**Takeover of a stale lease is an atomic CAS on the record** (B3), NOT delete-then-recreate: the healer writes its own record only if the on-disk record still matches the `(holderTopicId, acquiredAt)` it observed; a mismatch (another hand healed first) → the loser re-reads and yields. This closes the double-drive where two hands both heal the same stale lease.

### 3.4 Yield protocol (the non-holder's behavior), enforced at the real push chokepoint
**Enforcement lives in a PreToolUse Bash hook** (`pr-hand-lease-guard.js`, registered on the `Bash` tool; sibling to the existing `dangerous-command-guard.sh` which already pattern-matches `git push`/`-f`). This is the correction from round 2 (B1-REGRESSION): the #1183 thrash was two **Claude Code sessions** each running `git push --force` from their **Bash tool**, which never enters instar's TypeScript and so never reaches `src/core/SafeGitExecutor.ts` (instar's *internal* git wrapper — its only `push` callsites are internal managers like `nuke`/`SyncOrchestrator`, NOT agent Bash pushes). A gate in SafeGitExecutor would test green and protect nothing against the observed failure. The PreToolUse Bash hook IS where the competing pushes are — AND it has the session's **topic ambient** (injected via the session-start / `telegram-topic-context.sh` wiring), which a `static SafeGitExecutor.execSync(args, opts)` call does not (it carries no topic field), so it is also the only place the `holderTopicId`-match (B2) can be evaluated.

**Command detection (codex#1 R3) — match the full Bash input, like `dangerous-command-guard.sh`.** The hook inspects the entire `tool_input.command` string (not just the first token), so it catches `cd repo && git push`, `git push; …`, an env-prefix (`GIT_SSH=… git push`), `git -C <path> push`, `command git push`, and multiline scripts — exactly the shapes the existing `dangerous-command-guard.sh` already handles. The `-C <path>`/`cd` target sets the `cwd` for the dry-run resolver (§3.1).

**Chokepoint honesty (round-3 HIGH) — this is a COOPERATIVE guard, not a security boundary.** The hook reliably catches a *literal* `git push` in the command string (the demonstrated #1183 shape). It does NOT catch a push hidden inside a called script body (`bash deploy.sh` whose contents push), a git alias (`git deploy`→push, expanded inside git), or deliberate string obfuscation (`git pu"sh"`). Per §7 this is self-coordination among the agent's OWN cooperative hands, so these indirect paths are **accepted residual evasions, explicitly out of scope** — closing them would require forbidding the agent from scripting git, which is neither feasible nor warranted for a cooperative guard. Stated in §2 non-goals so a reader does not mistake the hook for complete coverage. (Defense-in-depth below catches instar's own internal scripted pushes.)

**Hook own-crash fail-OPEN (round-3 MEDIUM — the most consequential safety guarantee).** A PreToolUse hook that exits non-zero for ANY reason blocks the Bash command, so a crashing `pr-hand-lease-guard.js` would block EVERY `git push` on the agent — strictly worse than the thrash it fixes (the `hook-event-reporter.js` ESM/CJS lockout class). Therefore the hook wraps its ENTIRE body in a top-level try/catch: on ANY uncaught error (malformed ambient topic, throw in key derivation, partial-migration `require` failure, dry-run resolver crash) it exits **0 (ALLOW)** + logs a degradation. A broken lease guard must never block a push. Tested (§6) by inducing an internal exception → asserts exit-0/allow.

**Defense-in-depth (not primary):** `SafeGitExecutor.ts` MAY *additionally* carry the same lease check for instar's own internal (TypeScript-issued) pushes (cheap, correct, and it catches instar's own scripted pushes that the Bash hook's literal-match would miss) — but it is never the primary/only enforcement.

The hook fires on a `Bash` command whose parsed form is a `git push` with a derivable canonical branch key (§3.1). Steps:
1. **Read** the lease (single JSON read; no liveness probe on this path). The read relies on the atomic-`rename` write (§3.3) so a concurrent write is **never** observed as torn — only a genuinely damaged file is "corrupt." On unreadable/corrupt/absent → **fail-OPEN**: treat as no lease, allow the push, log a degradation (M10); a recurring fail-open on the same branch raises ONE attention item (M-B R2 — so a corrupt-file-driven bypass cannot run silently). A missing lease file must NEVER block all pushes (distinct from the liveness fail-closed below).
2. **No lease, or `holderTopicId` == my topic:** acquire/renew (allow the push). The topic-match is the respawn-survival guarantee (B2): a restarted same-topic session re-adopts its own lease and never yields to itself.
3. **Live foreign lease** (different topic; holder confirmed live per §3.5; within max-hold): **DENY the push** (hook exits non-zero). Surface the holder (topic/intent) on the audit + attention surface; the hand stands down to other work. This is the anti-thrash core. The denial names the holder's `intent` so a genuinely-distinct second-hand fix is **escalated as a SEPARATE commit/PR, never silently dropped** (Signal-vs-Authority reconciliation, §7).
4. **Stale foreign lease** (holder dead per §3.5, OR past `maxHoldMs`): **auto-heal** — atomic-CAS take over (§3.3), allow the push. If the liveness probe is itself UNAVAILABLE/erroring (not "holder uncertain" but "can't evaluate"), degrade to **TTL-only** takeover so a broken probe can't wedge the branch (B4) — made safe by the atomic-CAS takeover (§3.3).

### 3.5 Liveness & dead-holder detection (TTL-gated, host-aware)
- The probe runs **only when `now - renewedAt > ttlMs`** (M4) — never on a fresh lease, so the common push path does a single JSON read with no process spawn.
- The probe reads **SessionManager's in-memory running set** (no `tmux` exec; M4).
- A holder is "gone" when **its session is absent from the live set AND the TTL has expired** (Q5: AND, the safe direction). A live holder session keeps its lease past TTL (covers a >30m rework, M11 — the session-present signal IS the keepalive; TTL is only the dead-holder backstop).
- **Session-id key consistency (M-C R2):** `holderSessionId` is written at acquire and looked up at probe using the **same key the SessionManager running set is keyed by (the tmux session name)** — never a uuid at write and a name at lookup, which would make "absent" trivially always-true and wrongly kill a live holder at TTL. Tested: a live holder past TTL is found present.
- **Host-aware (M6, critical):** a holder on a **different `holderMachineId`** is NEVER judged dead from local session absence (a machine-B holder is trivially absent from machine-A's session set). A foreign-machine holder → fail-closed → treat as live → yield. This is the v1 cross-machine safety floor (mirrors ResumeQueue FD1/FD2 foreign-host conservatism); full cross-machine coordination is §9.
- **Cross-machine precedence — `maxHoldMs` OVERRIDES foreign-machine conservatism (codex#1 R2):** the two rules above ("foreign-machine never judged dead" + "can never wedge") would otherwise conflict — a machine-A holder that vanishes would have machine-B yield forever. Resolution: the absolute max-hold ceiling (§3.9) is the tiebreaker — a lease past `maxHoldMs` is force-releasable **even when the holder is on a foreign machine**, and that forced release raises ONE attention item naming the stale cross-machine holder (so the override is loud, not silent). Within the ceiling window a foreign-machine lease is honored (yield); past it, the ceiling wins. This makes "can never wedge" true even cross-machine (bounded at `maxHoldMs`), at the cost of a possible cross-machine double-drive in the rare window where machine A is actually still alive but unreachable past 90m — an explicitly accepted, attention-surfaced v1 residual (full cross-machine coordination is §9).
- **Sleep/wake (M11-adj):** re-validate against the live set immediately after a `SleepWakeDetector` event rather than trusting `renewedAt` deltas across a wall-clock jump.

### 3.6 Enforcement strength (frontloaded — hard-refuse at the push, self-coordination carve-out)
v1 **denies the push** at the PreToolUse Bash-hook chokepoint (§3.4) on a live foreign lease (Q2). This is the spec's decision, not a lean: the observed failure was a hand that pushed *anyway*, so a signal the hand can reason past is insufficient. The dryRun ladder (§5) provides the observe-before-bind safety, so a separate signal-only mode is redundant. Reconciled with Signal-vs-Authority (§7): the refuse is **self-coordination among the agent's own cooperative hands**, NOT brittle authority over a principal — and it must surface the holder + intent so a non-redundant fix is escalated, never silently lost.

### 3.7 Release & tombstones
Release on: successful merge, explicit abandon, or stale auto-heal. A released lease is **tombstoned with a terminal status** (`released`/`merged`/`abandoned`) for an explicit **≤5s** window (just past the re-grab race). A successor acquires THROUGH a tombstone immediately without yielding — a tombstone is NEVER a live holder to yield to (M8). Tombstones prune on the next CAS write.

### 3.8 MergeRunner interaction (frontloaded — same-machine soft-hold)
The green-pr-automerge watcher (`src/monitoring/MergeRunner.ts`) is NOT a competing committer (never pushes commits) and holds NO lease. It reads the lease **read-only** and treats a **live same-machine `intent:rework`** lease as a soft `[HOLD]` (reuses the existing HOLD-suppression path; does not merge). Bounded by lease TTL — a quiet/expired rework lease NEVER stalls auto-merge (M7). All other intents (`build`/`merge`) and any expired/absent lease do not block the watcher. Cross-machine: the existing PR-label `[HOLD:]` marker remains the machine-independent authoritative hold; the lease soft-hold is a same-machine optimization in v1 (the cross-machine watcher-vs-rework gap is part of §9).

### 3.9 Absolute max-hold ceiling (liveness-discriminated — codex#3 R3)
Independent of renewal, `maxHoldMs` (90m) bounds how long a lease holds a branch. But forced release **discriminates on holder liveness** so a legitimately-long rework (a big rebuild + slow CI + delayed review can exceed 90m) is NOT torn out from under a live holder — that would reintroduce the exact push competition this spec prevents:
- **Holder confirmed DEAD/unreachable past `maxHoldMs`** (dead same-machine session, OR a foreign-machine holder we cannot verify — §3.5 precedence): **force-release via the §3.3 atomic-CAS takeover** (never a bare delete; the CAS loser re-reads and yields), + ONE attention item. This kills the "stale holder wedges forever" failure.
- **Holder confirmed LIVE same-machine past `maxHoldMs`**: do NOT seize. **Deny the second hand's push and escalate ONE attention item** ("hand on topic N has held branch X for >90m; still alive — long rework, or wedged? operator may intervene"). The live holder keeps its lease; the human decides. This prevents both the wedge (the holder is alive and progressing) AND the thrash (the second hand does not seize a live holder's branch). A genuinely-stalled-but-alive holder is then an operator decision, surfaced — not an automatic mid-rework seizure.

This refines M1: the ceiling's job is "never let a DEAD holder wedge forever, and never let a LIVE holder be silently invisible past 90m" — not "always seize at 90m."

## 5. Dev-gating & rollout (graduated, bounded soak)
Ships **dark** on the fleet, **live-in-dryRun on the dev agent**. `dryRun:true` provides **ZERO collision protection** — both hands still push (M13): the soak observes decision QUALITY, it does not yet protect. The LIVE hand still **WRITES + renews its lease** during dryRun (acquisition is NOT dryRun-gated; only the yield-DECISION is), so a peer can see it and the audit captures real would-yield decisions. **A dryRun-written lease carries a `dryRun:true` marker, and ALL non-acquisition readers MUST ignore it (codex#4 R3) — in particular the MergeRunner soft-hold (§3.8) NEVER acts on a dryRun lease** — so dryRun has zero BEHAVIORAL effect beyond the audit (it cannot perturb auto-merge or another reader). Tested: a dryRun lease does not trigger the MergeRunner soft-hold. The soak is **bounded**: if `logs/pr-lease-decisions.jsonl` shows ≥N would-yields on the same live branch, surface ONE attention item recommending the `dryRun:false` flip — do not soak open-ended while real thrash continues. ## 7. Signal-vs-authority
The lease is **coordination among the agent's own cooperative hands**, not authority over anyone else. **This is stated as a deliberate, bounded P2 EXCEPTION, not a claim that it naturally satisfies the principle (codex#3 R4):** P2 says a brittle deterministic detector must NOT hold blocking authority — and a Bash-command parser IS a brittle detector. The exception is admissible because its blast radius is tightly limited: (a) it blocks ONLY the agent's own cooperative hands, NEVER a different principal or an operator action; (b) it blocks ONLY a `git push` (one verb, not a general gate); (c) it FAILS OPEN on every uncertainty (corrupt state, unknown liveness past ceiling for a live holder is an escalation not a block, hook crash, resolver hang/timeout) so the brittle parser can only ever *slow a redundant own-hand*, never wedge or block real work; and (d) it surfaces holder/intent so a genuinely-distinct second-hand fix is escalated as a SEPARATE commit, never silently dropped. Outside these limits the lease is a signal, not authority. Fail-closed is narrow: "if unsure whether the other hand is alive, yield" (worst case the redundant hand waits) — bounded by the max-hold ceiling so it can never wedge.

## 8. Side-effects (for the artifact at build time)
Touches the git push path (the PreToolUse Bash hook; defense-in-depth in SafeGitExecutor) → highest care. Failure mode to avoid: a stale lease wrongly read as live wedging a branch — mitigated by TTL + host-aware liveness + the absolute max-hold ceiling (which overrides even foreign-machine conservatism, §3.5) + fail-open-on-corrupt + dryRun-first. Reversible: flag off → byte-identical-today. The state file is **ephemeral** (TTL-bounded, self-healing, reconstructable) → needs NO backup/restore. Exclusion mechanism (M-B R2/R3 — named target, imperative): the build **MUST add `state/pr-hand-leases.json` to `BackupManager.BLOCKED_PATH_PREFIXES`** (`src/core/BackupManager.ts`, the `path.normalize().startsWith()` denylist enforced at snapshot time, alongside the existing `.instar/secrets/` / `state/pending-inbound.` entries) **and add a unit test** (extending `tests/unit/backup-manager.test.ts`) asserting the path is excluded — so "restoring a stale lease as live" is structurally prevented, not merely asserted. Residual risks stated honestly: cross-machine collision (§2) and shared-worktree edit-bleed (§1.1b/§9) remain open after v1.

## 9. Alternatives considered (codex#6)
- **git-ref compare-and-swap lock** (a lock ref pushed with `--force-with-lease`): aptly matches the push surface and is cross-machine for free. codex (R1+R2) rightly pressed that the "network dependency" objection is weak — the push itself already requires the network, so the lock ref rides the same connection. The genuine v1 reasons to defer it: (a) it puts coordination state on the *shared canonical remote* (lock refs visible to every clone, and a crashed holder leaves a lock ref needing cleanup that a machine-local TTL handles for free); (b) it cannot be enforced at the same PreToolUse Bash chokepoint without an extra synchronous network round-trip on every push (latency on the hot path), whereas the local file read is sub-ms; (c) the observed incident was same-machine, so the local lease closes the demonstrated failure now with zero remote footprint. **This is an honest scope cut, not a dismissal:** git-ref CAS is named the PRIMARY candidate for the §9b cross-machine follow-up <!-- tracked: topic-12476-cross-machine --> (it solves the shared-branch race across machines, which the local lease provably cannot), and that §9b work must weigh making it the cross-machine enforcement layer rather than replicating the JSON lease.
- **SQLite `BEGIN IMMEDIATE`**: robust concurrency, but adds a DB dependency for a ≤5-record, same-machine, ephemeral coordination surface — overkill; instar's convention is file-based state.
- **`flock(2)`**: fine for the single-file lock, but provides no record/TTL/auto-heal semantics; we already have the ResumeQueue `'wx'` pattern that does. The chosen JSON + one-process-lock + atomic-CAS reuses proven in-repo machinery at the right scale.
