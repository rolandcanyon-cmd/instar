# Built-in job manifest missing required fields

**Status:** DRAFT → /spec-converge (abbreviated) → Justin approval → /instar-dev
**Author:** Echo · 2026-05-27 · found while deploying the mentor system to Echo's real server

## Problem statement

Every built-in agentmd job fails to load — `GET /jobs` returns `jobCount: 0` — and has since
~2026-05-20 (1200+ `[JobLoader] agentmd problem (manifest-invalid)` lines in Echo's server log).
The scheduled-jobs backbone (health-check, reflection-trigger, insight-harvest, the evolution
pipeline, the overseers, identity-review, memory-hygiene, …) is silently dead on any agent whose
per-slug manifests were (re)generated by the affected version.

### Root cause (confirmed)

`AgentMdJobLoader`'s `PerSlugManifest` interface **requires** `priority: JobPriority` and
`expectedDurationMinutes: number` (and reads `model`), and the loaded `JobDefinition` is built from
`manifest.priority` / `manifest.expectedDurationMinutes` / `manifest.model`
(`AgentMdJobLoader.ts` ~line 1023–1027). The manifest validator rejects a job when
`manifest.priority` is `undefined` (`"priority" must be one of critical, high, medium, low, got
"undefined"`).

But `InstallBuiltinJobs` — which **generates** those per-slug manifests at
`.instar/jobs/schedule/<slug>.json` on every `init`/update — writes a manifest object containing only
`{ slug, origin, schedule, enabled, execute, manifestVersion }` (`InstallBuiltinJobs.ts` ~line
161–168). It parses the `.md` frontmatter (which **does** carry `priority`, `expectedDurationMinutes`,
`model`) and copies `schedule` + `enabled` from it, but **not** `priority` / `expectedDurationMinutes`
/ `model`. So every generated manifest is missing fields the loader requires → every built-in job is
rejected.

This is a pure producer↔validator mismatch: the producer dropped fields the consumer requires.

## Two producers are affected (not one)

Convergence (adversarial F1) found a SECOND manifest producer with the identical defect:
`src/commands/jobMigrate.ts` `writeManifest()` (~line 145) builds the same `{slug, origin, schedule,
enabled, execute, manifestVersion}` object, dropping `priority`/`expectedDurationMinutes`/`model`. It
runs for both `origin:'instar'` and `origin:'user'` jobs. **User jobs migrated through this path are
permanently dead** — `InstallBuiltinJobs` never regenerates user manifests, so the self-heal below
does NOT reach them. Both producers must be fixed.

## Proposed design

**Root-cause fix — one shared, typed manifest constructor (Structure > Willpower).** The deeper cause
(adversarial F6) is that two hand-rolled producers each build the manifest object literal inline, with
no compile-time link to the `PerSlugManifest` contract the loader requires — so one drifted silently.
Extract a single `buildPerSlugManifest(args): PerSlugManifest` helper, **typed to return
`PerSlugManifest`**, and use it in BOTH `InstallBuiltinJobs` and `jobMigrate`. Because the return type
is the loader's required interface, TypeScript fails the build if a required field is dropped again —
converting this bug class from "runtime, fleet-wide, 1200 log lines" to "`tsc` error." The helper
copies, from the parsed frontmatter / source entry:
- `priority: frontmatter.priority` (string enum, e.g. `"low"` — the manifest is read via plain
  `JSON.parse` with NO coercion, so the validator does string-equality against `VALID_PRIORITIES`;
  a plain string is correct)
- `model: frontmatter.model` (optional `ModelTier` string; set only when present)
- `expectedDurationMinutes`: coerce `Number(...)` ONLY when the value is present and
  `Number.isFinite` — otherwise OMIT the key entirely (never write `NaN`/`null`). When omitted, the
  loader correctly rejects that single job (fail-loud), which never happens for shipped templates
  (all carry a positive value) but is the right behavior for a malformed one (adversarial F4).
- pass-throughs the loader reads from the manifest but that were also being dropped — `tags`,
  `unrestrictedTools`, `gate`, `telegramNotify` (integration LOW / Open-Q2): `manifestToJobDefinition`
  sources these from the MANIFEST, not the body frontmatter, so omitting them silently dropped e.g.
  `health-check`'s `unrestrictedTools: true`. Same root cause, same copy — include them now.

**Producer-side self-check (replaces the rejected consumer-leniency layer).** After building each
manifest, run it through the loader's exported `validateManifest()`; on failure, push a LOUD
`report.errors` entry at generation time rather than writing a manifest that will be rejected at load.
This guarantees a producer that drops a required field fails loudly at install/migrate time (the
correct layer), without weakening the consumer.

**Consumer leniency: REJECTED.** All three reviewers converged: silently defaulting a missing
`priority` to `medium` (the originally-floated defense layer) INVERTS the lesson — it converts a
loud, diagnosable failure (jobCount=0, 1200 visible log lines) into a silent one (every job loads
mis-prioritized; a `critical` health-check silently demoted under load shedding). The loud rejection
IS the correct behavior and is what surfaced this bug. `validateManifest` is a full-context
correctness gate with legitimate blocking authority (feedback_signal_vs_authority); we do not loosen
it. The always-overwrite self-heal (below) means there is no durable population of legacy partial
manifests to be lenient toward anyway.

**Self-heal of already-broken manifests (migration parity).** No separate `PostUpdateMigrator` entry
is needed: `PostUpdateMigrator.migrateBuiltinJobs()` (`PostUpdateMigrator.ts:211`) runs
`installBuiltinJobs` UNCONDITIONALLY on every update, and `installBuiltinJobs` always-overwrites each
manifest (`InstallBuiltinJobs.ts:173`, preserving only `enabled` + `disabledAtBodyHash`). So a fixed
producer stamps the missing fields onto every broken built-in manifest on the next update — the same
always-overwrite idiom the Migration Parity Standard uses for built-in hooks. (User manifests are
healed by fixing `jobMigrate`'s producer, which re-runs on the legacy-jobs migration path.)

**Lockfile unaffected.** The signed `instar.lock.json` hashes the `.md` body + frontmatter
(`AgentMdLockFile.ts`), NOT the manifest JSON — so adding manifest fields needs no lockfile regen/re-sign.

## Decision points touched

The loader's `validateManifest` is a correctness gate; this change does NOT loosen it (consumer
leniency rejected). It strengthens the PRODUCERS to honor the contract + adds a producer-side
loud self-check. No runtime agent-behavior gate is added or relaxed.

## Operator-customization note (adversarial F3)

Regeneration preserves only `enabled` + `disabledAtBodyHash`; all other built-in manifest fields
(now including priority/schedule/duration/model) are instar-owned and reset on update. This widens
the existing `schedule`-reset behavior. Policy (stated explicitly): built-in job manifest fields are
instar-owned; an operator who needs a different schedule/priority clones the job to a `user` slug.

## Testing plan (meets the bug-fix evidence bar — reproduce, then prove gone)

- **Pin-the-bug (round-trip) unit:** the strongest assertion — generate a manifest via the producer
  and round-trip it through the loader's `validateManifest()`, asserting it PASSES (stronger than
  checking individual fields; an idempotent snapshot of the broken shape is what let this ship — so
  we validate against the consumer, not a snapshot). Pair it: a manifest built the OLD way (missing
  the fields) is asserted to FAIL `validateManifest` → pins the bug.
- **Unit (both producers):** `InstallBuiltinJobs` AND `jobMigrate.writeManifest` each produce a
  manifest carrying `priority`, numeric `expectedDurationMinutes`, `model`, and the pass-throughs;
  add a `jobMigrate` test that a migrated USER job manifest carries them.
- **Unit (coercion edges, F4):** absent / non-numeric / empty-string `expectedDurationMinutes` →
  the key is omitted (never `NaN`/`null`); `Number("0")` path rejected by the loader (fail-loud).
- **Unit (self-heal):** running `installBuiltinJobs` over a pre-existing manifest that lacks the
  fields overwrites it with the complete manifest, preserving `enabled`.
- **Unit (typed-helper guard):** the shared `buildPerSlugManifest` returns `PerSlugManifest` so a
  dropped required field is a compile error (documented; enforced by `tsc`).
- **E2E ("jobs are alive"):** boot the real server through the production install path over the
  SHIPPED templates and assert `GET /jobs` `jobCount >= <number of shipped built-in slugs>` (NOT
  just `> 0` — a loose `>0` is satisfiable by the buggy state if one non-agentmd job loads) AND that
  the loader reports ZERO `manifest-invalid` problems for `origin:instar` slugs.
- **Live evidence (test-as-self, mandatory):** the bug is reproducing on Echo's real server NOW
  (`jobCount: 0`, 1200+ `manifest-invalid` log lines). After building, run the fixed producer against
  Echo's real `.instar/jobs/`, restart, and capture the before/after: `jobCount` 0 → ≥(shipped count)
  and no new `manifest-invalid` lines. This is the reproduce→fix→verify-gone evidence the bug-fix
  evidence bar requires, on a live system.

## Resolved open questions

1. **Producer-only + producer self-check; consumer leniency REJECTED** (fail-loud preserved).
2. **Include the pass-throughs** (`tags`/`unrestrictedTools`/`gate`/`telegramNotify`) — same root cause.
3. **Both producers fixed** via the shared typed helper (`InstallBuiltinJobs` + `jobMigrate`).
