# Duplicate-Build Guard — catch "someone is already building this" BEFORE the build is spent

## 0. Origin & traceability

Earned from a live incident (2026-07-12, topic 11960): a session built **ACT-562** (LLM-decision provenance wiring, PR #1460) to completion — full instar-dev ceremony, all three test tiers green — and only discovered *at merge time* that **PR #1458** ("LLM-Decision Quality Meter", ACT-1193/1194) had merged the same substrate to `main`. The two efforts carried different tracking IDs and ran in parallel, blind to each other. #1460 was closed as a duplicate; a full session of build was wasted. Tracks **ACT-592**. Parent principle: **Structure beats Willpower**.

**Round-1 correction (load-bearing).** The first draft claimed a "census point already `wired` on main" check "would have fired for ACT-562." Git history refutes it: `src/data/provenanceCoverage.ts` was *created inside* #1458's PR (`4502e4e9`), the tone-gate point went `wired` only in #1461 (`88da6009`). At ACT-562's build-start the census either did not exist on base `main` or the point was `pending`, so a "wired-on-main" test yields `clear`. The incident's defining feature is **parallel in-flight births, not finished-on-main state**. Primary signal = **concurrency**; primary catch = **structural and at build-start**.

**Round-2 correction (load-bearing).** Reviewers (internal + codex/GPT-5.5) proved v2's "structural early catch" was still enforced only at PRECOMMIT (commit time = after the whole build), and the build-start marker write was itself un-gated willpower — the Structure-vs-Willpower gap relocated one layer up. v3 adds a genuine **build-start chokepoint** (§3.4): a stop-hook that refuses to proceed into implementation on a `likely-duplicate` without a recorded disposition, so the wasted-build prevention is real, not asserted. Precommit is demoted to a presence-only backstop.

## 1. Problem statement (verified against live source)

The instar-dev flow has NO overlap/concurrency check:
- `skills/instar-dev/SKILL.md` Phase 2 ("Build location re-grounding") verifies the worktree is off *current* `JKHeadley/main` + records `git remote -v`/version — it checks base **freshness**, never **overlap**, and it is skill-prose the agent is trusted to execute (no structural hook).
- `skills/spec-converge/SKILL.md` reviews design quality — never "is someone already building this?".
- `scripts/instar-dev-precommit.js` (commit-time) and `scripts/pre-push-gate.js` (push-time) enforce trace/artifact/release discipline — neither checks overlap, and both fire only AFTER the build cost is paid.

Consequence: an agent (or a second same-box session) can spec → converge → build → test → open a PR for work *already in flight under a different tracking id*, discovering the collision only at merge.

## 2. Scope of THIS increment

A **concurrency-first duplicate-build detector** whose PRIMARY catch is **structural and at build-start**. In scope:
- A deterministic, fail-open, security-hardened check library (`scripts/lib/duplicate-build-check.mjs`).
- The **build-start structural gate** (§3.4): an instar-dev build-start step that runs the check + records a `duplicateBuildCheck` disposition, enforced by a stop-hook that refuses implementation on a `likely-duplicate` without a recorded disposition. This is the wasted-build-prevention teeth.
- A **precommit presence backstop** (`instar-dev-precommit.js` refuses a commit whose trace lacks a `duplicateBuildCheck` field — presence-only, never gating on the verdict's VALUE) and a **pre-push advisory** (`warnings[]` only, never `errors[]`).
- Graceful, **loud** degradation (a degraded scan → `verify`, never a silent `clear`).

Out of scope, each with its OWN durable tracked action (Close the Loop — NOT filed under this build's ACT-592): auto-abandon-on-overlap `<!-- tracked: ACT-594 -->`; a durable cross-machine in-flight registry `<!-- tracked: ACT-595 -->`.

## 3. Proposed design

### 3.1 Target extraction (with a bounded, deterministic fuzzy floor)
Derive a TARGET SET from the spec, each with a match strength:
- **Decision-point / census identities** (STRONG-exact): ids named in `## Decision points touched` or a census-style entry.
- **New exported symbols / substrate files** (STRONG-exact): matched against what open/recent PRs are ADDING (the incident signal), not only against `main`.
- **Feature-description fingerprint** (FUZZY floor): survives renames + a missing `## Decision points touched` section. **Deterministic algorithm (no embeddings): token-set Jaccard** over normalized (lowercased, stop-worded, stemmed) tokens of the spec title + problem-statement + touched-subsystem vs open-PR titles/bodies + recent merged-commit subjects. **Threshold is a frontloaded conservative constant (FD6), tuned toward recall** because a fuzzy-only hit only ever yields `verify` (the cheap verdict). **Bounds enforced BEFORE any similarity math (FD6/§3.2a): each PR body truncated to ≤8KB, each spec section to ≤4KB, candidate set bounded by the `gh --limit` page + the `-n 500` merge cap.** Token-set Jaccard is chosen over unbounded-window Levenshtein precisely to keep cost linear in capped token counts.
- **Touched src files** (WEAK): reuses the pre-push gate's existing `${remoteBranch}...HEAD` changed-file computation (`scripts/lib/pre-push-scope.mjs`), never a re-implemented diff.

**Stated accepted false-negatives (honest, at the ladder level):** (a) a renamed symbol whose spec ALSO rewrites the problem statement in different vocabulary can fall below the Jaccard threshold → `clear`; (b) two builds on DIFFERENT machines before either opens a PR are invisible to every source here. The compensating signals are the local sibling ledger (same-machine) and the open-PR/file-overlap layer (once either pushes); the cross-machine early window is the tracked follow-up `<!-- tracked: ACT-595 -->`.

### 3.2 The concurrency sources (the load-bearing signal) + `main` corroborator
1. **Local sibling in-flight ledger** — catches the incident geometry for same-agent-home concurrent sessions (two sessions sharing this agent's ledger root, neither PR open — the common single-operator case; the cross-agent-home / container residual is honestly scoped in the Multi-machine posture). Lifecycle + concurrency invariants (all MANDATORY):
   - **Write-FIRST-then-scan (TOCTOU fix):** at build-start append my marker `{agent, host, branch, specSlug, targets[], startedAt, pid, procStartToken, worktreePath}`, THEN read the whole ledger. If I see any OTHER live overlapping entry with an earlier `startedAt`, I lose the race → `likely-duplicate`; ties broken lexicographically by (pid, branch) so exactly one of two simultaneous builders yields. (Unit-tested for the simultaneous-append case.)
   - **Liveness is a conjunction (PID-reuse fix):** an entry is LIVE iff its pid is alive AND its `procStartToken` (process start-time stamp) matches the live process AND its `worktreePath` still exists. Only a *provably dead* marker is ignored; an old-but-still-live long build STILL fires (age alone never marks live work stale).
   - **Terminal lifecycle (unbounded-growth fix):** the marker is REMOVED at the terminal transition (commit success / abandon), mirroring `PendingInjectStore` ("recorded at spawn, cleared after"). A crashed-build orphan is swept at read-time compaction and at boot. Reads are bounded (compact past a size threshold); "ignored if stale" is the backstop, not the primary cleanup.
   - **Concurrent-read integrity (torn-line fix):** parse line-by-line; skip only an unparseable/torn line (never fail the whole scan to `clear`); a torn last line on a substrate-introducing spec raises `verify` (No Silent Degradation); an ENOENT on a sibling-worktree dir mid-scan is skipped, not fatal.
2. **Open PRs** — bounded, non-interactive, timed (§3.2a). ONE list call `gh pr list --state open --limit 100 --json number,title,headRefName,files,body` returns per-PR FILE lists + titles/bodies directly (no per-PR call needed for file-overlap OR the fuzzy fingerprint). For census/symbol IDENTITIES a PR ADDS (not derivable from file paths alone), a bounded SECOND stage `gh pr diff <n> --patch` runs ONLY on the ≤5 PRs already matched by file-overlap or title, each with a diff-size cap; on cap-exceed or timeout it falls back to file-overlap-only for that PR (a verify-worthy signal is never silently dropped — it degrades to `verify`, never `clear`). The whole two-stage scan stays within the ≤5s gh / ≤8s total budget or degrades loudly.
3. **Recently-merged lookback** — `git log origin/main --since=<max(build-start, 30d-floor)> --grep/-S -n 500` (argv-safe) — catches a PR that MERGED between this build's start and its push. **Pre-push MUST re-fetch `origin/main` and re-key the cache** (a HEAD move invalidates it) so this lookback actually runs at push time.

`main`-HEAD state overlap is a WEAK corroborator only (never a block source): "extends an existing wired point" is indistinguishable from "duplicates it" to a deterministic matcher (#1461 legitimately extended #1458's census).

### 3.2a Security hardening (git/gh AND the local ledger) — MANDATORY
Spec-derived strings, open-PR titles/bodies, AND ledger-derived values are **attacker-influenceable untrusted input** (a sibling session, or anyone with write access to the agent home, authors the ledger). The check MUST:
- Route every git call through `SafeGitExecutor.readSync` with an **argv array** (never a shell string); every `gh` call through `spawnSync('gh', argv)`.
- Validate every untrusted value (spec-, PR-, OR ledger-derived) BY TYPE for its argument position — a `ref` (branch/sha) against git's ref grammar, a `path` against a repo-relative pathspec, a `pr-number` as an integer, a `token`/symbol against a `[A-Za-z0-9_.:/-]`-class charset — and pass it AFTER a `--` end-of-options separator in the argv array. Type-validation (not a blanket leading-`-` reject, which would wrongly refuse valid branch names / slugs / paths) is the option-injection defense; a value that fails its type check is dropped from the scan, never shelled.
- Scope `git grep` to a fixed pathspec (census file + `src/**`) with a max-match cap AND a per-match line-length clamp (a long minified line is clamped identically to gh output before entering evidence).
- Treat gh/PR AND ledger output as untrusted: length-clamp, strip control/escape chars, delimit before any value enters `evidence`, a log, or the agent surface. Enforce the fuzzy byte caps (§3.1) BEFORE similarity math (resource-exhaustion / push-path DoS defense).
- **Ledger path safety:** "agent home" is precisely `~/.instar/agents/<agent>/` (the sandbox-safe root the worktree convention already jails worktrees to, under `.worktrees/`). The ledger lives at `<agent-home>/state/dup-build-inflight.jsonl`, created mode-0600, absence tolerated. Realpath-resolve the ledger path + each sibling-`worktreePath`; a `worktreePath` is accepted only if it resolves under an ALLOWED-ROOTS list = { the agent home's `.worktrees/`, plus any root in `worktree.allowedRoots` config } — reject symlinks and anything outside (defends against a planted marker redirecting a read/write). A malformed JSONL line is skipped (fail-open), never fatal.
- Run `gh` strictly non-interactive with a bounded timeout; on any non-zero exit degrade to local-only + a GENERIC note — NEVER echo gh stderr (token/org/repo internals). Emit repo-relative paths + PR numbers only.

### 3.3 Fail-open TOTALITY + the TOTAL verdict ladder (FD4)
The check is TOTAL and fail-open: any internal error/timeout/bad-parse/git/gh/ledger failure → a non-blocking verdict + **exit 0**. Per-subprocess timeouts (git ≤3s, gh ≤5s), total budget ≤8s; a single combined `git grep -e … -e …` over ≤20 capped targets; results cached keyed on `(specSlug, origin/main-HEAD-sha, pr-list-fetch-hash)` so build-start + pre-push share ONE computation — but pre-push re-fetches origin/main and re-keys so a HEAD move re-runs the scan.

The ladder is TOTAL over (cause × substrate?):
- `likely-duplicate` ⇔ a STRONG-exact target matches a **concurrency source** (live local sibling, open PR adding it, or a merge within lookback).
- `verify` ⇔ a fuzzy-only hit (cause `fuzzy`), OR a STRONG target only on `main`-state (cause `main-only`), OR any degraded scan on a **substrate-introducing** spec (cause `degraded`). **The `verify` cause is tagged** (`fuzzy` | `main-only` | `degraded`) in the verdict, the recorded disposition, and the audit trail — so an environmental-degradation `verify` is distinguishable from a real overlap-suspicion `verify` (alarm-fatigue defense). When more than one cause holds at once, the real-overlap causes (`fuzzy`, `main-only`) OUTRANK the environmental `degraded` tag (the point of tagging is to surface real suspicion over environment noise); `cause` may also be emitted as a set for the audit trail.
- `clear` ⇔ no overlap and all concurrency sources scanned successfully; **a degraded NON-substrate scan is `clear` with a `degraded` audit flag** (inspectable, not silent — the ladder cell is defined, not undefined).
- WEAK-only overlaps → silent `clear` + a quiet note, NEVER `verify` (precision over recall; the author is never trained to skim).

### 3.4 Two enforced moments — the build-START teeth + the precommit backstop (the load-bearing structure)
Per `docs/signal-vs-authority.md` the detector never auto-blocks; the author is the proceed/abandon authority. Structure > Willpower is delivered at TWO enforced moments with an HONEST division of what each prevents:
- **Build-START (prevents the wasted local compute).** The instar-dev build-start step (at Phase 2 / worktree-creation, BEFORE any implementation) runs the check and writes the `duplicateBuildCheck` record into the trace stub + the ledger marker. The teeth are a **`PreToolUse` hook** (`duplicate-build-start-gate.js`) that fires on the FIRST mutating write (`Write`/`Edit`/`MultiEdit`) to ANY tracked repo path inside an instar worktree EXCEPT the trace/log/state paths themselves (`src/**`, `tests/**`, `scripts/**`, `docs/**`, config, migrations, fixtures — the first costly implementation write may be to any of these, not just `src`), and BLOCKS that first write when the recorded verdict is `likely-duplicate`/`verify` and no disposition is yet recorded. Gating the first *implementation tool call* (not turn-exit) is deliberate: a `Stop`-event hook (the `/build` pattern) only gates turn-EXIT, so first-turn implementation would slip before it fires. **Disposition schema (so the gate is not a checkbox):** `duplicateBuildCheck: { verdict, cause, decision: "proceed"|"abandon", reason, acknowledgedEvidenceIds[] }`; a `likely-duplicate` proceed REQUIRES a non-empty `reason` AND at least one `acknowledgedEvidenceId` (the author must name the concrete overlap they judged non-duplicative). **Fail-open reconciled with precommit:** a hook error / absent-or-errored verdict never blocks — but on a hard check error the gate WRITES an auto stub `{ verdict: "check-errored", cause: "check-error", decision: "proceed", reason: "auto: check errored (fail-open)" }` so the build proceeds AND the precommit presence-gate still finds the field (no fail-open→precommit-block contradiction; the errored run is visible in the audit, not silently dispositioned).
- **Precommit (backstop — prevents the duplicate becoming a pushed/merged PR even if the start-write was skipped).** `instar-dev-precommit.js` refuses a commit whose trace lacks a `duplicateBuildCheck` field. **Presence-only:** it gates on the FIELD EXISTING, never on the verdict's VALUE — it MUST accept `disposition:"proceed"` on a `likely-duplicate` (the human is the authority; a value-gate would silently make the validator a meaning-authority, forbidden by signal-vs-authority). This is a structural field-validator, which signal-vs-authority.md's "hard-invariant validation" carve-out explicitly permits. It DOES distinguish a `check-errored` auto-stub (§3.4 fail-open) from an author disposition: the stub satisfies presence (never wedges the commit) but precommit WARNS loudly on it and the disposition remains `check-errored` in the audit — so a build that ran on a failed check is a visible second-look signal, not silently indistinguishable from author-reviewed proceed.
- **Pre-push (advisory).** Appends to `pre-push-gate.js` `warnings[]` only (never `errors[]`); honors `INSTAR_PRE_PUSH_SKIP`/`CI`. Never blocks a `git push` (a duplicate build is recoverable — close the PR — so the irreversible-action carve-out does NOT apply; a deterministic push-block would be brittle authority over a judgment call).

**Honest residual:** the guaranteed-structural moments are the build-start `PreToolUse` gate (compute-saving, fail-open) and the precommit presence-gate (PR-collision-preventing). "Guaranteed structural" precisely means *structural whenever the check produces a verdict* — two paths do NOT save the local compute and both fall through to the precommit backstop (which still prevents the duplicate becoming a pushed PR): (a) a session whose FIRST implementation mutation is a shell command (a generator / formatter / package-install) rather than a `Write`/`Edit`/`MultiEdit` tool call — the file-edit `PreToolUse` gate doesn't see it (a secondary net on mutating `Bash` is the tracked hardening follow-up `<!-- tracked: ACT-600 -->`; the precommit backstop still catches the PR); (b) a hard check error → fail-open exit-0 → the gate waves the build through, recording the `check-errored` auto-stub (§3.4) so precommit stays consistent (the correct fail-open direction per FD5 — a broken guard must never wedge a build — with the errored run visible in the audit, not silently clean).

### 3.5 Audit trail + outcome reconciliation (measurability)
Every verdict + `cause` + recorded disposition is appended to `logs/dup-build-check.jsonl` (metadata only — no untrusted PR body text, no secrets). The was-it-actually-a-dup **outcome** (knowable only at PR close/merge) is a tracked follow-up, NOT claimed as shipping here `<!-- tracked: ACT-596 -->`: this increment logs verdict+disposition+cause; outcome reconciliation (a `instar dup-build outcome <verdict-id> <dup|not>` lever or a PR-close hook) lands later, so §3.5 honestly does not claim a measurable false-pos/false-neg ratio until that write-back exists.

## 4. Testing (all three tiers + wiring + semantic + concurrency)
- Unit: target extraction (exact + deterministic Jaccard floor; missing `## Decision points touched` still yields a fingerprint); the TOTAL verdict ladder incl. every `cause` and the non-substrate-degraded cell; **fail-open on every error path** (bad spec, gh crash, git error, timeout, torn ledger line → non-blocking + exit 0); option-injection defense (a value failing its TYPED validation for its arg position is dropped, never shelled — not a blanket leading-`-` reject, per §3.2a) for spec-, PR-, AND ledger-derived values; ledger write-first-then-scan simultaneous-append race (exactly one builder yields); liveness conjunction (pid-reuse → not live; old-but-live → still fires); fuzzy byte-cap enforcement before similarity.
- Integration: pre-push appends `warnings[]` never `errors[]` + honors `INSTAR_PRE_PUSH_SKIP`/`CI`; the precommit backstop refuses a commit whose trace lacks `duplicateBuildCheck` AND accepts `disposition:"proceed"` on `likely-duplicate` (presence-only, wiring-integrity — the gate consumes the field, not a no-op).
- Semantic boundaries: degraded substrate scan → `verify` never `clear`; degraded non-substrate → `clear`+`degraded` flag (inspectable); WEAK-only → silent note, never escalate.
- E2E regression — the REAL incident geometry: a committed `tests/fixtures/` mini-repo where the census is being ADDED by an open-PR fixture (NOT already wired on the fixture `main`) + a live local-ledger sibling entry → `likely-duplicate`. Pinned census artifact path + `wired` marker format; open-PR source injected via a fixture adapter (no gh/network).

## 5. Rollout & operations
- Config flag `instarDev.duplicateBuildGuard` (default: build-start-record-on, stop-hook-on, precommit-backstop-on, pre-push-warn-on). Off-switch `INSTAR_DUP_BUILD_CHECK=off` no-ops the check (mirrors `INSTAR_PRE_PUSH_SKIP`). Rollback: flag off; the check no-ops; no data migration, no agent-state repair.
- **Migration Parity Standard does NOT apply**: a dev-lifecycle tool (instar-dev/pre-push/precommit scripts + a dev stop-hook), not an agent-installed fleet file — nothing propagates via `PostUpdateMigrator`. Stated so a reviewer doesn't flag a missing migration.
- Fail-open + bounded + `CI`-skipped open-PR scan: no material push latency, never blocks on its own failure.

## Decision points touched
- **Overlap DETECTION** (does a target match a concurrency source / main state) — `invariant`. Deterministic reads (argv-safe git/gh + the fixed §3.3 ladder). Emits a signal and stops; never weighs signals into a block/allow. CONDITIONAL + stated: holds ONLY because the tool never gains a stand-alone block path — a hard block would route through the existing authority (the author's recorded disposition), never a new brittle blocker.
- **Target MATCHING strategy** (exact vs deterministic Jaccard; rename tolerance) — `invariant` with a declared conservative floor: fuzzy-only capped at `verify`; the reworded-duplicate + cross-machine-early-window false-negatives are stated (§3.1), not hidden. No LLM arbiter (deterministic similarity + the author as disposition authority).
- **Proceed / abandon disposition** — `invariant` (from the tool's perspective: the tool's ONLY role is deterministic field-recording/validation — a presence-only gate — never a judgment; the proceed/abandon judgment itself is the human author's, explicitly NOT a tool decision-point, so no LLM arbiter / floor is introduced). STRUCTURALLY RECORDED (PreToolUse gate at build-start; precommit presence-gate as backstop), so author-authority and Structure>Willpower coexist. The precommit gate is presence-only (never value), keeping it a structural validator, not a meaning-authority.

## Frontloaded Decisions
- **FD1 — Pre-push disposition: warn-only (`warnings[]`), NEVER a hard `git push` block.** Teeth are the build-start stop-hook + the recorded disposition. *cheap-to-change:* yes (advisory-first; any escalation cheap only under fail-open here).
- **FD2 — Concurrency scan:** bounded/non-interactive/timed `gh pr list` + local sibling ledger + merged-commit lookback; LOUD degradation (degraded substrate → `verify`). A same-author open PR touching a STRONG target → `likely-duplicate`; WEAK-only → silent `clear`+note, NEVER `verify` (matches §3.3 + the §4 boundary test). *cheap-to-change:* yes.
- **FD3 — Hook placement: PRIMARY is the build-START stop-hook (compute-saving) + the precommit presence-backstop (PR-collision-preventing); the skill-prompt advisory is a UX nicety, never the catch.** Resolves the round-1+round-2 finding that precommit-only fired after the build. *cheap-to-change:* yes for the advisory; the stop-hook + precommit ship as the structural core.
- **FD4 — The TOTAL verdict ladder (§3.3)** incl. `cause` tagging + the non-substrate-degraded cell. *cheap-to-change:* yes (a deterministic table behind the flag).
- **FD5 — Fail-open on ALL errors → non-blocking + exit 0.** A safety invariant, not a tunable.
- **FD6 — Fuzzy = token-set Jaccard (no embeddings), a named conservative threshold tuned toward recall, byte-caps enforced BEFORE similarity.** A judgment-shaped number resolved as a stated conservative default (per signal-vs-authority: a stated floor, tuned cheap-side because fuzzy only yields `verify`). The threshold is CALIBRATED against a committed fixture corpus of known duplicate / non-duplicate PR-pairs (a §4 test: recall on the dup pairs, a precision floor on the non-dup pairs) before warnings enable — an evidence-set constant, not a guess, since platform words recur across unrelated specs. *cheap-to-change:* yes (threshold is a tunable constant).

## Multi-machine posture
- **Open-PR + merged-commit sources** — `unified`. Read from `origin/main` (fetched/normalized, NOT a bare local `main` — so a stale checkout can't diverge the verdict) + GitHub open PRs (shared truth). Not durable per-machine state (an ephemeral in-process cache, not a durable per-machine file).
- **Local sibling in-flight ledger** — `machine-local` BY DESIGN. `machine-local-justification: hardware-bound-resource` — a concurrently-building sibling session/worktree/process (its pid, procStartToken, worktree) physically exists on ONE machine's disk; a build on a *different* machine leaves no local trace. This signal exists specifically for the same-machine parallel-session case (the actual incident). **Honest scope of the guarantee:** it catches two builds that share the SAME agent home / ledger root (the common single-operator-agent case). Two sessions under DIFFERENT agent homes, separate sandboxes/containers, or different users on the same physical box do NOT share the ledger and are NOT caught by it — that cross-agent/cross-home case rolls up with the cross-machine early window into the durable-registry direction `<!-- tracked: ACT-595 -->`. Cross-machine parallel builds are covered by the `unified` GitHub layer once either pushes; the pre-PR cross-machine early window is a stated known false-negative tracked at `<!-- tracked: ACT-595 -->`. **Why the local ledger ships first** (not a distributed lease/claim registry or draft-PR reservation, the industry-standard "reserve before build" answers): the incident that motivates this work was two sessions on ONE box (the common case for a single-operator agent), which the zero-infrastructure local ledger catches completely; a durable cross-machine registry (ACT-595) adds a new always-on coordination surface + failure modes (lease expiry, split-brain, a reservation outliving a crashed build) that must not gate a build fail-closed — so it is deliberately sequenced after the local ledger proves the verdict-ladder + disposition UX, rather than shipped as speculative distributed infrastructure.
