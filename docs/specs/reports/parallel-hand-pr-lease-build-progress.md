## BUILD-TIME DESIGN DECISION (mine, reversible ‚Äî record in the ¬ß4 artifact as a deviation-with-rationale)
- **Enforcement = PreToolUse hook ‚Üí `POST /pr-leases/evaluate {command, cwd}` (localhost), NOT a hook-local JSON read.** The spec ¬ß4 idealized "single local JSON read, no HTTP"; the build surfaced that a generated standalone `.js` hook cannot cleanly import the TS store, and reimplementing the security logic in JS would DUPLICATE it (a maintenance + drift hazard on a security path). The codebase pattern (every instar hook ‚Äî hook-event-reporter, action-claim) is hook‚Üílocalhost-server. So: the hook reads `tool_input.command` + `cwd` + env (INSTAR_AUTH_TOKEN/INSTAR_SESSION_ID), POSTs to the server; the SERVER resolves the topic (from the session‚Üítopic map it already has) and runs `PrHandLease.evaluate` + `canonicalPushKey` (tested TS), returns `{decision}`; the hook exits 0 (allow) / 2 (deny, stderr shown). **Server-unreachable / any hook error ‚Üí fail-open exit 0** (the own-crash fail-open guarantee, ¬ß3.4). Latency: a localhost round-trip (~ms), the git dry-run resolver runs server-side ONLY on a detected push. This is a refinement, not a security-posture change (same logic, server-side where it belongs).
