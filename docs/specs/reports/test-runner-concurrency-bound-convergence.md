A real external (non-Claude) pass ran on every round through the agent's own `codex` (GPT-5.5) and `gemini` (2.5-pro) CLI logins — both families, every round, the honest mechanism. **Honest note on the final external posture:** gemini converged to MINOR (complexity observations only, no design gap, endorsing the design in principle). codex held a **SERIOUS** verdict through the final round — but its round-10 items, cross-examined by the six internal reviewers, did not survive as *material* spec-changes: the argv-brittleness is ship-gated per-version with a wrapper fallback (confirmed by the adversarial reviewer), the lock "rename-back" race resolves in the fail-open-safe direction bounded by the storm ceiling (independently confirmed by the adversarial reviewer, then further hardened in round 10), the clamp-active-mandatory point is a soak-policy preference now adopted as the recommended default, and the residue is codex's standing **"the bespoke file-lock is complex — reconsider SQLite/proper-lockfile post-soak"** view. That view is recorded here as a tracked post-soak option, not a blocking gap: the spec deliberately rejects a native SQLite dependency on the test path (a broken `better-sqlite3` rebuild would block ALL testing host-wide — a false-BLOCK with no degradation path), and both externals frame the re-evaluation as something the dry-run soak's own ledger evidence should drive.

## ELI10 Overview

When many test suites run at once on one machine — several agent build sessions, every `git push`, every build check — they each spawn a worker pool sized to the machine, and the sum can hit 300+ threads. That starves the always-on agent servers, their health-monitors read the stall as sickness, and healthy servers get killed. The safety mechanism becomes the outage. This spec adds a **host-wide "ticket counter"** for test runs: full suites take a ticket and, by default, run one-at-a-time per machine; small inner-loop runs get their own roomier lane so day-to-day iterating isn't blocked. It's the same *class* of fix as the existing limit on AI subprocesses (the "fork-bomb cap"), one layer up.

The single most important design decision is that this limiter **fails in the opposite direction** from the AI-subprocess cap. For AI subprocesses, "when unsure, block" is right (an out-of-memory crash is catastrophic). For test runs it's exactly wrong: wrongly blocking would wedge every push and build on the machine — and then the agent can't even ship the fix for the very overload it's trying to prevent. So when the limiter is unsure (a corrupt bookkeeping file, a stuck lock, an ambiguous run), it **lets the run through** — a brief extra suite is cheap and self-clearing; a wedged dev loop is not.

It ships the safest possible way: **watch-only first.** For a bounded soak (at least 14 days, 50+ real suite runs, zero false blocks) it takes tickets, counts, and writes a durable log of every "I would have blocked this" — but blocks nothing. Only after the log proves it's harmless does the owner flip one file to turn real blocking on. A recommended middle stage turns the "keep runs small" clamps real (still no blocking) so even that milder behavior is proven on real load first. A separate, opt-in, off-by-default mode can force-stop a genuinely hung suite, but it's fenced four ways and stays off through the entire trial.

## Original vs Converged

The draft was a near-verbatim reuse of the AI-subprocess cap. Ten rounds of review changed it substantially:

- **Fail-direction inverted.** Round 1 established that copying the subprocess cap's "when unsure, block" default would *re-create* the exact meltdown this feature prevents. The whole design now biases toward letting runs through on uncertainty.
- **Two lanes instead of a lane + a bolted-on exemption.** Small "targeted" runs became a second symmetric ticket lane (same proven mechanism) rather than a bespoke skip-and-count that generated a bug per round.
- **Killing became opt-in, off-by-default.** The original reclaimed a hung suite by killing it; review made the default "just take the ticket back, never kill" — process-signaling is a separate switch, fenced four ways, off through the whole soak.
- **The clever nested-run bookkeeping was deleted, then its bound restored correctly.** Round 8 (on both external models' advice) retired an intricate "reservation" mini-scheduler for nested test runs in favor of simply keeping every nested run small. Round 9's confirmation caught that this simplification had removed a bound a command-line flag could exploit — so the "keep it small" limit was made command-line-proof. A clean example of the review process catching a regression the same round it was introduced.
- **Every "quiet twin" closed.** A recurring pattern — a lever with a ceiling but no floor, or a setting that could silently disable the guard — was closed for the run caps, the posture, the process-kill arm, and (round 8, a high-severity find) the hung-suite timer, which had a ceiling but no floor and so could be zeroed to switch the whole limit off.
- **The bespoke lock made correct-then-simplified.** Rounds 9-10 hardened the stale-lock reclaim against a path-vs-inode race, then simplified the residual by recognizing that for *this* lane a lost bookkeeping row is the safe (over-admission) direction anyway.

## Convergence verdict

**Converged at round 10 for a dry-run-first ship.** Zero material findings in the final round across all six internal reviewers; gemini MINOR with no design gap; codex's SERIOUS verdict examined and found non-material (its items are ship-gated, fail-safe, soak-policy, or the standing complexity view).
