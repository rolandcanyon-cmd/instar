# Spec-converge round 1 findings — multi-machine-lease-self-heal

6 internal reviewers (security, adversarial, scalability, integration, decision-completeness, lessons-aware). External cross-model: UNAVAILABLE (codex/gemini not installed — Claude-only agent). Standards-conformance gate: unavailable against dev-clone spec (shadow-install server can't see it; fail-open).

## CRITICAL (blocking)

- **C1 — F2 keys on `acquiredAt`, which `renew()` NEVER refreshes.** Verified in source by 5 of 6 reviewers: `LeaseCoordinator.renew()` re-signs with the ORIGINAL `view.lease.acquiredAt`; only `expiresAt` advances. So `now - acquiredAt` grows unbounded on a healthy holder → F2 marks every healthy holder takeover-eligible after `ttl*factor` (4 min) → steals live leases → causes the dual-awake/flap it was meant to prevent. The spec's proposed unit test would enshrine the bug. **FIX: detect non-renewal from a LOCALLY-OBSERVED MONOTONIC signal** — the standby stamps (on its OWN monotonic clock) when it first observes a new (epoch, nonce) from the holder; non-renewing = no advance in `factor*ttl` of the observer's monotonic time. Skew-immune, unforgeable, single-clock.
- **C2 — F1's same-loop setInterval cannot fix a true event-loop stall, and the real root cause is most likely a hung never-settling await leaving a reentrancy guard stuck `true`.** Live signature (HTTP /health 200, tick silent 91 min, survived `launchctl kickstart`) ⇒ NOT a dead loop; ⇒ a parked promise in `renew()`/`pullFromPeers()` whose `finally` never runs → `leaseTicking` stuck true → all future ticks early-return. **FIX (reordered): PRIMARY = bounded await (timeout/AbortController) around renew/acquireIfEligible/pullFromPeers/broadcast so a tick can never hang; SECONDARY = monotonic-clocked watchdog that re-arms the timer + resets the guard ONLY when a recorded `tickStartedAtMs` exceeds a hard ceiling (never preempt a legitimately-slow live tick).** Scope: F1 covers lost-timer + stuck-guard + hung-await; a true loop stall is layer-2 (out-of-process fleet/launchd watchdog). Use monotonic clock. Wrap watchdog callback in try/catch (never crash the process).

## HIGH

- **H1 — F4 `preferredAwakeMachineId` is unauthenticated, machine-local, and can DIVERGE.** Both-prefer-self ⇒ each declares itself winner ⇒ worse-than-baseline flap (baseline lower-machineId converges). Config-write attack: set preferred=self to win every tie. **FIX: validate against the SAS-verified registry; GOSSIP the effective preference in the lease/heartbeat payload and apply F4 only on AGREEMENT; fall back to lower-machineId on disagreement/absence/unknown-id; raise one deduped Attention on disagreement.**
- **H2 — F4 tie-break override must be HEALTH-gated.** A frozen-but-contesting preferred would win every tie ⇒ mesh converges onto the dead machine. **FIX: unify both F4 effects behind one `preferredIsHealthy()` predicate (locally-clocked, same signal as C1); a non-renewing preferred falls back to lower-machineId.**
- **H3 — F3 has no mid-run transition hook (config needs a restart) and `relinquish()` doesn't broadcast; `forceLocalExpiry` is git-less-only.** The live zombie is a PERSISTED record from a prior process. **FIX: make F3 LEVEL-TRIGGERED inside the observe-only tick branch — if observe-only AND the durable lease names self AND not-yet-relinquished → `relinquishAndBroadcast()` once (build it; relinquish today does not broadcast). Handle the boot/persisted-zombie case. The tombstone must NEVER compete at the epoch floor or be folded as authoritative; a higher-epoch takeover always dominates.**
- **H4 — F1 ships ENABLED but mutates concurrency guards (an authority action), and a blind guard-reset can induce a concurrent double-tick** (two CAS/broadcast in flight, clobbering the contested-episode latch). **FIX: guard-reset gated on `tickStartedAtMs > ceiling`; bounded await makes the guard never stick anyway; self-disarm + Attention if re-arms exceed N/window.**
- **H5 (decision-completeness) — no `## Frontloaded Decisions` section; F4 operational mechanics (who sets it, both machines, divergence) undecided.** **FIX: add the section; resolve F4 config symmetry + F3 boot path + F2 signal precondition explicitly.**

## MEDIUM

- **M1 — F2's `factor*ttl` (4 min) vs presumed-dead (15 min) window: other consumers (dashboard awakeMachineCount, escalation, MeshRpc) disagree about "awake" during the gap.** Document + surface "taken-over-while-lastSeen-fresh"; consider larger factor (4-6 missed renews).
- **M2 — Migration parity points at the wrong mechanism.** It is `src/config/ConfigDefaults.ts` (applyDefaults/deepMerge add-missing), NOT a hand-written `migrateConfig()` block. Add the 4 keys there + a migration-parity test (and don't overwrite an operator-set `preferredAwakeMachineId`).
- **M3 — overloaded `telegramPolling` flag** (means muted-ingress AND lease-observe-only AND deferential). Diagnosis item E (decouple) was dropped. **FIX: introduce first-class `multiMachine.leaseSelfHeal.leaseRole: active|observe-only|deferential` (back-compat: `telegramPolling===false` ⇒ observe-only); F4 sets the non-preferred machine's role deterministically (Structure>Willpower; closes item E).**
- **M4 — flag consolidation + range validation.** Fold F1-F4 under one `multiMachine.leaseSelfHeal` namespace; add startup range validation (reject `staleFactor<1`, `nonRenewalFactor<1`) joining the existing `multiMachine.*` nonsensical-combo rejection.
- **M5 — live-verify harness must be DETERMINISTIC** (injectable stall seam, not "wait for the bug") and read BOTH machines' audit logs (F3 relinquish confirmed from the relinquishing machine's `security.jsonl` role_transition, never inferred from the holder's view). Induce a clock offset for F2's skew test.
- **M6 — F4 deferential-standby coverage gap under a flapping preferred** ⇒ never sit at awakeMachineCount=0 > 1 TTL; add dwell/hysteresis + flapping detector.
- **M7 (Agent Awareness)** — add CLAUDE.md template note (preferredAwakeMachineId, watchdog self-heal) + `/health` syncStatus fields (`leaseTickWatchdog`, `preferredAwakeMachineId`) + proactive trigger ("why does the laptop keep taking over?" → preferredAwakeMachineId).
