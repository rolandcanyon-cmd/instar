## ELI10 Overview

We keep a map of *which doors we can walk through to reach an AI model* (the Claude app, the Codex tool, pi, a paid Google/OpenRouter key, Antigravity) and *what the best model behind each door is*. That map quietly went stale once — some of our helpers used a year-old Google model for weeks because our list said it was best and nothing ever re-checked. This spec builds three things so that can't happen silently again: (1) a proper, growing **registry** of doors + top models (with exact ids + prices), split into a *reviewed* half that ships with instar and a *live per-machine* half of "what this computer could actually reach when it last looked"; (2) a **robot that re-checks on a schedule**, tells us **only what changed**, is careful about money (the normal check is free; anything paid needs a manual switch AND a spending limit or it refuses) and secrets (it only ever handles key *names*), and physically can't edit our real files; (3) a written **standard** that says a stale map is a bug, kept current by machinery, not by someone remembering.

What changed most during review is that the design got dramatically more careful about the *seams* — the places where two safety features touch. The first draft's "run a probe and tell someone" idea was correct; convergence turned it into a design where the robot can't spend money by accident (four independent locks), can't leak a poisoned value from its own scratch file into a message or a web page (every field re-checked on read, not just on write), can't quietly lose a "something changed" notice if the message-sending layer holds it (it re-tries until confirmed and never advances its baseline until delivery), and can't flood you with "everything's new!" the first time it runs on a new machine (an initial-seed rule). None of these were bugs a casual reader would spot — they were found by pushing hard on how the pieces interact.

The main tradeoff the review kept surfacing (and accepting): the scan runs as a *supervised* job with a small command surface guarded by a shell-command allowlist, rather than a plain scheduler-native invocation. That preserves the "an LLM sanity-checks the probe result before anything surfaces" layer, at the cost of a guard that must be built carefully; the fully-non-shell version is a tracked follow-up. Everything ships **off by default**, and the whole thing never spends a cent or edits a file without an explicit, reversible operator step.

## Original vs Converged

The spec entered this convergence run having already been through two prior review rounds (which produced the round-1/round-2 machinery: a deterministic prober, a derived frontier set, a debounce+breaker, a machine-qualified attention id, a `perMachineIndependent` job flag). This run folded a set of round-2 *material findings* the operator supplied, then ran four more review rounds to convergence. In plain terms, what changed:

- **The scan can't accidentally spend money — now four independent ways.** Originally the "free by default" rule lived only in the prober's default scope. Review added: a fixed literal `--scope free-probes` in the scheduled job body; a manual-marker env the scheduler never sets (so a prompt-injected session can't select a paid scope); a fail-closed budget (absent/zero cap → refuse every metered probe); a hard per-token ceiling that bounds spend even if the price is stale; and — the last brick — the prober itself refuses any paid scope when it detects it's running as the scheduled job, independent of the guard that could fail open.
- **"Never edits our source files" went from a 3-idiom denylist to a real allowlist.** The original guard blocked `>`, `sed -i`, `tee` — which misses `cp`, `dd`, `python -c`, `git checkout`, `patch`, heredocs, etc. Review inverted it to a command-shape allowlist that parses the command with a real shell grammar (one simple command, no chaining/redirection/substitution/env-prefix, no interpreters), scoped only to this job's session, fail-open on scope / fail-closed on match.
- **The map's own scratch file is now treated as untrusted every time it's read**, not just clamped when written — because anything with disk access could plant well-formed poison that a corrupt-file check misses. Every field is re-validated on read (exhaustively, with a coverage test), and free text is HTML-escaped before it can reach the private-view page.
- **Notices can no longer be silently lost.** The original design would `POST` a change and move on; review found the message body is tone-gated (and made of file paths/jargon → would be blocked), and that advancing the baseline before delivery confirmed could bury a change forever. Converged: a jargon-safe body + a private-view link (never a localhost link), and a `last-surfaced` baseline that advances *only* on confirmed delivery — so an undelivered change re-surfaces next cadence instead of vanishing.
- **Honesty fixes.** "Notices brand-new doorways" → honestly "detects candidate/newly-configured doorways" (it checks a known list + vault keys, not open-ended discovery); a door that *answers but whose output no longer parses* is now a *surfaced* "parser needs attention" alarm instead of silently ceasing to grow the map (the exact rot the spec exists to prevent); `frontier:true` is explicitly operator judgment with structured evidence fields, not scan truth.
- **A real bug the review caught:** the config kill-switch was seeded `false`, which — under the deny-wins rule — would have meant the scan *never ran even after being enabled*. Fixed to default the field absent.
