## §10 Open questions

- **[material] Q1 — Active-provider probe: `buildProvider` truth (a) vs. `which <cli>` (b)?** → **Type B (engineering).** Reversible, internal, no external side-effect, no user-visible interface. The spec already leans (a) with sound justification (single source of truth — the active-set must match exactly what the router's `resolveProvider`/`buildProvider` will actually return at call time; a second `which`-based notion of "available" can drift and route to a framework that then degrades). The agent can and should decide this. **Resolution (in-spec):** use (a) — call `buildProvider(fw) !== null` once per framework at boot to compute the active set, reusing the SAME `buildProvider` closure already wired at `server.ts:4699`. This is strictly more honest than `which`: it captures pi-cli's TWO-precondition reality (binary present AND a `frameworkDefaultModels['pi-cli']` model pattern configured — see `intelligenceProviderFactory.ts` pi-cli case, which returns null without the model pattern). A `which pi` probe would falsely mark pi "active", make it the primary, and then degrade every call. (a) cannot make that mistake. Memoize the result at boot; the live-read of operator config is preserved separately (§4.4). NOT a user decision.

- **[material] Q2 — Runtime-computed default vs. one-time migrated config block?** → **Type B (engineering).** Internal, reversible, no durable external side-effect that a user must own. The spec leans runtime-computed with the correct reasoning (self-heals when the operator later installs/removes a provider CLI; a persisted block pins a stale active-set and re-introduces a migration-parity hazard). **Resolution (in-spec):** runtime-computed, NO persisted block. `migrateConfig()` writes nothing for this feature (leave `componentFrameworks` unset so the resolver stays live). The "migration" is purely the new code shipping — zero migration-parity risk, always reflects the currently-installed providers. This also keeps the operator-override precedence (§4.3) trivially correct: "unset" is an unambiguous signal to apply the computed default; a migrated block would blur "operator set this" vs. "we wrote this," breaking the §4.3 verbatim-passthrough rule. NOT a user decision.

- **[resolved ✅] Q3 — Is the outbound tone-gate flagged `attribution.gating`?** → Already resolved in §6 (GROUNDED ✅ — `MessagingToneGate.ts:269` sets `gating: true`, plus six other safety components confirmed). No action; this is a load-bearing regression-check fact, not a parked decision. Keep it as the named wiring-integrity test (§7).

- **[material] Q4 — Is `job` in the internal off-Claude set, or only `sentinel`+`gate`+`reflector`?** → **Type B (engineering) — but contest the framing.** This is NOT taste or product; it is an internal routing-default with a fully reversible, operator-overridable surface and a hard no-op floor (§4.2). The only real consideration is blast-radius surprise. **Resolution (in-spec): INCLUDE `job`, but verify the concrete members first.** Justification grounded in `componentCategories.ts`: the components actually tagged `'job'` are `PipeSessionSpawner`, `CartographerSweep`, `StandardsCoverageEnrichment` — all BACKGROUND/maintenance LLM work whose entire design intent (per the in-code comments) is "route OFF Claude so it never spends Anthropic quota." That is exactly this policy's goal, so excluding `job` would contradict the directive AND the existing component-level design. The spec's worry ("jobs can be heavy / may surprise") is real for HEAVY interactive jobs — but those run as spawned sessions on `topicFrameworks`, which this policy explicitly does NOT touch (§4.1). The `'job'` CATEGORY here only covers the internal `.evaluate()` callers above. Net: include `job`; add a one-line spec note that the category covers ONLY internal `evaluate()` job-callers, not spawned job sessions, so a future reader doesn't re-open this. NOT a user decision.

- **[minor] Q5 — Per-framework model-size preservation (a `fast` check stays `fast`)?** → **Type B (engineering), already structurally true — confirm, don't decide.** **Resolution (in-spec, grounded):** model SIZE is carried per-call in `IntelligenceOptions.model: 'fast' | 'balanced' | 'capable'` (`types.ts:847-849`) and flows through `evaluate(prompt, options)` UNCHANGED across every swap target — the router passes the same `options` to the primary and to each `failureSwap` provider (`IntelligenceRouter.ts:194,202`). Each provider maps the tier to its own model independently (e.g. Codex via `resolveCliModelFlag(options?.model)` at `CodexCliIntelligenceProvider.ts:298`). Size is therefore orthogonal to framework by construction; no special handling is needed and none should be added. The spec's own answer ("it should — `model` size is orthogonal to framework") is correct and now grounded. Add one unit assertion: a `fast` gating call that swaps codex→gemini still passes `model:'fast'` to gemini. NOT a user decision.

## Other latent decisions (not in §10) — swept for completeness

- **[minor] What if `active` is empty (no framework's binary builds at all, not even claude-code)?** → **Type B.** §4.2 covers `['claude-code']`-only as a no-op, but an EMPTY active set (claude binary also missing at probe time) is a distinct edge. **Resolution (in-spec):** treat empty-active identically to the no-op path — return `undefined` from the resolver (the engine's `!cfg` branch ⇒ "exactly today's behavior," delegating to `defaultProvider`). Never synthesize a `componentFrameworks` with no usable primary. Add a unit case (`active === []` → resolver returns undefined → byte-identical to today).

- **[minor] Where exactly does the computed default plug in vs. operator override (§4.4)?** → **Type B, already pinned by code.** The single wiring point is `server.ts:4693` `resolveConfig: () => config.sessions?.componentFrameworks`. **Resolution (in-spec):** wrap that thunk: `resolveConfig: () => config.sessions?.componentFrameworks ?? computedDefault` where `computedDefault` is memoized once at boot from the active-set. This preserves the live-read semantics exactly: the moment an operator edits config to SET `componentFrameworks`, the `?? ` short-circuits and their config wins verbatim (§4.3 satisfied for free). No other call site changes. Name this explicitly in §4.4 so the building agent does not invent a second resolution path.

- **[minor] `fallback: 'default'` vs `'none'` in the computed config.** → **Type B.** §4.2's example sets `fallback: 'default'`. **Resolution (in-spec, keep as written):** `'default'` is correct — it degrades an unavailable PRIMARY to the agent default (claude) rather than erroring, which matches "claude is the last resort, never absent." Note: this is the PRIMARY-unavailable branch, distinct from the `failureSwap` runtime-failure path (which already ends by re-throwing → fail-closed for gating callers). Both are already correct in the engine; the spec just needs to state that the computed config uses `fallback: 'default'` deliberately, not by accident.

- **[minor] Does the computed default also set `categories.other` / leave `default`?** → **Type B.** §4.1 says NOT `other`, NOT a top-level `default`. **Resolution (in-spec):** set ONLY `categories.{sentinel,gate,job,reflector}` + `failureSwap` + `fallback`; leave `default` and `categories.other` UNSET so anything uncategorized (the `'other'` bucket) keeps resolving to the agent default framework per `resolveFramework()` (`IntelligenceRouter.ts:108-113`). This keeps the blast radius to the four internal categories the directive names. State it explicitly to prevent the building agent from over-reaching to a top-level `default: 'codex-cli'` (which §3.2 already shows is the WRONG shape).
