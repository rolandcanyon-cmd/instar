# Round 1 — Adversarial Review

Target: `docs/specs/provider-fallback-default-policy.md`
Lens: failure modes, self-reinforcing loops, stale claims, authority ambiguity, silent degradation.
Grounded in: `src/core/IntelligenceRouter.ts`, `src/commands/server.ts` (~4674–4737, ~11253–11290), `src/core/MessagingToneGate.ts`, `src/core/intelligenceProviderFactory.ts`, the three CLI providers, `src/core/CircuitBreakingIntelligenceProvider.ts`.

---

- **[material]** Sequential failureSwap can stack timeouts and REINTRODUCE the delivery stall this spec exists to fix. The `evaluate()` swap loop awaits each target serially; each provider has a hard exec timeout (`DEFAULT_TIMEOUT_MS` = 30s Claude/Codex/Gemini, 60s PI) AND the tone gate passes `rateLimitWaitMs: 120_000`, so a target whose breaker is throttled can block up to 120s in `acquireOrWait` before throwing to the next link (`CircuitBreakingIntelligenceProvider.ts:187-190`). Worst case for the outbound tone gate on chain Codex→PI→Gemini→Claude with several providers slow/throttled is a multi-minute cumulative wait on the OUTBOUND path — the exact ~1h delivery strangle §1 is fixing, just relocated from "one slow Claude" to "four slow providers tried in series." **Resolution:** Bound the whole-swap budget explicitly. Add a per-call deadline that the swap loop respects (sum of attempts must not exceed e.g. the original caller timeout), and/or have gating callers pass a SMALL per-attempt `timeoutMs` so a swap-chain attempt fails fast instead of waiting the full 120s/30s. State the bound in §6 — "fail-closed preserved" is necessary but the LATENCY budget is the load-bearing property for the tone gate, and the spec never quantifies it.

- **[material]** Garbage / wrong-shaped output silently poisons gating decisions — a swap never fires because there is no error. A circuit breaker and the failureSwap loop only react to a THROWN error; a provider that is reachable but returns malformed or wrong content returns a normal string. `MessagingToneGate.review()` then runs `parseResponse` and, on an unparseable/invalid-rule answer, fails OPEN (`MessagingToneGate.ts:277-299, 308-318`) — i.e. it ships the message ungated and never tries the next provider. So routing the tone gate to a flaky off-Claude model that returns junk converts "fail-closed protection" into "silent fail-open, no swap, no degrade report." The directive's safety premise (swap before fail-closed) does not hold against bad-but-not-erroring answers. **Resolution:** §6 must acknowledge that the swap chain protects against ERRORS only, not against wrong answers, and that a low-quality off-Claude model changes the gate's effective failure mode from closed to open. At minimum require the dogfood (§9) to compare off-Claude tone-gate verdicts against Claude on a sample before relying on it; ideally have callers treat an unparseable gating response as a swap trigger (throw → next provider) rather than an immediate fail-open.

- **[material]** Detection of "operator has not set componentFrameworks" is fragile — existing code MUTATES the config block in place. §4.4/§4.3 hinge on "if `config.sessions.componentFrameworks` is unset, inject the computed default; if set, pass through verbatim." But the CartographerSweep wiring at `src/commands/server.ts:11260-11268` already does `config.sessions.componentFrameworks ??= {}` and writes `overrides.CartographerSweep` into it at boot when the freshness sweep is enabled. After that runs, `componentFrameworks` is DEFINED (non-empty), so a naive "is it set?" check sees operator config that the operator never wrote — and the whole default policy silently disables itself on exactly the agents that enabled the cartographer sweep. **Resolution:** The "operator set it?" signal cannot be "is the object present." Capture the operator's ORIGINAL intent at load time (e.g. a separate `operatorSetComponentFrameworks` boolean derived from the on-disk config before any runtime mutation), or have the resolver merge the computed default UNDER any runtime-injected keys rather than treating presence as opt-out. Call this out explicitly — it is a real ordering bug, not a hypothetical.

- **[material]** Active-set probe must build providers with the SAME option-bag the router uses, or "active" will diverge from "actually runnable." §4.2 leans on candidate (a): `buildProvider(fw) !== null` as the active truth. But `buildProvider('pi-cli')` returns null unless `piModel` is threaded (`intelligenceProviderFactory.ts:201-206`), and the router's real closure threads `piModel`, `piAllowAnthropicProviders`, `resolveExecJson`, and `subscriptionPath` per framework (`server.ts:4699-4721`). If the boot-time active-probe calls a bare `buildProvider(fw)` without that same bag, PI (and any future config-gated framework) is judged INACTIVE even when the runtime build would succeed — or vice versa — so the computed primary/swap chain won't match what actually runs. **Resolution:** The policy resolver must reuse the router's exact `buildProvider` closure (or the identical option-bag) for its active probe — a single source of truth, as §4.2 itself argues. Pin this in the spec: "active = the router's own buildProvider closure returned non-null," not a fresh bare call.

- **[minor]** PI is effectively dead-on-arrival in the default chain for almost every agent, making the advertised "Codex→PI→Gemini→Claude" misleading. PI requires an explicit `frameworkDefaultModels['pi-cli']` pattern that no default ships, so on a stock agent `pi-cli` is always filtered out as inactive. The spec's worked example (`active = ['codex-cli','gemini-cli','claude-code']`) quietly drops PI but the headline chain keeps advertising it. **Resolution:** Note in §4.1/§4.2 that PI participates ONLY when the operator has configured a pi model pattern, so the realistic default chain on most agents is Codex→Gemini→Claude. Avoids a stale claim that PI is a working second link out of the box.

- **[minor]** No bound on swap breadth / no "tried everything" observability beyond per-degrade events. When all off-Claude providers are down and the chain falls to Claude (correct per directive), or when every target is down and it re-throws (fail-closed), the spec relies on `onDegrade` per-hop reports. But there is no aggregate signal that "the default policy is currently routing everything to the Claude tail" — i.e. the policy has silently collapsed back to the pre-spec world and no one is alerted, which is the steady-state the spec was written to avoid. **Resolution:** Add to §6/§7 an observable for "primary chain exhausted, serving from Claude tail" (e.g. a distinct degrade reason or a `/intelligence/routing` flag) so the operator can SEE the policy has degenerated to Claude-only and isn't lulled by the "off Claude by default" promise while every call actually hits Claude.

- **[minor]** `/intelligence/routing`'s `for()` diagnostic reports only the PRIMARY framework + its availability, not the swap tail — so the spec's §7 integration assertion ("reflects the computed default") under-verifies. `IntelligenceRouter.for()` (`IntelligenceRouter.ts:119-130`) returns a single resolved framework and an `available` boolean; it has no notion of `failureSwap`. An operator reading the route sees "sentinel → codex-cli, available:true" but cannot see that the tail is `[gemini-cli, claude-code]`, nor whether those tail breakers are open. **Resolution:** Either extend the diagnostic surface to expose the resolved `failureSwap` tail + each tail framework's availability/breaker state, or scope §7's integration test to what `for()` can actually prove and add a separate assertion over the resolved config's `failureSwap` array. As written, the test plan claims coverage the current surface can't deliver.

- **[minor]** Model-size preservation (§10 Q5) holds, but the assumption "`fast` is equivalent across frameworks" is unaudited for the gating callers. Size→model mapping is per-framework and orthogonal to framework (confirmed: `fast` → Codex gpt-5.2 / Gemini flash / Claude haiku via the tier tables), so a swap won't silently jump to a HEAVIER model — that part of Q5 is sound. The unstated risk is the inverse: a `fast` tier model on the swapped provider may be materially WEAKER at the gating judgment (e.g. a small Gemini flash parsing the tone-gate rule grammar), feeding the garbage-output finding above. **Resolution:** Close Q5 affirmatively in the spec (size is preserved, no heavier-model surprise) AND note the residual: equal SIZE ≠ equal CAPABILITY for gating, so the dogfood must validate verdict quality per provider, not just that a call resolves.

- **[cosmetic]** §4.4 cites the construction site as `server.ts (~line 4687)` but the actual file is `src/commands/server.ts` (top-level `src/server.ts` has no such code). Minor, but a reader following the pointer lands in the wrong file. **Resolution:** Correct the path to `src/commands/server.ts:~4687` throughout (§4.4, and the §3.1 reference).
