# Round 1 — Integration / Deployment + Multi-Machine review

Reviewer lens: migration parity for existing agents, config knobs, rollback lever,
observability surface, and multi-machine posture. Grounded against the live worktree
(`src/commands/server.ts`, `src/core/IntelligenceRouter.ts`, `src/core/PostUpdateMigrator.ts`,
`src/core/intelligenceProviderFactory.ts`, `src/core/Config.ts`, `src/server/routes.ts`).

- **[material]** Cartographer-sweep in-memory mutation collides with "is `componentFrameworks` set?" detection. At boot the cartographer block (`src/commands/server.ts:11257-11271`) reads `config.sessions.componentFrameworks`, and when the sweep is enabled MUTATES it in place — `s.componentFrameworks ??= {}` then sets `c.overrides.CartographerSweep`. The spec's §4.3/§4.4 default kicks in only when `componentFrameworks` is "unset". If that unset-check is evaluated once at construction (§4.4 "computed once at boot", router built at :4687, BEFORE the cartographer mutation at :11257), the cartographer's later override never reaches the router; if the unset-check is re-evaluated live, the cartographer mutation flips `componentFrameworks` to "set" and the entire off-Claude default policy silently disables itself for that agent. Either branch is a real bug, and it lands precisely on echo (codex active + cartographer sweep are the dogfood combo). **Resolution:** the spec must specify how "operator has set `componentFrameworks`" is detected so it is robust to in-memory mutation — e.g. snapshot operator-intent at construction (capture whether the *file/config* had it set, not the live object), or have the default-resolver MERGE the computed default *under* any later-injected `overrides` rather than treating any non-empty object as operator authority. Add an explicit test for "cartographer sweep enabled + unset componentFrameworks → sentinels still route off Claude AND CartographerSweep override is honored."

- **[material]** `pi-cli` in the preference chain is dead-by-default and the "active" probe must use option (a) for honesty. `buildIntelligenceProvider('pi-cli')` returns `null` unless `frameworkDefaultModels['pi-cli']` is set (`src/core/intelligenceProviderFactory.ts:200-207` — pattern-less pi is refused by design). A `which pi` existence probe (§4.2 option b) would falsely mark pi active and route sentinels to a framework that null-builds every call, forcing a per-call degrade. The router's `buildProvider` truth (option a) correctly returns null. **Resolution:** §4.2 should resolve to option (a) explicitly and CALL OUT the pi-cli gotcha: pi is "active" only when both the binary AND the model pattern are present. Add a unit case: chain with codex absent + pi binary present but no model pattern → primary = gemini/claude, NOT pi.

- **[material]** §8 Agent-Awareness will NOT land for existing agents — the migrateClaudeMd guard blocks re-editing the section. The existing migration (`PostUpdateMigrator.ts:5504`) inserts the "Per-Component Framework Routing" section only `if (!content.includes('Per-Component Framework Routing'))`. Every already-deployed agent ALREADY has that section, so the new default-behavior text §8 wants to add will be skipped — existing agents update the code but never learn the behavior changed (violates the Agent Awareness + Migration Parity standards). **Resolution:** add a SEPARATE content-sniffed migration block keyed on a NEW sentinel string (e.g. `!content.includes('run off Claude by default')`) that appends the default-policy note, mirroring how the pi-cli note (:5525) was added as its own guarded block rather than editing the existing section. Update §8 to name this explicitly. Also update the `generateClaudeMd` template (`src/scaffold/templates.ts:838`) for new installs.

- **[minor]** §5 "self-heals when the operator later installs/removes a provider CLI" is overstated — it is restart-scoped, not live. Binary detection is memoized for the process lifetime (`_frameworkBinaryCache`, `Config.ts:146`, only test-resettable), and §4.4 computes the active-set "once at boot" and memoizes it. So installing codex on a running agent does NOT route sentinels onto it until the next server restart. This is acceptable behavior, but the spec should state it honestly rather than implying live self-healing. **Resolution:** reword §5 to "self-heals on the next restart after a provider CLI is installed/removed (binary detection and the active-set are boot-memoized)"; optionally note `_resetFrameworkBinaryCache` exists if a live re-probe is ever wanted.

- **[minor]** Multi-machine posture is correct but undeclared — make it explicit. The active-set is inherently machine-local (different machines have different CLIs installed). Runtime-computed-per-machine (§5, no persisted block) is the RIGHT call: it guarantees machine A's installed-providers are never pinned onto machine B, and `config.sessions.componentFrameworks` is NOT in any `stateSync` store (confirmed — only relationships/preferences/learnings/etc. replicate), so config never crosses machines. The mandatory posture is: **machine-local BY DESIGN** (reason: provider availability is a property of the physical host). The observability surfaces (`/intelligence/routing`, `/metrics/features`) are already per-machine reads. **Resolution:** add a one-line "Multi-machine posture" subsection stating the policy is machine-local-by-design and explaining WHY a persisted/replicated block would be wrong (it would pin one host's active-set onto another).

- **[minor]** Rollback lever §9 contradicts §3.2's own bug analysis. §9 says rollback = operator sets `componentFrameworks: {}` (explicit empty) ⇒ "everything back on the agent default framework." But an explicit empty object is "operator-set" (§4.3) so the default policy is bypassed — correct — yet the spec elsewhere (§3.2) notes a static `default: 'codex-cli'` degrades wrongly; reviewers should confirm `{}` truly resolves every component to `defaultFramework` via `resolveFramework` (`IntelligenceRouter.ts:108-113`: `!cfg` is false for `{}`, `overrides`/`categories`/`default` all undefined → returns `defaultFramework`). It does — so the lever works — but the spec should also offer the simpler/safer rollback: set `sessions.componentFrameworks.default: 'claude-code'` is NOT it (that re-introduces the §3.2 degrade); `{}` is the right lever. **Resolution:** keep `{}` as the documented rollback but add the one-line proof that `{}` ≡ "all default framework" and warn operators NOT to use `default: 'claude-code'` as a rollback (it loses the swap chain). Also confirm rollback is hot (live `resolveConfig` read — yes) so no restart is needed.

- **[cosmetic]** Open question §10.4 (should `job` be in the internal set?) is a real deployment risk worth resolving in-spec, not deferring. Routing the `job` category off Claude by default means scheduled jobs with `supervision` tiers (Haiku LLM-supervised pipelines) silently move to codex/gemini — a behavior change to job execution that an operator may not expect. **Resolution:** decide explicitly. Recommend `sentinel`+`gate`+`reflector` in the default internal set and leaving `job` on the agent default (jobs can be heavy and operator-authored), with a note that operators can opt jobs in via `categories.job`.
