# Round 1 ‚Äî Scalability / Performance review

Reviewer lens: hot-path cost, boot cost, concurrent calls, memory churn, fail-open
semantics, latency. Grounded against `src/core/IntelligenceRouter.ts`,
`src/core/intelligenceProviderFactory.ts`, `src/core/Config.ts` (detect memo),
`src/commands/server.ts` (router construction ~4687, cartographer mutation ~11260),
`src/core/LlmCircuitBreaker.ts`, the CLI providers' 30s timeout, and
`docs/specs/per-component-framework-routing.md`.

---

- **[material]** Serial failure-swap latency grows with the longer default chain ‚Äî worst case `(N+1)√ó30s` of head-of-line blocking ‚Äî The `evaluate()` swap loop (`IntelligenceRouter.ts:193-218`) `await`s each provider **sequentially**, and every CLI provider defaults to a 30s timeout (`ClaudeCliIntelligenceProvider.ts:18`, `CodexCliIntelligenceProvider.ts:36`). A clean failure (rate-limit/circuit-open) returns fast, but a **slow-hang** (the exact tonight symptom ‚Äî "transient fetch errors", a provider that neither answers nor errors) burns the full 30s on the primary *before* the swap even starts, then potentially 30s per swap target. With the new default chain putting up to 4 providers in play, a gating call (e.g. the tone-gate) can block ~90-120s under a multi-provider brownout, where today's single-provider config caps it at ~30s. The spec's ¬ß6 "fail-closed preserved" is about correctness, not latency, and doesn't address this regression. **Resolution:** Add the analysis explicitly to ¬ß6/¬ß9: confirm gating callers pass a bounded `options.timeoutMs` (the provider honors it) and that the *aggregate* swap budget is acceptable for the tone-gate's delivery path; consider documenting (or capping) total swap walltime so a brownout across 3 providers can't serialize into a multi-minute outbound stall ‚Äî the very stall this spec exists to kill.

- **[material]** Boot-time active-set probe must reuse the router's provider cache, or it double-builds every framework's provider+breaker ‚Äî ¬ß4.2 says compute the active-set "once at boot via `buildProvider(fw) per framework`," but the router maintains its OWN `framework ‚Üí provider` cache (`IntelligenceRouter.ts:88,133-145`) and `server.ts:4699` allocates a **fresh `new LlmCircuitBreaker()` per `buildProvider` call**. If the boot probe calls the factory directly (outside the router instance), it constructs N providers + N breakers that are immediately discarded, and the router rebuilds them again on first real call ‚Äî wasted work AND, worse, the boot probe's breaker is a *different object* than the runtime one, so a framework deemed "active" at boot could differ from what the router actually routes to. **Resolution:** Compute the active-set by calling the router's own `providerFor()`/`for()` path (which populates and reads the shared cache), not a parallel `buildIP()`. State in ¬ß4.2 that the probe and the runtime route MUST resolve through the same cache instance ‚Äî one build per framework, period.

- **[material]** "Operator has not set componentFrameworks" is mutated at boot by the Cartographer sweep ‚Äî false "unset" never triggers the default ‚Äî `server.ts:11260-11268` mutates `config.sessions.componentFrameworks.overrides.CartographerSweep` at boot when the cartographer freshness sweep is enabled (and `Config.ts:864` only carries the block through when it's a non-empty object). The spec's ¬ß4.3/¬ß4.4 gate the default on `componentFrameworks` being "unset". On an agent that runs the sweep, the block is already populated by boot ‚Äî so the policy resolver would see "operator set it" and **skip the default entirely**, silently defeating the whole feature on exactly the dogfood agent (echo runs the sweep on codex). **Resolution:** ¬ß4.3/¬ß4.4 must define "operator-set" precisely ‚Äî not "the object exists" but "the operator authored a primary selector (`default`/`categories`/top-level `failureSwap`)". System-injected `overrides.CartographerSweep` must NOT count as operator intent; the resolver should fill in the default categories/failureSwap *around* a system override. Add a test for "sweep-injected override present ‚Üí default still applies".

- **[minor]** Memoized effective config can go stale vs. a CLI installed/removed mid-run ‚Äî but the staleness direction matters ‚Äî ¬ß4.4 memoizes the active-set "once at boot". `detectFrameworkBinary` is process-memoized for the whole lifetime (`Config.ts:146`, caches negative results too), so a provider CLI installed *after* boot is invisible until restart, and the router's per-framework cache (`IntelligenceRouter.ts:143`) also never re-probes. Practically low-impact (operators rarely `npm i -g codex` mid-run), and the spec's "self-heals when the operator installs/removes a provider" claim in ¬ß5 is **only true across a restart**, not live. **Resolution:** Correct the ¬ß5 wording ‚Äî it self-heals on next boot, not live ‚Äî and confirm that's acceptable (it is, given config is the live-read layer and provider availability legitimately changes only at install/restart boundaries). No code change needed; just don't overclaim "live self-heal".

- **[minor]** Routing many sentinels onto one Codex CLI concentrates load, but the per-framework breaker bounds the herd ‚Äî The default routes `sentinel`+`gate`+`job`+`reflector` all to `active[0]` (Codex). That's a real shift of background volume onto a single codex account/CLI. However each framework has its own breaker (`server.ts:4701`, one `LlmCircuitBreaker` per framework) and gating calls swap on failure, so a codex slowdown trips ITS breaker and gating calls fail over to the next link ‚Äî the throughput risk is bounded, not unbounded. The genuine residual is **non-gating** internal calls (heuristic fallback, no swap): those just pile onto codex with no relief valve other than the breaker. **Resolution:** Acknowledge in ¬ß6 that the default concentrates background load on one provider by design; note that non-gating calls rely solely on the per-framework breaker for relief (no swap), and that this is the intended trade (spread off Claude) ‚Äî but flag it so an operator with a thin codex quota understands sentinels now consume it.

- **[minor]** `resolveConfig` stays O(1)/cheap per call only if the resolver memoizes ‚Äî the active-set filter must not run per call ‚Äî `resolveFramework` (`IntelligenceRouter.ts:103-113`) is already O(1) string lookups, and `resolveConfig` is read on EVERY `evaluate()`/`for()` call (`:126,159`). If the policy resolver wired at `server.ts:4693` recomputes `INTERNAL_FRAMEWORK_PREFERENCE.filter(isActive)` on each invocation it would call `providerFor`/detect per call ‚Äî still cheap (memoized) but allocates a new config object each call ‚Üí GC churn on the hot gating path. **Resolution:** The resolver closure must compute the effective config **once** (memoize the object) and return the SAME reference on every `resolveConfig()` call, so identity-stable and zero per-call allocation; explicitly say so in ¬ß4.4. (The existing `resolveConfig: () => config.sessions?.componentFrameworks` returns a stable reference today ‚Äî the policy version must preserve that property.)

- **[cosmetic]** Probe choice (a) `buildProvider` vs (b) `which` ‚Äî (a) is correct and cheaper than it looks ‚Äî ¬ß4.2's open question leans (a). Grounded: `buildProvider`‚Üí`detect*` is process-memoized (`Config.ts:146`) AND provider/breaker constructors are pure field assignments (`LlmCircuitBreaker.ts:243`, no I/O), so (a) costs one memoized detect + one cheap object alloc per framework at boot ‚Äî genuinely "already cached" as the spec claims. Option (b) `which` would be a SECOND notion of availability that can drift from what the router actually routes to (e.g. pi-cli is "present" by `which` but the factory returns null without a model pattern ‚Äî `intelligenceProviderFactory.ts:201`). **Resolution:** Resolve the open question to (a) and cite the pi-cli `which`-says-present-but-factory-returns-null case as the decisive reason (b) is wrong ‚Äî the router's `buildProvider` truth is the only honest "active" signal.

---

**Verdict:** The performance shape is fundamentally sound ‚Äî the engine is reused,
the hot path stays O(1), and per-framework breakers bound the herd. But three
material items need spec changes before implementation: the longer chain's
worst-case serial-swap latency on a brownout (the regression the spec is meant to
prevent), the boot probe sharing the router's cache, and the
cartographer-mutates-config false-"unset" trap that would silently no-op the
feature on the dogfood agent.
