# Round 1 — Security review (provider-fallback-default-policy)

Lens: attack surfaces, leaks, privilege escalation, auth on new surfaces, prompt-injection, rotation races, fail-closed integrity. Grounded in `IntelligenceRouter.ts`, `commands/server.ts` (router construction ~4687 + cartographer mutation ~11268), `MessagingToneGate.ts`, `CodexCliIntelligenceProvider.ts`, `intelligenceProviderFactory.ts`, `Config.ts` (binary detection).

## Findings

- **[material]** Boot-order interaction with the cartographer in-place mutation of `componentFrameworks` can defeat the "operator-set" precedence check. The router is constructed at server.ts ~4687 with `resolveConfig: () => config.sessions?.componentFrameworks` (a LIVE read), but later at ~11268 the cartographer block MUTATES the same object in place (`(c.overrides ??= {}).CartographerSweep = routed.framework`), auto-vivifying `config.sessions.componentFrameworks` if it was unset. If the new default resolver decides "operator has NOT set componentFrameworks" by testing object presence/identity at the live `resolveConfig` site (per §4.3/§4.4), this boot-time auto-vivification flips that test AFTER construction, so the computed default silently disappears for the cartographer-enabled path (and any future in-place mutator). This is a fail-toward-Claude correctness/safety hole exactly on the gating path the spec exists to protect. — **Resolution:** Decide "operator set it" from an immutable snapshot captured ONCE at boot BEFORE any mutation (e.g. an `operatorSetComponentFrameworks: boolean` flag or a deep-frozen copy of the raw file value), and have the resolver branch on THAT flag — never on live object presence. Add a unit test: cartographer-enabled + unset componentFrameworks ⇒ computed default still applies.

- **[material]** Making Codex the default provider for ALL gating prompts materially widens the prompt-content exposure surface, and the spec's safety analysis (§6) does not address it. Every gating call (MessagingToneGate, MessageSentinel, ExternalOperationGate, InputGuard, IntentLlmJudge, RelationshipAnomalyScorer, LlmIntentClassifier) sends its FULL prompt — which routinely contains the user's outbound/inbound message text, relationship/PII context, and operation details — to a different vendor's CLI child process by default, for agents that today send none of it there. The exec-json path is hardened (prompt via stdin, scratch-dir hygiene, env allowlist via `buildCodexChildEnv`), but the kill-switch `evaluatePlain` path passes the prompt as a ps-visible positional argv and embeds it in thrown `error.message` (its own SECURITY NOTE). Flipping a default that routes safety-gate content to codex by default + the legacy-path disclosure channel is a real data-residency/exposure change the operator should consciously accept. — **Resolution:** Add an explicit "data-exposure posture" subsection to §6 stating that gating-prompt content (incl. message text + relationship PII) now reaches the Codex/Gemini provider by default, that the default exec-json path keeps the prompt off argv/stderr, and that the `codexExecJson:false` kill-switch re-opens the ps-visible disclosure channel. State that gemini-cli (likewise default-chain) is acceptable for this content class, or scope PII-bearing gates (RelationshipAnomalyScorer) out of the off-Claude default if not.

- **[minor]** "Active framework" probe via `buildProvider`/`detectFrameworkBinary` trusts `$PATH` and `npm config get prefix`, so the primary-provider selection is influenceable by anything that can shape the server's environment or shim a binary earlier on PATH. This is not a NEW surface (the router already builds providers this way) but the spec PROMOTES this probe to author-time authority over which provider serves every safety gate by default — a planted `codex`/`gemini` shim earlier on PATH would be silently selected as the primary gate provider. — **Resolution:** Note in §4.2 that the probe inherits the existing PATH/npm-prefix trust model (already the runtime truth — option (a) is correct precisely because it can't drift from what executes), and that binary-shadowing is an out-of-scope pre-existing threat shared with all framework routing. No new mitigation required if the probe IS the same `buildProvider` truth the router already uses; do NOT add a second `which`-based notion of "active" (option (b)) — a divergent probe would be the spoofable gap.

- **[minor]** Fail-closed is preserved by the engine but the policy resolver must not emit a chain that can degrade-to-Claude on an UNAVAILABLE (not failed) primary and thereby skip the swap. In `evaluate()`, the `!primary` branch (binary missing) degrades to `defaultFramework` (Claude) via the unavailable-primary path BEFORE the failure-swap loop is even reached — the swap loop only runs when `primary` IS resolvable but its `.evaluate()` THROWS. The §4.2 design (primary = first ACTIVE framework) is what makes this safe: because the resolver only ever picks an active primary, `!primary` should never fire for the computed default, so the swap chain governs. If the resolver ever emitted a categories value for an INACTIVE framework, the unavailable-branch would silently route that gate to Claude (defeating the policy) rather than down the chain. — **Resolution:** Make it invariant + tested: the resolver MUST set `categories.*` only to `active[0]` (a built/available framework) and `failureSwap` only to other actives; add a unit assertion that no emitted framework is inactive, so the binary-missing degrade-to-Claude branch is unreachable for the computed default. (Re-throw-when-all-down fail-closed for the runtime-failure path is correctly preserved by the engine — confirmed at IntelligenceRouter.ts:215-217.)

- **[cosmetic]** No new HTTP surface, so no new auth gap. `/intelligence/routing` (the visibility surface) is an existing Bearer-gated diagnostic and is read-only; the policy resolver runs at construction, not behind a route. No secret/credential crosses the swap that didn't already (per-framework providers build their own breaker; `buildCodexChildEnv` already strips `process.env` to an allowlist so OPENAI_API_KEY etc. are not leaked into siblings) — confirmed sound. — **Resolution:** none; note in §6 that the change adds no network surface and no new credential flow (the swap reuses the already-shipped per-framework providers and their existing env hygiene).

## Verdict

The fallback ENGINE's fail-closed + per-framework-breaker security properties are sound and unchanged. The two material items are policy-layer specifics the spec must nail before build: (1) the boot-order/`operator-set` snapshot so an in-place config mutator can't silently disable the default, and (2) an explicit acknowledgement of the gating-prompt data-exposure posture shift (safety-gate content + PII now reaches a different vendor by default). Both are addressable with spec text + tests; neither is a blocker on the mechanism.
