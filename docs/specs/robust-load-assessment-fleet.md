# Robust Load Assessment — fleet-wide + compaction-surviving

## Problem statement

On 2026-06-19 the agent twice judged the machine "under heavy load" and held off on work, based on the
`uptime` 1-minute load average (~40). The robust measurement showed the CPU was actually ~60% IDLE —
the high load average was macOS Spotlight (`mds`/`mdworker_shared`) reindexing the disk after a cold
boot. Load average is the WRONG signal for "can I do work": (a) the 1-min figure is spike-prone, and
(b) on macOS it counts threads stuck in uninterruptible disk I/O, so a high load average can coexist
with a mostly-idle CPU.

The operator (Justin) directed two things:
1. A **robust, durable, go-to default method** for evaluating machine load — so this misjudgment
   cannot recur, baked into structure rather than relying on the agent remembering (Structure >
   Willpower).
2. It must **survive compaction**, not just session start, and be a **fleet-wide capability** (every
   agent), not local to one agent.

A local `load-assess.sh` + a session-start self-knowledge fact already exist on the echo dev agent,
but (a) it is not shipped to other agents, and (b) the session-start hook does NOT re-inject
self-knowledge facts, so the awareness does not reliably survive a mid-session compaction.

## Proposed design

Three parts, all small and well-bounded:

### 1. Ship `load-assess.sh` as a fleet template
Add `src/templates/scripts/load-assess.sh` — the canonical load-assessment script. It reports the
RIGHT signals and emits a verdict (OK / ELEVATED / SATURATED):
- **Real CPU idle%** (sampled via `top -l 2`, second sample — the verdict's primary basis).
- **instar ResourceLedger** time-windowed agent CPU avg/peak/current (`/resources/summary?sinceHours=1`).
- **Per-core load** normalization (context only).
- **Top CPU consumers**, classified agent-work vs external-transient (Spotlight/WindowServer/backupd).

Verdict thresholds: idle <12% = SATURATED, 12–30% = ELEVATED, >30% = OK. **Scope honesty (round-2
codex):** the verdict is a CPU-CAPACITY signal — it does NOT assess memory/swap, thermal throttling,
or disk-I/O saturation, so `OK` means "CPU has headroom," not "everything is fine." The verdict line
says so, so a reader does not over-trust it. (Memory is already covered separately by the
ResourceLedger/reaper-pressure surfaces; this script's job is to kill the load-average misread, not to
be a universal health oracle.) Load average is shown as CONTEXT ONLY with the explicit caveat that it
is spike-prone + I/O-inflated. `--json` is **human-diagnostic only** (unversioned; no consumer today —
a future programmatic consumer must add a `schemaVersion` first, not assume the current shape).
Fail-soft, two axes: (1) server down → degrade to local-only (CPU idle% + top), never error.
(2) **Cross-platform (round-1 catch — `top -l 2` is macOS-only):** detect OS — macOS uses `top -l 2`,
Linux falls back to a **TWO-SAMPLE `/proc/stat` delta** (read, sleep ~1s, read; idle% = Δidle/Δtotal —
a SINGLE `/proc/stat` read cannot compute idle%, per round-2 codex). Avoid hard deps: no `bc`/`jq`
required (use awk/shell arithmetic). If neither CPU read works, fall back to the ResourceLedger +
loadavg-only and say so in the verdict. The script must degrade cleanly on a non-mac agent, never
crash. (instar's fleet is macOS-primary today, but "fleet-wide" must not assume it.)

### 2. Wire a load-assess AWARENESS block into the IN-CODE session-start hook (compaction-surviving)
**CORRECTED after round-1 review:** the deployed session-start hook is NOT sourced from the template
file `src/templates/hooks/session-start.sh` (that file is dead for this path). It is generated by the
IN-CODE method `PostUpdateMigrator.getSessionStartHook()` (PostUpdateMigrator.ts:8917), written to
`.instar/hooks/session-start.sh` by `migrateHooks()` (line 3085), which **always overwrites** built-in
hooks — so editing the in-code method reaches both new agents (init) and existing agents (migration)
with no new migration code.

**CRITICAL mechanics (round-1 catch):** `getSessionStartHook()` delegates the `compact` event with
`if [ "$EVENT" = "compact" ]; then exec bash compaction-recovery.sh; fi`. `exec` **replaces the
process** — control never returns, so a block added near the hook's tail fires on startup/resume/clear
but NEVER on compact. To deliver the compaction-survival guarantee (the operator's core ask), the
awareness block MUST be inserted **ABOVE the compact-delegate branch**, right after the `EVENT=...`
line — so it prints on EVERY event (its stdout is emitted before the `exec` on the compact path). One
insertion point covers startup/resume/clear AND compact.

The block (static, no network, no dev-gate entanglement — fleet-safe, cannot fail or blow the 5s hook
timeout):

> `--- MACHINE LOAD ---` / "To assess machine load, run `.instar/scripts/load-assess.sh` (--json to
> parse). NEVER judge load from `uptime` 1-min load average — spike-prone + macOS Spotlight-I/O-
> inflated (a high load avg can coexist with an idle CPU)."

(Belt-and-suspenders option weighed and rejected as redundant: also emitting from
`getCompactionRecovery()`. The above-the-exec insertion already covers compact, so a second copy would
just duplicate the line on the compact path. Single insertion above the branch is the clean answer.)

### 3. Migration parity — existing agents receive both on update
- `PostUpdateMigrator.migrateScripts()` installs `load-assess.sh` into `.instar/scripts/` (mirror the
  existing `secret-get.mjs` install at PostUpdateMigrator.ts:7328 — `loadRelayTemplate` → write →
  `chmod 0755` → record in `result.upgraded`). Idempotent (overwrite-on-migration, like other
  built-in scripts, so a fixed template always reaches deployed agents).
- The session-start hook is a built-in hook, already overwritten on every migration by
  `migrateHooks()` — so the awareness-block edit propagates to existing agents automatically with no
  new migration code.

### 4. Agent-Awareness (CLAUDE.md template) — added after round-1 review
Per the Agent-Awareness Standard ("a feature an agent doesn't know about effectively doesn't exist"),
add a short **Machine Load Assessment** capability entry to `generateClaudeMd()`
(src/scaffold/templates.ts → routed via PostUpdateMigrator) — a building-block + proactive trigger:
"before deferring work for load, run `.instar/scripts/load-assess.sh`; NEVER judge load from
`uptime`." Migration-Parity corollary: add a content-sniffed `migrateClaudeMd()` insert
(PostUpdateMigrator.ts:3968) so EXISTING agents' CLAUDE.md receives the section, not just new installs.
(The runtime hook block from Part 2 is the always-present reminder; the CLAUDE.md entry is the
discoverable capability doc — the standard requires both.)

## Decision points touched
No new block/allow/route gates. This adds an observe-only diagnostic script + a static awareness
string + a script-install migration. It changes NO authority, NO messaging, NO external side-effect.

## Frontloaded Decisions
- **Static awareness block, not a self-knowledge fetch (D1).** The hook injects a fixed string, not a
  live `/self-knowledge/session-context` fetch. Rationale: simpler, fleet-safe, cannot fail or block
  the 5s-timeout hook, and avoids the developmentAgent-gated self-knowledge surface. Cheap-to-change:
  the exact wording.
- **Script is observe-only (D2).** `load-assess.sh` reads CPU/ledger and prints; it mutates nothing,
  gates nothing. No authority, never cheap-vs-not question — it is read-only by construction.
- **Verdict thresholds (D3) are cheap-to-change** config constants in the script (idle% cutoffs).

## Multi-machine posture
- `load-assess.sh` is **MACHINE-LOCAL BY DESIGN** — load is a property of the machine it runs on;
  there is nothing to replicate. (It already reads the local ResourceLedger + local CPU.) The
  existing `/guards?scope=pool` and `/resources/summary` cover cross-machine reads separately.
- The awareness block is static template content installed per-machine — identical on every machine,
  no replication needed.

## Rollout
No dark/dry-run needed — it is observe-only (a diagnostic script + a static doc string). It ships
**live on EVERY agent (including echo) immediately from migration** — there is no dev gate and no dark
phase, because there is no action-bearing surface to gate (matches `secret-get.mjs` + the session-start
hook itself). echo's existing hand-built local `load-assess.sh` is superseded by the shipped template
(overwrite-on-migration). (Note the inverse of the usual "dark for fleet, live on me" — this is "live
everywhere" because it is a read-only diagnostic.)

## Testing (Testing Integrity Standard)
- **Unit:** `load-assess.sh --json` returns valid JSON with `verdict` ∈ {OK,ELEVATED,SATURATED} and a
  numeric `cpuIdlePercent`; verdict logic maps idle% to the correct tier at the boundaries (11/12,
  29/30); fail-soft when the ResourceLedger endpoint is unreachable (ledger=unavailable, still emits a
  verdict from local CPU).
- **Unit (portability):** the script emits a valid verdict on a Linux-shaped environment (no
  `top -l`), degrading to the ledger+loadavg path — never a non-zero exit / crash.
- **Integration (the compaction guarantee — the load-bearing test):** run `getSessionStartHook()`'s
  output with `CLAUDE_HOOK_MATCHER=compact` and assert the `MACHINE LOAD` block IS present — i.e. the
  block is emitted BEFORE the `exec compaction-recovery.sh` delegation. This test is what catches the
  round-1 bug (a tail-placed block would fail it). Also assert it appears for `startup`.
- **Migration / E2E:** after `PostUpdateMigrator` runs against a fixture agent home lacking
  `load-assess.sh`, the script is present + executable in `.instar/scripts/`, the regenerated
  `session-start.sh` contains the awareness block above the compact branch, and (Agent-Awareness)
  `migrateClaudeMd()` inserts the Machine Load Assessment section into an existing CLAUDE.md.
  Idempotent on re-run (content-sniffed, no duplicate inserts).
- **Wiring-integrity:** `migrateScripts()` actually writes load-assess.sh (not a no-op);
  `getSessionStartHook()` actually contains the block above the compact `exec`; `generateClaudeMd()`
  output contains the capability entry.

## Out of scope (tracked separately)
The broader "ALL self-knowledge facts survive compaction" generalization is explicitly OUT OF SCOPE
here and tracked separately; this ships the load-assess awareness via a static block, which fully
