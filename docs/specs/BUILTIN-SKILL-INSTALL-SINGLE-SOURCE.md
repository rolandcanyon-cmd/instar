# Built-in Skill Install â€” Single Source of Truth

> v3 â€” after convergence round 2 (3 internal + gemini external). Round 2 found the
> decisive issue: `src/data/builtin-manifest.json` is **auto-generated** by
> `scripts/generate-builtin-manifest.cjs` (`scanSkills()` does `readdirSync('.claude/skills')`
> + `sha256File`, run on every `npm run build`). So the manifest is a *derived index of
> whatever sits in the dev agent's live `.claude/skills/`*, not an authored source â€” and it
> is **already polluted**: 16 of its 22 `skill:*` rows point at untracked, non-shipped files.
> v3 reframes around an **authored allowlist** as the true single source, makes the generator
> *validate* against it, and makes the npm-pack + git-tracked assertions the load-bearing
> ratchet.

## Problem statement

Instar ships built-in skills to each agent's `.claude/skills/` so they're invokable as slash
commands. Today this is driven by **two disjoint sources**, plus a third tracked source that
never ships, plus a generated manifest that has silently drifted from all of them:

1. **`installBuiltinSkills()` in `src/commands/init.ts`** â€” a hand-maintained inline dictionary
   of 16 skills as TypeScript strings (install-time `${INSTAR_PORT:-${port}}` substitution),
   plus `build`/`autonomous` via dedicated file-copy functions.
2. **The root `skills/` directory** â€” 14 git-tracked static-file skills (incl. `spec-converge`,
   `instar-dev`). **No code installs them; the dir is not in `package.json files[]`, so it never
   ships.**
3. **`src/data/builtin-manifest.json`** â€” auto-generated by scanning the live `.claude/skills/`
   dir at build time. Because the inline-dict skills are *written to disk at runtime* by
   `installBuiltinSkills()`, the generator scooped up those untracked runtime copies: **16 of 22
   `skill:*` rows have `sourcePath`s that are untracked and absent from `files[]`** â€” phantom
   rows that fail `npm pack`.

Net effect: the headline dev tools (`spec-converge`, `instar-dev`, `systematic-debugging`)
reach no agent through any supported path; on the *dev agent only*, `.claude/skills/spec-converge`
exists as an **untracked local bootstrap copy** (hand-placed during this work) that masks the gap.

Constitutional roots: **Agent Awareness** (an uninstalled skill is the hardest form of
"a capability it doesn't know it has") and **Framework-Agnostic â€” "one shared source of truth,
never hand-maintained per engine"** (the inline dict diverged from `skills/`, and the generated
manifest diverged from both).

Operator goal: **every Instar agent has the complete capability to develop Instar and leverage the
same tools Instar uses to develop itself.**

## Proposed design

### D1 â€” The authored allowlist is the single source of truth

The true source of truth is a small, **authored, committed allowlist** of canonical built-in skill
slugs (a `BUILTIN_SKILLS` constant in `src/data/`, or `src/data/builtin-skills.json`). Each entry:
`{ slug, devOnly?: boolean }`. This is the one hand-edited list; everything else is derived from or
validated against it.

- **The files** live as tracked skill dirs under `.claude/skills/<slug>/` (the established bundled
  location â€” `build`/`autonomous`/`setup-wizard`/`secret-setup` already live there and are in
  `files[]`). The 10 canonical `skills/` entries MOVE here; the 16 inline-dict skills are
  **materialized as tracked files** here (deleting the TS-string source and the untracked runtime
  copies that polluted the manifest).
- **Packaging:** replace the four per-dir `files[]` lines with a single glob `.claude/skills`
  (ship the whole dir) â€” removing the per-skill omission surface. Safe ONLY in tandem with the
  no-untracked-residue invariant (D4.3): npm packs from the filesystem, so an untracked stray dir
  would otherwise ship.
- **The manifest** (`builtin-manifest.json`) stays a **generated index**, but `scanSkills()` is
  changed from *discovery* to *validation*: it reads the authored allowlist and **fails the build**
  if (a) a live `.claude/skills/` dir is not on the allowlist (stray/untracked phantom), or (b) an
  allowlisted slug has no tracked source dir. The manifest then carries, per skill: `slug`,
  `sourcePath`, `contentHash`, `previousHashes: string[]` (D7), `builtin: true`, `devOnly`.
- **Installer:** `installBuiltinSkills()` installs every allowlisted skill by copying its bundled
  `.claude/skills/<slug>/` into the target, substituting a `{{INSTAR_PORT}}` placeholder at copy
  time (deterministic â€” replaces the unverified "shell expands at runtime" claim). The 3 install-set
  files that hardcode `localhost:4040` (`instar-identity`, `instar-scheduler`, `knowledge-base`) are
  rewritten to the placeholder during the move.

### D2 â€” Frozen canonical set (reconciled against the live 22-row manifest)

The allowlist = the frozen set. Resolved now (round 1 required freezing; round 2 required reconciling
with the existing 22 `skill:*` rows, which include `instar-project` and `test-as-self`):

**INSTALL (allowlisted):**
- The 16 inline skills (now tracked files) + `build` + `autonomous` + the already-bundled
  `setup-wizard`, `secret-setup`, `instar-project`, `test-as-self`.
- From `skills/`: `spec-converge`, `instar-dev`, `systematic-debugging`, `smart-web-fetch`,
  `knowledge-base`, `instar-scheduler`, `agent-memory`, `agent-identity`, `instar-identity`,
  `credential-leak-detector`. Rationale: dev tools, or capability/teaching surfaces with no
  installed-skill superseder (KnowledgeManager exposes 0 `/knowledge` routes; the jobs API is the
  scheduler mechanism but this skill is its only teaching surface; no built-in credential hook
  exists). Ambiguous rows default to KEEP (losing a teaching surface is the costlier error).
- `devOnly: true` candidates (only relevant if D6 chooses dev-gating): `spec-converge`,
  `instar-dev`, `systematic-debugging`.

**DOCUMENT-ONLY (kept in-repo, NOT installed, each with a cited superseder):**
- `instar-feedback` â†’ installed `feedback` (Rising Tide). `instar-telegram` â†’ `telegram-reply.sh` +
  CLAUDE.md relay. `instar-session` â†’ `/sessions` API + SessionManager. `command-guard` â†’
  `dangerous-command-guard.sh` hook.

### D3 â€” Migration parity via the EXISTING migrator

`PostUpdateMigrator.migrateBuiltinSkills()` (`:1236`, registered `:186`) already calls
`installBuiltinSkills()` non-destructively each migration. Once the installer installs the
allowlisted set, existing agents get the new skills through it â€” no net-new migrator for the
install-if-missing case (Migration Parity mechanism #5). A dedicated migrator is reserved only for
the content-UPDATE case (D7).

### D4 â€” Load-bearing ratchet (CI-executed config, not quarantined)

Round 2: manifestâź·bundle parity is a tautology (the manifest is generated from the bundle). The real
assertions:
1. **Allowlist âź· live dir:** every allowlisted slug has a tracked `.claude/skills/<slug>/SKILL.md`;
   every live `.claude/skills/` dir is on the allowlist. (This is the generator's validation pass,
   also asserted as a test.)
2. **Package presence:** every allowlisted slug appears in `npm pack --dry-run` output (catches
   "tracked-but-not-shipped" directly â€” the class that bit the 16 phantom rows).
3. **No untracked residue:** `git status --porcelain .claude/skills/` reports no untracked dir
   (else the glob would ship a phantom). 
4. **Content lint:** no bundled skill contains a bare `localhost:<port>` (only `{{INSTAR_PORT}}`).
5. **Wiring-integrity ("feature is alive"):** run `installBuiltinSkills()` into a tmp dir; assert
   each allowlisted `SKILL.md` materializes on disk.

**Testing tiers (signed):** Tier 1 = assertions 1/4/5 (unit + lint + install-materialization) plus
a unit test of the generator's validation-fail path. Tier 2 (HTTP route) and Tier 3 (503/200
lifecycle) are **signed N/A** â€” this feature exposes no route and no server-init path (see Decision
points). Assertions 2/3 are CI/package-integrity gates.

### D5 â€” Framework-agnostic by construction

The authored allowlist is the engine-neutral single source. Round 2 confirmed the provider
scaffolders' `bundledAssets` feed has **no producer** (Codex gets an empty skills dir; no Gemini/pi
scaffolder). v3 commits: the allowlist is the one producer the per-engine scaffolders read. Claude-Code
wiring ships in this spec; wiring Codex (`bundledAssets`) + future Gemini/pi to the same allowlist is a
**tracked staged follow-on** â€” a real commitment is opened (its id recorded in this section before
build, not asserted) â€” under the hard constraint that **no second hand-maintained per-engine list is
ever created**.

### D6 â€” Scope: fleet-wide vs dev-gated (OPERATOR DECISION)

The canonical set installs to EVERY agent unless gated. Operator goal points to **fleet-wide**
(default, built toward). Security's least-privilege counter: gate the `devOnly: true` skills
(`spec-converge`, `instar-dev`, `systematic-debugging`) behind the existing `developmentAgent` flag â€”
which the `devOnly` allowlist flag (D1) makes a one-line installer conditional. Surfaced in the
convergence report for the operator's explicit call. Bounded either way (install-if-missing +
overwrite-by-marker).

### D7 â€” Provenance + overwrite (resolves OQ#4)

Built-in and user skills share the flat `.claude/skills/<name>/` namespace, so "fix stale built-ins"
and "never touch user skills" conflict without provenance. v3: each canonical skill carries
`metadata.builtin: true`; the manifest carries `contentHash` + **`previousHashes: string[]`** (the
history of all versions ever shipped). Overwrite policy (mirrors the always-overwrite-built-in-hooks
lesson):
- Overwrite a canonical skill on migration ONLY if its on-disk content matches the current OR any
  `previousHashes` entry (an unmodified built-in, even across several versions).
- A skill whose content matches none of those (user-edited) or lacks the `builtin` marker
  (user-authored) is **never touched**.
This fixes stale built-ins without clobbering user work, and is the content-update path D3 reserves.

## Decision points touched

- New authored allowlist (`src/data/builtin-skills.json` or a TS const).
- `scripts/generate-builtin-manifest.cjs` â€” `scanSkills()` becomes validation-against-allowlist
  (fail on drift), emits `previousHashes`/`builtin`/`devOnly`.
- `installBuiltinSkills()` (`src/commands/init.ts`) â€” installs the allowlist set from bundled
  `.claude/skills/` with `{{INSTAR_PORT}}` substitution; preserves the `claudeEnabled` gate.
- `package.json files[]` â€” single `.claude/skills` glob (replacing the 4 per-dir lines).
- `PostUpdateMigrator` â€” no new install-if-missing migrator; a content-update migrator per D7.
- CI â€” the D4 ratchet (5 assertions) in the CI-executed config.
- The 16 inline-dict skills are relocated to tracked files; untracked `.claude/skills/` residue is
  removed (incl. the bootstrap `spec-converge` copy, so the installer path is genuinely exercised).
- No runtime/message/session-lifecycle gate. No new auth surface.

## Build sequencing & reconciliation notes (folded from round-3 findings; non-design)

These are implementation-ordering details round 3 surfaced â€” not design changes, but the
build MUST honor them or the first CI run red-screens:

- **B1 â€” Atomic ordering (round-3 M1).** In ONE commit: (a) materialize all 16 inline-dict
  skills + move the 10 `skills/` entries into tracked `.claude/skills/<slug>/` dirs and
  `git add` them; (b) flip `files[]` to the `.claude/skills` glob; (c) regenerate the
  manifest. A materialized-but-unstaged dir is "untracked residue" â†’ D4.3 fails; flipping the
  glob first ships nothing. Do all three together.
- **B2 â€” Canonical casing (round-3 M2).** The current generated manifest rows use lowercase
  `.../skill.md`, which resolves only on the dev agent's case-insensitive macOS FS; on the
  case-sensitive Linux CI runner `sha256File('.../skill.md')` ENOENTs (git tracks `SKILL.md`).
  The `scanSkills()` rewrite MUST read/emit canonical uppercase `SKILL.md`, and the D4 Tier-1
  validation test MUST run in CI (case-sensitive), not only locally.
- **B3 â€” Full current-disk reconciliation (round-3 M3 + N1).** The live `.claude/skills/`
  already holds ~27 dirs, and ALL of the generated manifest's skill rows currently point at
  untracked files (only 5 dirs are git-tracked). The authored allowlist must enumerate EVERY
  legitimate currently-shipped skill â€” including ones D2 didn't name (`agent-passport`,
  `agent-readiness`, `iterative-converging-audit`, `setup-wizard`, `secret-setup`,
  `instar-project`, `test-as-self`, â€¦) â€” classifying each as KEEP-into-allowlist or
  REMOVE-as-residue (e.g. the untracked `spec-converge` bootstrap copy, any `fork-and-fix`
  stray). The allowlist-validation pass (D4.1) is green on day one only if this reconciliation
  is complete. The frozen set (D2) is therefore the FULL legitimate built-in roster, not just
  the dev-tool additions.
