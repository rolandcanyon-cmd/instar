# Self-Action Backpressure — plain-English overview

Instar is good at asking "is this one action correct?" and bad at asking "is the agent about to do this same kind of action a thousand times in a row?" Every time the agent does something to *itself* — spawn a helper, send a notice, restart a session, swap to another account, retry a failed call — that action was individually sensible but had no shared brake on *how often* it could fire under pressure. So over and over (20 separate incidents since April) a sensible action turned into a runaway loop: a fork-bomb of helper processes, a flood of notifications, a session swapping accounts 72 times a day. Each incident got its own one-off patch afterward. This spec builds the brake that was missing: **one shared chokepoint every self-triggered action passes through before it's allowed to fire.**

## What it is

We add a single in-process gatekeeper called the **SelfActionGovernor**. Before any self-triggered action fires, it "asks permission" by calling `admit()`. The governor looks at how fast that kind of action has been firing, whether it's making things better or worse, and whether the system is already under strain — then answers one of three ways:

- **allow** — go ahead (and it hands back a one-time token the action must show to actually fire, so nothing can "ask permission, get told no, and do it anyway").
- **coalesce** — don't fire a fresh one; fold into a running/aggregated action instead (this is how notifications already avoid spamming you).
- **queue** — not now, but *not dropped* — hold it and fire it when it's safe (so a real piece of work is never silently lost).

The key rule: it **never silently throws work away**. The worst it does is delay or merge.

## Why it matters / what changes

Nothing changes for you day-to-day when things are healthy — the brake only bites under *sustained* pressure, exactly like the existing fork-bomb cap. What changes is that the *next* new self-action someone builds inherits a safety bound automatically, instead of being unbounded until its own incident teaches us to add a patch. It closes a whole class of "the agent got stuck doing the same thing too fast" failures by construction.

## The main tradeoffs / honest caveats

- **This bounds the swap-thrash problem; it does not cure it.** Capping how often the agent swaps accounts stops the thrash from spiraling, but the real fix (making a swap not require a disruptive restart) is a separate follow-on piece of work. We say that plainly rather than overselling this.
- **A safety brake must not become the thing that breaks you.** So if the governor itself errors, cost-heavy/safety-critical actions fail *safe* (hold/queue) while harmless notification actions fail *open* — the direction is chosen per action-type, never a blanket "just allow everything," which would let the runaway proceed during exactly the storm we're guarding against.
- **Relief actions get a faster lane — but never an unlimited one.** Some self-actions *reduce* pressure (killing a stuck process, recovering a dead session). Rate-limiting those under pressure would deadlock the system (high pressure → block the thing that relieves pressure → pressure stays high), so a genuinely-effective relief action gets its *speed limit* relaxed. But there's still a hard cap on how many times the *same* target can be hit: a kill that keeps getting re-fired against the same thing (because that thing keeps coming back) is caught by that cap — so "I killed it" is never mistaken for "it's actually fixed." Reviving a genuinely-dead important job is treated even more gently (it's never bounded to death; a separate existing safety net decides when to give up), because losing that work is worse than retrying it.
- **It ships carefully.** First it runs in "watch only" mode — recording what it *would* have blocked, blocking nothing — so we can measure it before any action-type flips to actually enforcing. If someone turns the whole thing off in an emergency, that shows up on the safety dashboard *immediately* (a switched-off guard is itself treated as an incident) — and quietly cranking a limit to a meaningless number is made just as visible as switching it off.

## What the fifth review round added (the big ones, plainly)

- **Your emergency stop always wins — and only a real human counts.** The same "kill this session" plumbing serves both the agent's automatic cleanup *and* your dashboard kill / "stop everything" command. Reviewers caught that once the brake enforces, it could technically have held up *your* emergency stop during a storm — exactly backwards. Now every action carries a tag saying whether a human or the agent asked for it, and human-initiated actions bypass every limit (logged, never blocked). A sixth review round then hardened the tag itself: "human" requires a genuinely human credential (your dashboard PIN or a verified sender) — the server's internal API token, which the agent itself holds, does NOT count, so the agent can never stamp its own actions as yours.
- **The brake now survives restarts — even rapid-fire ones.** The storms this guards against are exactly the ones that crash the process — and a crash used to wipe the brake's memory, handing the runaway loop a fresh budget every bounce. The counting state is now saved and reloaded across restarts, saved *on meaningful change* rather than only on a timer (so even a crash-loop faster than the save interval can't outrun it), and a missing or damaged state file triggers a cautious posture plus a loud audit note instead of a silent fresh budget.
- **A delayed kill can't hit an innocent bystander.** If a "kill this stuck session" order waits in the queue and meanwhile the stuck session died and a healthy new one took its name, firing the stale order would kill the healthy one. Held actions now re-check "is this still the right target, and is the reason still true?" before firing — a mismatch means the order is dropped (loudly), not fired blind.
- **We almost rebuilt something that already exists.** The design said "no shared cross-machine quota-ledger exists, so build one." A reviewer re-checked against the *deployed* code: it shipped two versions ago. The design now reuses that shipped machinery instead of building a duplicate (two bookkeepers for one budget is how books drift). Fittingly, this is the exact "acting on a stale copy of the facts" disease this whole effort is about — caught inside our own review. The sixth round then added two honest footnotes: the shipped machinery is real math but not yet *wired* (three concrete wiring jobs remain and are named), and its ledger never cleans up after itself — a bug we'd have inherited; the cleanup requirement is now part of the plan, along with a rule that only ONE bookkeeper per account can ever exist no matter which features are switched on.
- **Mass cleanups scale their own budget — with the input guarded.** A fixed cap like "60 kills per hour" can't tell a runaway apart from a legitimate big cleanup (200 stuck sessions genuinely need 200 kills). The cap now scales with how many live targets actually exist — "never remove more than a slice of the population per window" — and if it still has to slow things down, it does so loudly and the manual override is spelled out. The sixth round guarded the population count itself: it comes from an independent source (never the buggy component's own list), can only widen the budget when fresh and trustworthy, and is capped by an absolute ceiling no miscount can exceed.
- **The privileged fast lanes got a bouncer.** A couple of action types legitimately bypass the caps (reviving dead work, liveness heartbeats). Reviewers showed any new code could simply *claim* to be one of those and inherit the unlimited lane. Membership is now a hard-coded, build-checked list, and each member must name the *other* safety net that catches it — proven by a test, not prose.
- **It nags you less, not more.** Any alert the brake itself raises now follows the house rule: try to self-heal first, tell the human only when self-healing failed (or immediately, for genuinely critical things). A hiccup that fixes itself is a log line, not a ping.

## Open questions

**None left open** — the three questions that were on the table have all been decided during review:

- **How enforcement turns on:** each action-type flips from watch-only to enforcing *one at a time*, once its own safety test passes (not all at once). The actual flip is always a deliberate human action.
- **Account-quota shared across machines:** account-quota is shared across the agent's machines, so two machines could each lean on the same account. The decision: on a **single machine**, the local cap *is* the complete cap, so it can enforce right away. On a **multi-machine fleet**, the cross-machine cap now *reuses the already-shipped quota-ledger machinery* (behind its own new switch), and the swap/notify caps stay in **watch-only** mode until that wiring is reviewed and turned on. We state that plainly rather than implying fleet swap-thrash is already bounded.
- **Reactive swaps:** a swap triggered by a rate-limit event is *held in a queue*, not dropped, so no real work is lost.

## The honest bottom line

This bounds a whole *class* of "the agent got stuck doing the same thing too fast" failures, and makes every *future* self-action inherit a safety cap automatically. It does **not**, by itself, cure the specific swap-thrash problem on a multi-machine fleet (that needs a separate piece), and it says so out loud rather than overselling.
