# Calm Transient-Episode Alerting (machine-coherence + rope probe)

**Operator directive (topic 29836, 2026-07-11):** "Can you look into all of the alerts I've been getting? Either they are too noisy or this needs to be fixed. Also, we should take this opportunity to review why these issues are being handled/fixed internally automatically, and how we can upgrade our system to handle them automatically (if they are real issues)."

**Glossary (one line each):** *raiser* ‚Äî the one machine elected to speak for the machine-coherence guard (all machines compute; one narrates). *rope* ‚Äî one transport link (tailscale/lan/cloudflare) between two of the agent's machines. *Attention hub* ‚Äî the single üîî Telegram topic all alerts post into. *reopen latch* ‚Äî existing per-item damper that silences reopen chatter after 3 reopens/hour. *dev-agent gate* ‚Äî `resolveDevAgentGate`: a flag omitted in config resolves LIVE on the development agent, DARK on the fleet. *speaks()* ‚Äî the episode manager's check that only the elected raiser emits user-facing effects.

## Problem statement

The operator's üîî Attention hub received a burst of HIGH machine-coherence alerts + NORMAL rope-probe notices on 2026-07-11 (9 machine-coherence-guard items since 7/10; 8 of 13 pool items raised by the Laptop ‚Äî the NON-lease-holder). **Every flagged condition was real AND self-healed within ~30 minutes** ‚Äî version skew during a routine rolling auto-update, and rope latency demotion while the Laptop's mesh links warmed up after rejoining.

Verified mechanics (from source, v1.3.827):

- `classifyVersionSkew` grace-gates patch-only skew behind `versionSkewGraceMs` (45 min); major-minor confirms in 2 ticks.
- **The grace/ceiling clocks cannot survive a version advance**: row identity embeds every participant's version value, so any advance mints a NEW identity ‚Äî `updateConfirmation` deletes the old row and mints a fresh `firstSeenAtMs`. A "crawling laggard" (one patch per 25 min) NEVER accumulates 45 min of continuous row age.
- Further reset vectors: `rowState.clear()` + reconcile skipped entirely when online comparable members dip below 2 (sentinel early-return); raiser restart/handoff (update waves restart the raiser; election is per-tick); sentinel warmup (~4 ticks) freezes reconcile post-boot.
- The effects consumer (an inline closure in server.ts) **hardcodes `priority: 'HIGH'`** for every raise. `createAttentionItem` silently no-ops on a duplicate item id. In the default single-topic hub mode **priority is buzz-inert** (`routeToAttentionHub` never sets `silent` ‚Äî priority is label/dashboard/legacy-mode semantics only).
- On heal, `closeEpisode` unconditionally emits a `resolve` effect ‚Üí a second, NOTIFYING hub message; nothing bounds resolve volume for a cycling condition. The reopen path's `raise` with a reused id is silently swallowed (the operator is never told a divergence returned).
- `maybeProposeFix` auto-creates `pendingFix` at raise (no operator involved) and every fix path clears it ‚Äî pendingFix presence at close is the wrong "operator interacted" signal in both directions.
- The rope prober's slow-but-alive escalation is informational by its own text; exhaustion items re-arm per episode (`openedAt` in the id) ‚Äî an oscillating rope can post ~24 hub items/day. The `rope-health-digest` job runs scheduler-side (lease-holder only), reports only that machine's own directional view, delivers only when `digestTopicId` is set (default UNSET ‚áí log-only), and `composeDigest()` today has NO condition class for a slow-but-alive (in-episode, not-reclaimed) rope at all.
- The episode manager's durable file is transitions-only-written, atomic, `version: 1`, lenient to unknown fields; the manager's reconcile input today carries only CONFIRMED rows. Non-raiser machines DO run reconcile (speaks() gates effects only). The attention store has no retention ‚Äî an unresolved item persists forever.

The system IS self-healing correctly. The defect is narration: it buzzes at decision priority on states it is mid-way through healing, speaks twice per episode, and its loud-escalation backstops are anchored on clocks that reset under the exact churn they measure.

## Design principles

1. **Alert = action needed or persistent.** A condition the system predictably heals is not operator-actionable; only a STALLED heal is. This is **Self-Heal Before Notify** for the version dimension: the heal is the auto-updater; observable version advance is `selfHealAttempted`; stall past the ceiling is `selfHealExhausted`; the ceiling is the `max-notification-latency` backstop ‚Äî and per that standard, the operator-facing BUZZ sits downstream of heal exhaustion, never on raw detection.
2. **A self-healing transient produces ZERO notifying messages; the first buzz is the first actionable moment** (Near-Silent Notifications: "if the message has to say 'no action needed,' it must not be sent"). Everything is still VISIBLE ‚Äî silent hub posts, the attention store, the dashboard, counters, jsonl ‚Äî the awareness surface is pull, not push. Loud paths: major-minor, stall ceiling, flap brake, rope exhaustion, interacted close-outs, escalated-episode close-outs, and the cross-key wave backstop. Scope honesty: reopen-appends of a calm-class episode are silent; appends/escalations on loud-class episodes notify as today.
3. **Signal-only changes.** Detection, episode lifecycle audit, duplicate-reconciliation, and the fix-it/leave-it consent authority are untouched. What changes: when confirmation happens (M-P1), what priority/copy/notification mode narration carries (M-P2), where informational rope content lands (M-P3).
4. **Never quieter than the operator can still see somewhere deliverable.** Every predicate failure falls back to LOUDER; every demotion has a live deliverability check or a documented accepted consequence.

## Proposed design

### M-P0 ‚Äî Durable, identity-independent skew anchors

*Design alternative considered and rejected:* an append-only event log with reducers (event-sourced episodes/eligibility/digest) would also solve restart/handoff correctness, but it replaces the episode manager's existing durable-state model wholesale ‚Äî a foundation rewrite for a narration change, contradicting principle 3 (signal-only) and multiplying blast radius. The anchors extend the existing model additively instead.

New tracking, outside the identity-keyed `rowState` and outside the <2-online clear:

- **Anchor keying:** per `(dimension|key)` (e.g. `version|instarVersion`) ‚Äî identity WITHOUT value classes, so version advances / value churn never re-key it.
- **Input path (named):** `EpisodeReconcileInput` is extended with the tick's RAW divergent rows (pre-confirmation), so the episode manager ‚Äî which owns the durable file ‚Äî both computes and persists anchors in one place. (The sentinel's confirmation engine continues to own confirm/grace exactly as today; it reads the anchor values the manager exposes.)
- **Fields per key:** `skewOnsetAtMs` (first appearance of ANY skew on the key), `activeSkewMs` (the accumulator ‚Äî see clock model), `anchorLastEvaluatedAtMs`, per-machine `{ lastSeenVersion, lastAdvanceAtMs }` (version dimension), flap history (see below) clamped to the most recent 64 events, `lastStalledFireAtMs`, `lastRecurringFireAtMs`.
- **Clock model (suspension is computable):** clocks are the machine-local wall clock at advert observation. The key's `activeSkewMs` accumulates `min(now ‚àí anchorLastEvaluatedAtMs, K √ó tickMs)` (K small, e.g. 4), **clamped to `[0, K √ó tickMs]`** (a wall-clock regression can never decrement), and **credits ONLY reconciles in which the key is observed divergent AND every skew participant is present in the compared set** ‚Äî a suspended anchor (participant departed) freezes BOTH the clear and the accumulator, so a ‚â•3-pool laggard offline for hours cannot tick its own ceiling while its heal is unobservable. Gaps longer than K ticks ‚Äî offline dips, <2-online windows, sentinel restarts, warmup ‚Äî contribute at most K ticks, retroactively treating the gap as suspended. Grace and the stall ceiling compare against `activeSkewMs`, never wall-clock-since-onset. Advances can NOT reset onset or the accumulator.
- **Participant-aware clear (never vacuous):** convergence-clear requires EVERY machine that participated in the key's skew to be PRESENT in the compared set AND agreeing, sustained for `resolveTicks`; a skew participant leaving the compared set (offline / stale / unverifiable) suspends the anchor regardless of how many other members remain, and a compared set of <2 can never clear an anchor. (This is strictly stronger than the <2-online rule and mirrors the episode manager's participant-departure suspension ‚Äî closing the ‚â•3-pool "blinking laggard resets the ceiling" hole.)
- **Flap history + the flap event definition (pinned):** a flap event = a confirm transition on the key FOLLOWED BY anchor-level genuine convergence (the participant-aware sustained clear). Intra-skew identity re-mints and bare re-confirms never complete a cycle ‚Äî a never-healed condition belongs to the stall path, not the flap path.
- **Progress definition (version dimension):** an "advance" counts only when the lagging machine's version moves strictly toward the pool's maximum observed version (numeric `major.minor.patch` triplet comparison; prerelease/build metadata ignored for ordering; unparseable ‚áí never an advance). The pool max is recomputed each tick from current adverts ‚Äî a leader rollback shrinks the gap without laggard action (safe: smaller skew or convergence); a laggard regression is not an advance.
- **Persistence discipline:** anchors are new fields in the episode manager's EXISTING durable per-machine file (`version: 1` unchanged; strictly additive ‚Äî the reader is lenient to unknown fields, and a binary rollback treats them as inert). Writes remain transitions-only: anchor fields persist on MATERIAL change (onset set/clear, an observed version change, a flap event, a derived-raise latch, or an `activeSkewMs` bucket boundary of ‚â•5 min) ‚Äî a tick observing no change writes nothing.
- **Lifecycle:** EVERY per-key structure (anchor, progress state, flap history, latches) is dropped only after SUSTAINED absence ‚Äî the key absent from the compared set for ‚â•24 h (covering the retire-return cycles that update waves and registry rebuilds cause; a transiently-departed key's latches survive their remaining window, so churn correlated with the tracked wave cannot re-arm a second `:stalled` HIGH inside the latch window). M6-suppressed flag rows SUSPEND their anchor, never retire it; the raw rows feeding anchors are the POST-suppression set. Bounds: keyspace ‚â§ the compared dimensions + manifest intersection (‚â§64 by the existing clamp); flap arrays clamped at 64 events (a fast-flapping flag can cycle in ~2.5 min ‚Äî the arithmetic bound is the clamp, not the grace math). Per-machine participant/progress tracking (`{ lastSeenVersion, lastAdvanceAtMs }` + the persisted per-key PARTICIPANT SET) is kept for every dimension ‚Äî the participant-aware clear and the flap-event definition read the participant set on all dimensions, not just version.
- **Every-machine computation:** all machines run reconcile today (speaks() gates only effects), so every machine accumulates its own durable copy of the anchors from the same shared adverts ‚Äî anchors survive raiser handoff (the new raiser has its own accumulated copy), restarts (durable file), and dips (the accumulator's bounded-gap rule). Residual honesty: a machine that never observed the skew's start (fresh join, corrupt-rebaseline) carries a LATER onset; if elected raiser, escalation stretches by that offset ‚Äî bounded drift, direction unchanged. **The correctness contract, named:** this is *at-least-once, bounded-duplicate narration* (per-machine reconstruction from shared adverts) ‚Äî never a single coherent global episode; every duplicate bound is disclosed (‚â§2√ó per handoff window per surface). Stronger semantics would require a shared event stream ‚Äî rejected above as a foundation rewrite for a narration change.

### M-P1 ‚Äî Progress-aware patch-skew confirmation

Confirmation for patch-only version skew reads the M-P0 anchors:

- `activeSkewMs` < `versionSkewGraceMs` (45 min): unconfirmed.
- Past grace: confirm only if NO gap-narrowing advance by the lagging machine within `versionSkewProgressWindowMs` (default 30 min); an advancing laggard extends.
- **Stall ceiling** (`versionSkewStallCeilingMs`, default 3 h of `activeSkewMs`): confirm regardless of progress; advances cannot reset it. Unit obligation: "N successive advances each inside the progress window still confirm at the ceiling"; "on a 3-machine pool, the laggard blipping offline for >resolveTicks does not clear the onset."
- `major-minor`: unchanged (2-tick confirm; escalates the same tick ‚Äî the critical class never enters the calm path).
- Unreadable/ambiguous adverts ‚áí no-advance ‚áí confirm at grace (louder).
- **Expectation honesty:** a single-hop laggard (old ‚Üí latest in one restart ‚Äî the auto-updater's normal shape) never "advances" mid-wave; its first advance IS the heal. M-P1 helps multi-version crawls and slow waves; the 2026-07-11 incident shape still confirms at 45 min ‚Äî its operator-visible improvement comes from M-P2's silent calm class. Live-pair verification is read against this calibration.
- **Config invariant:** `versionSkewProgressWindowMs` MUST exceed `updates.restartCascadeDampenerWindowMs` (default 15 min) with margin; boot-time config-sanity warning when violated.

### M-P2 ‚Äî Calm narration: silence for transients, honest escalation, bounded close-out

- **Priority + copy ride the effect:** the `raise` effect gains `priority` + copy class chosen by the episode manager; the server consumer becomes a pass-through (extracted to an injectable `machineCoherenceEffectsExecutor` module ‚Äî named build task). Mapping: patch-only version skew ‚Üí NORMAL, calm copy (no fix prompt; body = observed skew, self-heal in progress, will escalate if stalled); major-minor / flag / manifest / protocol ‚Üí HIGH with today's copy incl. the fix prompt; stall/flap escalations ‚Üí HIGH with the prompt + provenance line. Config priority values clamp to the known enum.
- **Calm raises are SILENT** (Near-Silent Notifications ‚Äî mandated default): the calm-phase raise posts to the hub with `disable_notification`, the item is created in the store (dashboard/pool visibility unchanged), and it is counted (`calmRaises` counter + jsonl row ‚Äî silent ‚âÝ dead). Rollback lever: `calmRaiseNotify: true` restores a buzzing calm raise. Reopen-appends on calm-class episodes are silent too; on loud-class episodes appends notify as today.
- **Cross-key wave backstop (zero-buzz can't hide a wave):** if the pool accumulates ‚â•`calmWaveThreshold` (default 6) calm-class episode onsets ‚Äî counting only NON-reopen onsets, so a single flapping key feeds the flap brake, not the wave count ‚Äî across ALL keys within 24 h, the raiser posts ONE aggregated, notifying NORMAL item ("N self-healing episodes today ‚Äî routine if updates are rolling; worth a look if not"), deduped per 24 h window via a durable `lastWaveNoticeAtMs`. The wave latch is per-machine: a raiser handoff inside the window can double the notice (‚â§1/machine/24 h ‚Äî the same accepted 2√ó-per-handoff residual class). A mass multi-key `:stalled` day (many keys genuinely stalled at once) is deliberately loud ‚Äî up to the ‚â§64 keyspace bound of individually notifying HIGHs ‚Äî accepted explicitly: simultaneous multi-key stalls are a real incident, not noise.
- **Escalation mechanics:** escalation of an open episode is a NEW raise under a derived id (`<episodeItemId>:stalled` / `<episodeItemId>:recurring`) ‚Äî HIGH, notifying, cap-EXEMPT (each ‚â§1 per key per 24 h via the durable `lastStalledFireAtMs`/`lastRecurringFireAtMs` latches ‚Äî keyed on the anchor, NOT the episode, so episode churn can't multiply them; restart-proof by persistence). One raise per class per episode ever (fixed ids; `createAttentionItem` id-dedupe is the intended idempotency); CONTINUED stalled-ness is narrated by the existing 24 h escalation append (`maybeEscalate`) on the base item ‚Äî never fresh daily items (no re-acting on a permanently-stuck condition at cadence).
- **Derived-item lifecycle (complete):** the episode state durably tracks `derivedItemIds`. `closeEpisode` resolves the base AND every derived item. An episode that escalated closes with a NOTIFYING resolve note whose copy adapts to the close reason ("healed ‚Äî the earlier stalled alert is withdrawn" / "that machine left the pool ‚Äî the earlier alert no longer applies"): the operator was loudly asked to consider a decision, so the stand-down notifies (decision-withdrawal is action-relevant). **Note authorship across raiser handoff (item-holder voice):** narration state is per-machine and the raiser role migrates, so the withdrawal note is authored by the machine CLOSING ITS OWN escalated item ‚Äî regardless of whether it currently speaks. That keeps one voice per ITEM (never zero notes when the old raiser holds the escalated item and the role moved), bounded by the already-accepted ‚â§2√ó-per-handoff residual when both machines escalated their own copies. Calm-only episodes close SILENT. Reopens append to the BASE item and never resurrect derived items. Unit obligation: "stall ‚Üí heal ‚Üí base AND derived items both DONE; exactly one notifying close note PER ITEM-HOLDING MACHINE (‚â§2 across a handoff window)."
- **Resolve-note bounding:** at most ONE resolve note per item per `reopenWindowMs`; a reopen-latched (flapping) episode closes to the jsonl audit only. (The wave backstop + flap brake carry the pattern-level signal.) **Precedence when latched ‚àß escalated:** latched wins toward silence ‚Äî the close note is suppressed (consistent with the ack-race rule "loses toward silent"); the derived items still resolve DONE via the derived lifecycle, and the flap/wave surfaces carry the pattern signal. This intersection joins the semantic-boundary test enumeration.
- **Flap brake** (Distrust Temporary Success): ‚â•`skewFlapThreshold` (3) flap events per key within 24 h ‚Äî counted from the durable M-P0 flap history (definition above) ‚Äî raises the `:recurring` derived HIGH. The existing reopen latch continues to govern per-item reopen NOISE; the brake governs pattern ESCALATION; thresholds independent.
- **Operator-interacted bit:** a durable per-episode `operatorInteracted` boolean set ONLY by authenticated operator actions ‚Äî fix approval and explicit ack ‚Äî with the setter carrying the same `verifiedOperator` evidence shape as `approveFix` (the ack path gains that parameter ‚Äî named build obligation; never derived from `pendingFix`, which is auto-created at raise and cleared on every fix path). Rides the `resolve` effect. Interacted episodes close notifying. Reopen does not carry it. An ack racing the close loses toward silent (documented; the raise/escalation already informed).
- **Legacy per-item routing:** `updateAttentionStatus` gains a silent option (named build task ‚Äî no such parameter exists today) so uninteracted-episode status lines don't buzz; the zero-buzz guarantee is scoped to the default hub mode, per-item mode gets "fewer buzzes," stated honestly.

### M-P3 ‚Äî Rope informational content: typed class at the source, delivery-true demotion

## Multi-machine posture

Default posture is **unified**; no new machine-local file or surface ‚Äî the one durable change is new FIELDS in the episode manager's existing per-machine durable file:

- M-P0 anchors are computed by EVERY machine from the SHARED adverts (reconcile runs on all machines; speaks() gates effects) and persisted per-machine ‚Äî surviving restarts (durable), raiser handoff (the new raiser has its own accumulated copy), dips and warmup (bounded-gap accumulator; reconcile's <2-online early-return and the ~4-tick warmup contribute at most K ticks each). Cross-machine anchor drift from advert timing shifts thresholds by seconds-to-minutes, never direction; a fresh-joined/rebaselined machine carries a later onset (bounded stretch, documented).
