# Test-Runner Concurrency Bound — a host-wide vitest semaphore (Bounded Blast Radius)

## Constitutional fit

This spec implements the **Bounded Blast Radius** constitutional standard (the same standard that mandates the fork-bomb LLM-spawn cap). An unbounded per-actor resource-spawn pattern whose host-wide aggregate is catastrophic is precisely what that standard exists to prevent; the test-runner storm is the second instance of that class (the first, LLM subprocesses, is already bounded). It also serves **Responsible Resource Usage** (OS resource hygiene) and directly closes the load-stall kill cascade documented in the 2026-07-02 respawn root-cause analysis.

**But it is NOT a verbatim copy of the spawn cap.** Round-1 convergence review established that the spawn cap's core reclaim policy and fail-direction were tuned for a *fundamentally different consumer* (one long-lived server process, second-scale holds, an OOM-catastrophe fail-direction). Reusing them unmodified would re-create the exact meltdown this feature prevents and inherit the 2026-07-01 stale-holder wedge. This spec therefore **extracts a parameterized core** and derives a *consumer-specific* policy for the test lane. §1.1 is the load-bearing section: the fail-direction inverts.

## 1. Problem

A vitest ROOT process is not one process: it forks a worker pool sized to the machine (~cores threads/forks). N concurrent roots ≈ N × cores workers competing for CPU, memory, and the filesystem. On an Instar dev host, many actors start suites independently and simultaneously:

- several agent build sessions, each running `pnpm test` / `pnpm test:integration` in its own worktree;
- every `git push` (husky pre-push runs a suite);
- `/build` phase gates and scratch verification runs;
- occasional operator-driven runs.

Nothing bounds the SUM. The existing fork-bomb spawn cap (`hostSpawnSemaphore`) bounds LLM subprocesses only — test runners ride completely outside it. 29 concurrent roots on a 16-core host ≈ 300+ runner threads: the event loops of the co-resident agent servers starve, their supervisors interpret the stall as ill-health, and healthy servers get killed. The safety mechanism becomes the outage.

This is the same *class* as the 2026-06-20 LLM fork-bomb, one layer over: an unbounded per-actor spawn pattern whose aggregate is catastrophic, needing a HOST-WIDE counting bound at a single structural chokepoint.

### 1.1 The fail-direction INVERTS from the spawn cap (the load-bearing design decision)

The spawn cap fails **closed** on uncertainty (can't confirm host-local disk → refuse to reclaim → keep counting; can't acquire → don't spawn). That is correct *for an OOM safety floor*: a delayed LLM spawn is cheap, an OOM is catastrophic, and its background consumers have a heuristic-degradation fallback when a call is shed. **None of that holds for a test cap.** The harms invert:

- A false **BLOCK** (over-counting, a stale/phantom holder, a wedged lock) has NO degradation path — it wedges *every* `git push` (husky pre-push), *every* `/build` phase gate, and *every* scratch run host-wide behind a loud timeout wall. On a host that can't push, the agent can't ship the fix for the very reachability/load incidents this family exists to prevent — the bound blocks the release valve.
- A false **PASS** (bounded over-admission) is one extra concurrent suite — transient, self-limiting CPU pressure that resolves in minutes.

Therefore the least-harmful direction on *uncertainty* for THIS detector is **fail-OPEN — admit the run**. Every uncertainty branch in this spec (df misclassification, pid-reuse, corrupt holders file, provably-wedged lock, unresolvable ancestry) biases toward *admitting* a run over *wedging* the dev loop. This is the single most important divergence from the spawn cap and it is applied consistently below. (State-Not-Symbol: "unknown fails toward the least-harmful action *for that detector*" — and this detector's least-harmful is the opposite of the spawn floor's.)

**Fail-open granularity matters (round-2 finding):** fail-open is a verdict about *persistent, provable* uncertainty — never a per-poll shortcut. A single missed lock attempt is "keep polling", not "admit"; the mass-admit failure mode (20 queued roots each treating one contended lock attempt as fail-open → the exact fork-storm this spec prevents) is designed out in §2.4.

**Honest naming (codex external finding):** because it fails OPEN, this is a **best-effort host-wide bound with fail-open recovery**, not a hard mutual-exclusion guarantee. Under a persistent lock wedge the bound is temporarily lost in the safe direction until the 10s age-reclaim fires (§2.4); the witness records make those windows auditable. Read every "bound"/"cap"/"host-wide" in this spec with that qualifier — the design deliberately prefers a brief over-admission to ever wedging the dev loop.

### 1.2 Prior incident this design must not repeat (2026-07-01) — root-cause is IN SCOPE

On 2026-07-01 a crashed agent server left **unreaped holders** in `host-spawn-holders.json` → **false saturation** → the tone gate (a fail-closed consumer) held → **the agent went unreachable**. The inherited "dead-holder auto-heal" did NOT fire; recovery was a manual dead-PID prune of the holders file. That incident is *this exact holder-set reclaim logic failing to self-heal in production.* Building a SECOND semaphore on the same core doubles that surface, and the test lane makes each stale holder worse (cap 1–2 vs 8 → one phantom is 50–100% of capacity; minute-scale suites vs second-scale spawns → wider stale window + more pid-reuse exposure).

**Mandatory work item of the extraction PR (Distrust Temporary Success):** root-cause WHY the 2026-07-01 auto-heal did not fire, against the real holders file and host state (the two live suspects in the foundation: `pruneDead`'s fail-closed df gate silently disabling all reclaim when `df` can't confirm host-local, and the `pidDead && heartbeatStale` conjunction making a reused pid's holder immortal). The finding is recorded in the extraction PR's description and this spec's §6 table is re-checked against it. The spawn-lane FIX (back-porting the prune lever and evaluating a spawn-lane max-hold TTL) is **out of scope for this PR** to keep the extraction reviewable — but it is NOT a loose "considered": a tracked commitment for the back-port decision is opened at build start (Close the Loop; see §4).

This design answers the incident directly with: fail-OPEN on *provable* uncertainty (§1.1), a max-hold TTL that reclaims even a pid-alive holder (§2.4), immediate reclaim of a provably-dead pid (§2.4), a first-class manual + passive prune lever (§2.6), and a durable event ledger so none of it is invisible (§2.8). §5 requires tests that specifically exercise crashed / SIGKILL / pid-reuse / hung holders on the *test* file.

## 2. Design

### 2.1 Primitive — extract a PARAMETERIZED core; do NOT reuse spawn semantics verbatim

Extract the holder-set mechanics of `src/core/hostSpawnSemaphore.ts` into a shared internal core `src/core/hostSemaphoreCore.ts`. The extraction seam is explicit (integration review M2):

- **Moves into the core:** the lock primitive (`O_CREAT|O_EXCL` + stale-lock reclaim), holders read/parse/write (atomic temp+rename), the host-local `df -P` determination, and the prune-dead pass — but the prune-dead pass takes a **ReclaimPolicy** parameter (see §2.4), not the spawn cap's hardcoded `pidDead && heartbeatStale && dfLocal`.
- **The core's inherited stale-LOCK reclaim is UNSAFE, and the test lane does NOT ride it (round-8 lessons-aware MEDIUM — foundation audit).** The foundation's `reclaimStaleLock` carries three defects the test lane's own reasoning already condemns: a NON-ATOMIC unlink-then-recreate reclaim (two contenders can both observe a dead/unparseable lock and both enter the critical section, silently losing a holders row to last-write-wins — the exact race §2.4's test-lane AGE-reclaim closes with atomic-rename); a torn-read hazard (the lock is written as `openSync('wx')` then a SEPARATE `writeSync`, so a contender can read an empty just-created lock, hit the JSON.parse catch, and unlink a LIVE lock); and a lock record of `{pid, at}` with NO hostname and reclaim NOT df-gated (on a synced `~/.instar` a peer machine pid-probes a foreign, locally-meaningless pid and unlinks a live lock — contradicting this design's own host-local-lock contract). In the SPAWN lane a lost holders row = UNDER-count = over-admission against its OOM floor (the dangerous direction there). **Resolution (quarantine, not inherit):** the test lane's ONLY lock reclaim is the §2.4 race-safe AGE-reclaim (atomic rename of the specific observed lock, one winner; an empty/torn lock is age-gated, never unlink-on-unparseable while fresh) — it does NOT use the core's pid-death lock reclaim at all. The `hostSemaphoreCore` extraction therefore parameterizes lock-reclaim policy the same way it parameterizes prune-dead: the test lane passes age-reclaim-only. Hardening the SPAWN lane's live lock reclaim (atomic rename + one-step atomic lock write + hostname stamp + df-gate) is added to the §4 tracked back-port commitment — it is a real foundation defect, surfaced and tracked, never carried silently into the new shared core. The golden test (below) additionally pins that the extraction does not CHANGE the spawn lane's lock behavior (the back-port is a separate, later change).
- **`acquire()` admission is parameterized** via an optional `admit(live, cap) => boolean` predicate. The spawn cap passes its F5 interactive/background reservation predicate; the test lane passes a plain `live.length < cap`. The F5 lane logic is NOT hoisted into the core — it stays in `hostSpawnSemaphore`'s predicate.
- **The singleton + config layer stays per-module** (`configureHostSpawnSemaphore` / `getHostSpawnSemaphore` / `_resetForTest` are duplicated thin per module — two holders files, two caps, two independently-tunable policies). It is NOT extracted (two caps can't share one singleton).
- **`hostSpawnSemaphore.ts`'s public export list AND its on-disk holders-file byte format are unchanged.** A golden/characterization test (§5) pins the spawn holders format (including the optional `lane` field's presence/absence) byte-identical pre/post extraction, and the 10 importers (4 src, 6 test) are verified against the unchanged export surface. The extraction is treated as its OWN reviewed change with its own tests — not an assumed no-op.
- **Doc-contract fidelity:** the foundation carries a doc-code contradiction (`HOLDER_STALE_MS`'s comment says pid-dead OR heartbeat-stale; `pruneDead` implements AND). The extraction fixes the comment to the AND semantics and states the pid-primary/heartbeat-secondary contract once, authoritatively, in `hostSemaphoreCore`'s ReclaimPolicy docs — the golden test pins the behavior either way.

New `src/core/hostTestRunnerSemaphore.ts` — holders file `~/.instar/host-test-runner-holders.json`, cap from `INSTAR_HOST_TEST_MAX`, the plain `admit` predicate, and the test **ReclaimPolicy** (§2.4). The test lane's holder record adds an **identity fingerprint** at acquire time — `{ pid, hostname, acquiredAt, startedAt, cmd }` where `startedAt` is the process start time (`ps -p <pid> -o lstart=`) and `cmd` a clamped copy of the command line — consumed by the §2.4 signal-corroboration gate. Two resource pools stay strictly separate (separate holders + lock files, no cross-talk); the two caps are **independent and additive**, not jointly bounded (§2.7).

*Acknowledged trade-off — alternatives considered (round-4, both externals):* the file-lock holder-set is not an OS-native semaphore and has known subtle failure modes on exotic filesystems. The alternatives and why each loses here: **`flock`/fcntl advisory locks** — released when ANY fd on the file closes in the same process (hazardous around vitest's fork/worker tree) and lost on NFS/synced homes in ways that still require the df-style detection this design already carries, while providing no holder METADATA (who, since when, which lane) that the TTL/corroboration/attribution machinery needs; **SQLite with busy_timeout** — buys a real mutex but adds a native dependency to the zero-dependency test path, and its busy-wait is the same polling loop in different clothes. This repo DOES ship SQLite server-side (PendingRelayStore et al.), so the objection is not "SQLite is foreign" — it is that the CHOKEPOINT must run in every bare worktree checkout, and coupling test-runnability to a native module's build health creates a brand-new wedge class (one broken `better-sqlite3` rebuild after a Node upgrade would block ALL testing host-wide — a §1.1 false-BLOCK with no degradation path), while the JSON path has zero build dependency; **POSIX named semaphores** — leak permanently on crash (the 2026-07-01 failure class with LESS visibility), carry no holder metadata, and have low macOS limits; **third-party lockfile libraries** (`proper-lockfile` etc.) — a new dependency for the same primitive `hostSpawnSemaphore` already ships with a production soak history and a byte-format the golden test pins; **a local broker daemon** — requires a running server on exactly the serverless builder/worktree population §2.6 identifies as the incident hosts. JSON+`O_EXCL`+atomic-rename is chosen for zero dependencies, holder metadata as first-class data, and continuity with the proven spawn-cap core. Migrating `hostSemaphoreCore` to an OS-native primitive is a possible future hardening, out of scope here.

*Build guidance (L12 — SafeFsExecutor funnel):* all new destructive-fs callsites this design adds (quarantine keep-newest-5 deletion, witness-record sweep, ~5MB ledger rotation, atomic lock age-reclaim) live in the `src/core` semaphore modules where `scripts/lint-no-direct-destructive.js` reaches them, routed through `SafeFsExecutor` — NEVER relocated into `tests/setup/…globalSetup.ts` (the lint may not reach it). The globalSetup stays a thin caller.

### 2.2 Chokepoint — vitest globalSetup, with hard performance constraints

A `tests/setup/test-runner-semaphore.globalSetup.ts` is added to the `globalSetup` of all five vitest configs (`vitest.config.ts`, `vitest.integration.config.ts`, `vitest.e2e.config.ts`, `vitest.contract.config.ts`, `vitest.push.config.ts`). Only the integration config already has a `globalSetup` array (`build-dist.globalSetup.ts`) — there the semaphore is **prepended** so `setup()` acquires before the dist build and `teardown()` releases after it (globalSetup teardown runs in reverse); the slot is deliberately held through the build (acceptable at this cap). The other four are a plain add. Integration review confirmed these 5 are exhaustive (no `vitest.workspace.ts`, no programmatic `startVitest`; every `package.json` path + husky pre-push routes through one of the 5). A §5 guard-test asserts every `vitest.*.config.ts` wires BOTH per-config seams — the globalSetup entry AND the shared config-eval helper (classification/clamp/reservation, §2.3/§2.5) — so a future 6th config can't silently escape the bound at either seam (round-5 integration: a config with the globalSetup but without the helper re-opens unclamped nested shelter for exactly that config), and FAILS on the introduction of any `vitest.workspace.*` file until the workspace path is reviewed.

**Why globalSetup and not a wrapper script as the primary chokepoint (cross-model finding):** a `guarded-vitest.mjs` wrapper only guards the `package.json` script paths that call it — a direct `npx vitest` / editor-integration / ad-hoc invocation bypasses it silently. The globalSetup rides INSIDE the config, so every invocation path that loads the config is bound, with zero script rewiring. The wrapper remains the named FALLBACK if the acquire-before-fanout invariant fails validation (item 5).

**Hard implementation constraints (each closes a round-1 finding — a naive reuse re-creates the meltdown):**

1. **Async yielding wait, never a busy-spin.** The between-poll wait MUST be `await new Promise(r => setTimeout(r, interval))`. The spawn cap's in-module `busyWaitTiny()` synchronous spin, multiplied across 20+ queued cold-starting roots, pegs every core for the whole wait — re-introducing the exact starvation this spec prevents. Only the sub-millisecond lock critical section may spin, and its deadline MUST exceed the critical-section duration. **Pinned defaults (cheap-to-change-after behind the ratified dry-run phase):** poll interval 5s ± up to 1s uniform jitter; lock-acquire deadline 250ms (≫ the sub-ms critical section).
2. **`df -P` OUTSIDE the lock.** The host-local determination is computed BEFORE taking the holders lock and passed in, so the critical section is file-read + count + write (sub-ms). (In the spawn cap a single long-lived server memoizes `df` once; here every root is fresh, so `df`-under-lock would hold the exclusive lock for up to 3s while all others spin-and-timeout.) The determination is cached to an on-disk marker at the frozen path `~/.instar/host-test-runner-dflocal.json` (`{ device, local, checkedAt }`) so cold-start roots skip the probe entirely. **The marker is revalidated, not trusted forever (round-2 security finding):** on each cold start the recorded `device` is compared against a fresh `fs.statSync(dir).dev` AND `checkedAt` is honored against a 24h TTL — on mismatch or expiry the `df` probe re-runs and the marker is rewritten. This closes the "home later moves onto iCloud/Syncthing/NFS → stale `local:true` permanently arms reclaim against a cross-machine holders file" hazard the whole host-local contract depends on. (An alternative — hostname-namespacing every rendezvous path, `~/.instar/<hostname>/…` — was considered and rejected: it would break the single-bound-per-host-user contract for multi-checkout hosts and change every frozen path; the cheap `statSync().dev` revalidation + the foreign-hostname drop (§2.4) are the chosen mitigation.)
3. **Jitter, not backoff, on the poll; write only on change.** The poll is cheap (0.2/s) so backoff isn't needed and would only delay noticing a freed slot; instead add ±jitter to desynchronize the thundering herd, and skip the holders rewrite on a failed acquire when prune removed nothing (the spawn cap rewrites on every failed poll → O(waiters) churn). This is the correct reading of the conformance gate's "no backoff" flag: **jitter-not-backoff**. P19 mapping, explicitly: *bounded ingress* = the hard wait ceiling caps total attempts; *breaker* = the typed-error exit at the ceiling; *cost cap* = constant-cost attempts + write-skip + bounded per-minute logging. All three brakes are present; backoff is the one substituted element and the substitution is reasoned (the poll queries capacity, not a failing target — growing the interval only delays noticing a freed slot).
4. **Throw a typed error on timeout, never `process.exit()`.** `process.exit` bypasses vitest cleanup and any prepended-after teardowns and can leak the slot; a thrown typed error is reported cleanly by vitest and leaves no holder. Confirm no slot is held on the timeout path.
5. **Acquire BEFORE worker-pool fanout — a load-bearing invariant that MUST be verified, not assumed.** The whole design rests on `globalSetup` running before vitest materializes the worker pool (and its transform/project-graph/reporter state). If a vitest version initializes any of that before `globalSetup`, the semaphore would bound only part of the blast radius. §5 carries an explicit validation item that instruments each pinned config/version to prove the slot is acquired before worker fanout; if a version cannot guarantee it, the chokepoint moves to the `guarded-vitest.mjs` wrapper (which then must ALSO be wired into husky pre-push and documented as not covering ad-hoc invocations). **Ship gate, stated plainly (round-4 codex):** failing this validation BLOCKS shipment until the wrapper fallback is wired and passing — there is no third path. The validation gate covers the CONFIG-EVAL seam identically (round-5 codex — config evaluation is not a stable concurrency-control boundary across editor integrations/config caching/future vitest versions, so the classification/clamp/reservation behaviors §2.3/§2.5 place there are proven per pinned version by the same §5 instrumentation, with the same wrapper fallback). Wrapper-as-PRIMARY was considered and rejected: it covers only the script paths that call it, silently un-bounding editor/ad-hoc invocations that the config-riding chokepoint bounds for free — a coverage regression traded for boundary stability the validation gate already buys. The wrapper's reduced coverage (ad-hoc shell `npx vitest` bypasses it) is explicitly ACCEPTED as the fallback posture because the constitutional target is the multi-actor storm, which flows through `package.json` scripts and the pre-push hook — both wrapper-covered and repo-lint-guarded (§5); ad-hoc single invocations are the documented residual either way. The spec does not ship on the *assumption* — it ships on the verified invariant or the wrapper fallback.

### 2.3 Run-class differentiation — TWO SYMMETRIC LANES, not a lane + a bespoke exemption

A single coarse root-count cap is caught between two real needs: the operator directive "NEVER run builders' full **suites** concurrently" argues for a very low cap on suites, while an inner-loop `npx vitest run one.test.ts` (1–2 workers, trivial load, nowhere near a fork-bomb) should not be blocked behind a full suite for up to the wait ceiling.

**Round-3 review (adversarial + scalability + integration + the codex external, converging) established that the earlier "exempt-and-post-count" design was the wrong shape** — its bespoke machinery (skip-then-count, a clamp at one evaluation point but lane-routing at another, an aggregate cap with no atomic reservation) generated a bug per round because it was NOT the proven holder-set primitive. The convergent fix is to stop treating targeted runs as an *exemption* and make them a **second SYMMETRIC semaphore lane** that reuses the identical, already-hardened acquire path — same lock, same atomic holder record, same throw-on-budget, same tuning-file cap authority, same reclaim policy. There is no "skip free" and no post-hoc counting; a targeted run ACQUIRES a targeted-lane slot exactly as a suite run acquires a suite-lane slot.

- **suite-class** — the integration / e2e / contract / push configs (always full suites), AND any `vitest.config.ts` run that does not meet the strict targeted test below. Acquires a **suite-lane** slot (cap `INSTAR_HOST_TEST_MAX`, default 1).
- **targeted-run** — a `vitest.config.ts` invocation whose argv filters are **exclusively positional file paths** that, under vitest's ACTUAL filter semantics, select at most K test files (default 5). **Verify the SELECTED set, not argv existence (round-8 adversarial MEDIUM — vitest positionals are substring FILTERS, not exact selectors).** An earlier draft required each positional to "resolve to a real file on disk" and treated that as "runs ≤K files" — but `vitest run e.test.ts` (where a real file `e.test.ts` trivially exists) classifies as 1 file yet matches EVERY test path containing that substring (`core.test.ts`, `route.test.ts`, …), i.e. a large fraction of the suite, silently sheltering suite-scale work in the roomier lane AND falsifying the ledgered "resolved file count" the soak's dodge-detection reads. The round-5 "verify the state" fix was applied to the POOL dimension but the FILE dimension still trusted an argv label. So classification happens at **config-eval (which sees the resolved include/exclude set)**: each positional is expanded under vitest's filter semantics and the run is targeted ONLY if the MATCHED test-file count ≤ K — and the ledger records the MATCHED count, not the argv count. If matching vitest's semantics exactly is judged too version-fragile, the pinned conservative fallback (same safe-superset direction as every other classifier uncertainty): a run is targeted only if each positional is a path within the resolved include set that matches EXACTLY ONE file under filter expansion; anything ambiguous (multi-match, directory, glob, `-t`/`--project`/pattern flags, or >K matched) routes **suite-class**. Acquires a **targeted-lane** slot (cap `INSTAR_HOST_TEST_TARGETED_MAX`, default 6) — a roomier lane so the inner loop isn't stuck behind the single suite slot, but STILL a bounded, throwing acquire: when the targeted lane is full for its budget the run **throws the same typed capacity-timeout as the suite lane — it NEVER fail-open-admits** (fail-open-admit is lock-wedge-ONLY per §1.1). This closes the shard-a-suite bypass (80 concurrent ≤5-file runs can't all admit — at most `TARGETED_MAX` do; the rest throw) by construction, not by a post-hoc cap. Worst-case targeted-LANE load ≈ `TARGETED_MAX × maxWorkers-clamp` ≈ 24 workers (the lane-slot term only — nested shelter under targeted slots is an additional additive term, counted in §3's budget table).
- **Single-source classification — VERIFY THE STATE, never a symbol (round-4: adversarial + security + decision-completeness + lessons-aware, converging).** Two evaluation points exist (config-eval sets the ≤4 maxWorkers clamp because only a config can reshape the pool before fanout; globalSetup does the lane acquire). They MUST NOT disagree in the dangerous direction (unclamped AND targeted-lane-routed). An earlier draft bound the two points with a per-run "clamp stamp"; round-4 review killed the stamp from four independent angles — a presence-based symbol leaks (env stamps inherit into descendant processes and persistent shells — the exact hazard §2.5 cross-checks for `INSTAR_TEST_SEMAPHORE_HELD`), strands (a marker file orphaned between config-eval and globalSetup satisfies a LATER run's check), forges, and forks an unresolved either/or for the builder. The binding is instead **direct observation of the live state**: globalSetup receives the RESOLVED config (the same context §2.6 reads `config.watch` from), so it routes to the targeted lane iff (a) its OWN strict argv classification says targeted AND (b) the resolved config's ACTUAL worker-pool bound (`maxWorkers`/pool equivalents) is ≤ 4. There is no symbol to leak or forge: a run whose final resolved pool exceeds 4 workers — because CLI pool flags outrank config values, because config-eval never clamped, for ANY reason — lands in the cap-1 suite lane BY CONSTRUCTION. The dangerous combination is impossible rather than merely tested-against. (State-Not-Symbol, applied literally.)
- **Pool-shaping argv disqualifies — the CLI-precedence bypass is closed at BOTH points (round-4 adversarial HIGH).** vitest resolves CLI options with HIGHER precedence than config values, so `npx vitest run one.test.ts --maxWorkers=32` would defeat a config-eval clamp. Two independent closures: (1) ANY pool-shaping argv flag (`--maxWorkers`, `--minWorkers`, `--pool`, `--poolOptions.*`, `--fileParallelism`, `--no-isolate`) disqualifies the run from the targeted class at BOTH evaluation points (configs are code and read `process.argv`); (2) even if a novel pool flag escapes the list, the resolved-state check above sees the >4 pool and routes suite-class anyway — the flag list is defense-in-depth, the state check is the guarantee. `VITEST_MAX_THREADS`/`VITEST_MAX_FORKS` env: config-SET values take precedence over those env vars, so the config clamp holds — §5 VERIFIES this on the pinned vitest version rather than assuming it.
- The maxWorkers ≤ 4 clamp is applied at CONFIG-EVAL time (`vitest.*.config.ts` are code; globalSetup receives an already-resolved config and per §2.2 item 5 cannot reliably re-shape a materialized pool). §5 asserts a targeted run's ACTUAL concurrent worker count ≤ 4 on the pinned vitest version; if the config seam can't guarantee it, the targeted class tightens (lower K, or targeted runs share the suite lane) rather than ship unclamped. **The matched-file-set expansion is likewise proven per pinned version, not assumed (round-9 codex — vitest filter semantics are not a stable protocol):** where the config-eval exposes vitest's OWN resolved include/exclude set the classifier reuses it directly (not a re-implemented globber); where a version does not, the conservative EXACTLY-ONE-MATCH fallback applies and every ambiguity routes suite-class — so version drift can only ever mis-route toward the STRICTER lane (safe-superset), never fail open to the roomy lane. §5's argv matrix runs on the pinned version so a semantics change is caught by a test, not in production. §5's agreement matrix covers the full adversarial argv surface (positional file, directory, glob, `-t`, `--project`, mixed, AND each pool-shaping flag), plus the state check: a targeted-argv run carrying `--maxWorkers=16` must route suite-class, asserted at the routing decision AND by the meta-test's actual-worker-count measurement.
- Peak concurrent targeted-lane occupancy is a named soak-review metric (§4).
- Every targeted-lane acquire is ledgered with the resolved file count (§2.8), so the dry-run soak review can flag any classification pattern that looks like cap-dodging.

This lets the suite-class cap sit at the low value the directive wants without punishing the inner loop, and it makes "bound every root" no longer the design (closing the scope decision by construction — see Frontloaded Decisions).

### 2.4 ReclaimPolicy for the test lane (parameterized, NOT the spawn defaults)

The test lane's ReclaimPolicy diverges from the spawn cap's on every axis where the workload differs:

- **Immediate reclaim of a provably-dead OR provably-not-the-holder pid (round-8 adversarial MEDIUM — pid-reuse must not wedge the default path).** On a df-confirmed-local disk, a holder whose pid is dead is reclaimed IMMEDIATELY — no 5-minute `heartbeatStale` gate. That gate exists to protect a *live-but-heartbeat-quiet* spawn from reclaim; the test lane has no heartbeat-refresh thread, so a live suite is already fully protected by pid-liveness alone, and the gate would only add up-to-5-min reclaim latency after every test crash/Ctrl-C (a frequent dev event that would otherwise stall the next push). **But pid-liveness ALONE is not enough at cap=1:** a crashed suite's pid (Ctrl-C/OOM/SIGKILL are frequent) can be REUSED by any longer-lived process within the hour (macOS pid churn on a heavy test host makes this plausible), and a pid-liveness-only check reads that reused pid as "holder alive" → NOT reclaimed until the 60-min TTL → a false-BLOCK of every push and build gate for up to an hour (the §1.1 catastrophic direction), which `POST /prune` (running the SAME policy) could not clear either. The identity fingerprint (`startedAt`) is already recorded on every row — so start-time corroboration runs on EVERY reclaim pass (acquire-path prune AND `POST /prune`), not only the signal gate: a holder whose LIVE pid's start time postdates the row's `acquiredAt` (+skew) is provably NOT the recorded holder → treated as dead → immediate CAPACITY reclaim, `reclaim-mismatch` ledgered, NEVER signaled (capacity direction only — this is the default, signal-arm-independent path). The same liveness+identity gate applies to witness and storm-slot occupancy. **The `ps -p <pid> -o lstart=` evidence is gathered OUTSIDE the holders lock (round-9 scalability), mirroring the df-outside-lock discipline (§2.2 item 2):** the cheap `kill(pid,0)` liveness syscall gates it (so dead rows reclaim with no `ps` at all), the pid set is snapshotted, `ps` runs on the live pids, and only the resulting reclaim decision is applied under the lock (re-validating each row) — so the sub-ms critical section stays sub-ms even though live-holder count is single-digit at cap 1/6. §5: a reused-pid holder is reclaimed immediately PRE-TTL by both the acquire path and `POST /prune`.
- **Max-hold TTL (the single most important test-lane addition) — DEFAULT is capacity-reclaim-ONLY; process-signaling is a separate, opt-in, post-soak arm.** A holder older than its ceiling is reclaimable EVEN IF its pid is alive — this closes the "hung-but-alive vitest root never reclaimed → two hangs wedge all host testing indefinitely" failure that pid-liveness alone cannot. **Round-2 review (three internal reviewers + the codex external pass, converging) established that KILLING the holder is a far riskier action than the design needs by default:** a semaphore's job is to restore *capacity*, not to execute processes, and every kill path (process-group blast radius, non-durable escalation, cross-process TTL divergence) adds a surface that can harm an innocent process. The reclaim therefore splits:
  - **Default — capacity-reclaim-only (no signal ever).** A holder past its TTL has its SLOT freed (capacity restored — the actual goal) and a loud `stale-holder-reclaimed` event is written to the ledger AND surfaced on `/guards`/the route. NO process is signaled. The over-admission while a genuine hung root's worker pool keeps draining is accepted and stated: bounded in practice by the host actor count, visible in the ledger, and the operator's lever (see below) exists if it bites. This is the shipped behavior through the entire soak and by default afterward.
  - **Opt-in — TTL process-signaling arm (`INSTAR_HOST_TEST_TTL_SIGNAL=1`, default OFF, never armed during the dry-run soak).** Only when an operator has explicitly turned this on (after the soak proved TTL classification is accurate) does reclaim escalate to killing a corroborated hung root. Even then it is gated by FOUR mandatory checks — a holder row is data, not a warrant:
    1. **Pid sanity clamp** — target must be an integer ≥ 2 and not the reclaimer's own pid/pgid. A row carrying 0, 1, negative, or non-integer is corrupt ⇒ quarantine path, slot freed, NO signal. (`process.kill(-1, …)` would signal every process the user can reach.)
    2. **Identity corroboration** — the live process at that pid must match the holder's recorded fingerprint: start time ≤ `acquiredAt` (+skew) AND a vitest/node test-runner command line. A reused pid fails the start-time check deterministically ⇒ slot freed, NO signal, `reclaim-mismatch` ledgered.
    3. **Group-leadership check (round-2 security/adversarial finding).** A group-targeted signal (`kill(-pid)`) is sent ONLY when `getpgid(pid) === pid` — the root is its own group leader, so the group is exactly its own fork tree. If it is NOT a leader (launched from a husky hook sharing `git push`'s group, a tmux pane job, an agent exec layer), signal the corroborated single PID only and ledger the downgrade — the group membership was never corroborated, so it is never the blast target. (To make group-signaling reliably available, the holding root's globalSetup best-effort `setsid`/`setpgid`s itself at acquire and records leadership in the fingerprint.)
    4. **Durable escalation (round-2 security/scalability/adversarial finding), with an independent timer for the quiesced host (round-3 adversarial finding).** SIGTERM does NOT delete the holder row. Instead the row transitions to a non-cap-counting `terminating{ signaledAt, fingerprint }` tombstone (capacity is freed for admission, but the kill obligation persists). The SIGKILL is driven by THREE completers, so it does not depend on one ephemeral process: (a) an **unref'd-but-loop-kept timer armed in the reclaiming process** for the grace window, which attempts the SIGKILL directly if that process is still alive; (b) ANY later acquire pass or `POST /prune`; (c) a lightweight **server-tick tombstone sweep** on a cadence WHERE a server exists. Each completer re-checks: grace elapsed AND fingerprint STILL corroborates AND leadership → SIGKILL the target, drop the tombstone; pid dead or fingerprint mismatch → drop the tombstone, no signal. This closes the round-2 "reclaimer exits mid-grace → SIGKILL never fires" hole. Honest residual: on a fully quiesced serverless host where the reclaiming process ALSO dies within the grace and no further test activity ever occurs, completion waits until activity resumes — bounded by activity, not unbounded time. This residual only bites an operator who armed `TTL_SIGNAL` on a quiet host (the DEFAULT reclaim never signals at all), and the CPU-draining root remains attributable in the ledger meanwhile.
    5. **Wall-clock / sleep-wake honesty (round-8 adversarial + gemini clock-skew).** TTL age is pure wall-clock from `acquiredAt`; dev hosts are laptops (this codebase ships a `SleepWakeDetector` for exactly this reason). A machine that sleeps ≥ the remaining TTL mid-suite wakes with the holder past TTL — and under the arm all four gates PASS against the genuine, healthy, resumed holder (identity corroborates, it is its own group leader), so the machinery would group-kill a healthy progressing suite — a wrong-kill the soak cannot observe (the arm is off throughout the soak). Before any signal, the reclaimer therefore best-effort checks for a sleep/clock-jump gap overlapping the hold where obtainable (`sysctl kern.boottime` + a wake heuristic on macOS; the server's `SleepWakeDetector` where a server is present) and, on evidence of a gap, RE-ARMS the TTL window once rather than signaling. Where a boot/wake signal is not obtainable, the sleep-wake wrong-kill is a NAMED residual of arming `TTL_SIGNAL` on a mobile host — stated here and in the arm's operator announcement (§2.9), never implied away by the four-gate story. The DEFAULT capacity-reclaim path is unaffected by clock jumps beyond a bounded early/late slot-free (safe, no signal).
  - **Per-row TTL, RE-CLAMPED on read (round-2 + round-3 adversarial findings).** The effective ceiling is stamped into the holder ROW at acquire time (`ttlMs`), read from `INSTAR_HOST_TEST_TTL_MS` (default 60 min — the flip requires it to exceed the soak's observed p99 suite+dist-build HOLD by ≥2× (round-6 codex: a legit long suite past TTL keeps its workers while its slot frees, so repeated TTL-crossing legit runs would stack sustained over-admission; the 2× margin plus the fact that a LIVE-pid `stale-holder-reclaimed` is a high-visibility route/guards surface — not just a ledger line — keeps that drift loud and correctable, §4). Every reclaimer honors the ROW's `ttlMs` (not its own env) so a peer running the 1h default can't prematurely reclaim a run that legitimately raised its own TTL to 2h — BUT the reclaimer **re-clamps the row value to the sane host max on READ, not only at acquire**. A same-user poisoned/buggy/hand-edited row carrying `ttlMs=999999999999` parses fine and is NOT caught by the poison-sanity COUNT ceiling, so trusting it verbatim would make that slot's hung root immortal (the max-hold TTL silently disabled for that slot — the exact failure §1.2 calls the most important addition). **`ttlMs` is sanity-RANGED on read, not merely max-clamped (round-8 adversarial HIGH — the last un-closed quiet twin).** Rounds 4–6 gave every quiet-twin lever a pinned RANGE (caps `[1, ceiling]`; the signal arm env-disarm-only) but `ttlMs` got only the UPPER clamp — and it is a per-process env that STAMPS rows, so a leaked/inherited `INSTAR_HOST_TEST_TTL_MS=1` (the exact tmux/husky/launchd env-inheritance hazard this design repeatedly treats as real) makes every row from that context instantly-expired: each acquirer's prune then stale-reclaims all prior LIVE holders, silently un-capping the host — and, once an operator legitimately arms `TTL_SIGNAL`, a tiny-ttlMs row passes ALL FOUR signal gates against a HEALTHY run (the target IS the genuine holder), converting the kill machinery into killing every healthy suite shortly after acquire. Worse, a max-ONLY clamp is dodged by a NaN/non-numeric `ttlMs` (NaN comparisons are false → the immortal-slot outcome the clamp exists to prevent). Therefore `ttlMs` is coerced to a **finite integer in `[HOST_TEST_TTL_MIN_MS, HOST_TEST_TTL_MAX_MS]`** on read: any non-numeric / NaN / non-integer / out-of-range value resolves to the **code-default TTL + a WARN ledger event**, applied at BOTH acquire (env read) AND every reclaim read. **The coercion uses `Number()` + `Number.isInteger` + finite range check (round-9 adversarial cosmetic), NEVER `parseInt`** — so `"300000abc"` and `"300000.9"` are definitively REJECTED to the default, not silently truncated into range, and `Infinity` / `-0` / scientific-notation resolve to their numeric value and are range-checked once. **Both bounds are PINNED CODE CONSTANTS (round-4 + round-8):** `HOST_TEST_TTL_MAX_MS` = 4 × the code-default TTL = **14400000 (4h)**; `HOST_TEST_TTL_MIN_MS` = **300000 (5 min)** — comfortably below any legitimate suite HOLD but above instant-expiry abuse. Deliberately NOT env- or tuning-file-tunable, so reclaimers can never diverge on them (a per-actor ceiling would let a divergent actor prematurely reclaim a peer's legitimately-raised row — the exact cross-actor split §2.9 exists to prevent). The legitimate raise-my-own-TTL case is honored up to the ceiling; no row value can exceed it or fall below the floor. A test that genuinely needs a sub-floor TTL injects it via the internal policy parameter, NEVER the public env. §5 asserts: a peer honors a row legitimately stamped 2h (no premature reclaim); a row beyond the ceiling is reclaimed AT the 4h constant; a row of `1` / `0` / `NaN` / non-integer resolves to the code-default + WARN and does NOT reclaim a young holder (and, arm on, does NOT signal one) — jointly satisfiable precisely because the range brackets the largest supported raise from above and instant-expiry abuse from below.
  - **The 80%-of-TTL warning** is emitted by an **unref'd timer armed in the holding root's own globalSetup** at acquire (fires against the row's re-clamped `ttlMs`, prints to its own stderr + ledger WARN); prune passes additionally ledger `approaching-ttl` as the durable backstop — the warning has a named owner, not a prose wish.
- **Fail-OPEN on reclaim uncertainty** (§1.1): if `df` can't confirm host-local, reclaim is disabled for the pass (runs still admit). If the holders file is corrupt/unparseable, the design **admits AND quarantines**: under the lock, the corrupt file is renamed aside to `host-test-runner-holders.corrupt-<ts>.json` for inspection (never silently discarded; only the newest 5 quarantine files are retained), a fresh empty holders file is created via the single repair path, the event is written to the ledger at WARN level, and the run is admitted. A "poison sanity ceiling" — **PINNED: `HOST_TEST_POISON_CEILING` = 64 rows, a code constant (round-6 adversarial: round 6 made this ceiling load-bearing — unknown-state rows count toward it — so it can no longer be the unreviewed number). Derivation: ~2× the 29-root incident, comfortably above worst plausible legitimate dry-run concurrency (which produces incident-scale live rows by design, §2.11) plus tombstones plus tolerated unknown-state rows — so live soak bookkeeping can never trip it, while genuine row-spam bloat does** — is treated as corruption and takes the same quarantine path, with one obligation preserved: **the quarantine path RE-HOMES any pending `terminating` tombstones into the fresh file** (all three SIGKILL completers re-read the holders file, so a quarantined tombstone would silently drop an armed kill obligation). **The UNPARSEABLE-corruption branch cannot parse the file to extract tombstones, so it best-effort SALVAGES them (round-8 adversarial): a line/regex scan for `terminating` rows carrying a parseable fingerprint (JSON rows are line-structured), re-homing any recovered; and — whether or not salvage succeeds — the quarantine WARN event ENUMERATES that N tombstones may have been dropped, and the next prune/route read surfaces it.** So an armed kill obligation lost to unparseable corruption is at worst LOUD, never silent (the §2.4 durability claim held only on the parseable-ceiling path before). §5 fixtures: a holders file at the ceiling quarantines (tombstones re-homed); one below it does not; a corrupt file with one intact `terminating` line → salvaged + re-homed; a fully-mangled file → WARN names the possible drop. A **foreign-hostname holder on a df-confirmed-local disk is bogus by this design's own host-local contract** — it is dropped (not counted forever) and surfaced loudly as a likely synced-`~/.instar` signal (§2.7).
- **Wedged-lock handling (fail-open granularity — round-2 finding).** A missed lock attempt within the poll loop is "retry next poll", NEVER an admit. The fail-open admit for lock trouble fires only on *provable wedge*: the lock file exists and is older than 10s (the critical section is sub-ms, so a 10s-old lock is wedged regardless of its holder's pid-liveness) — in which case the lock is **race-safely** age-reclaimed — or the lock has been continuously unavailable for the entire wait budget, in which case the run admits with a `fail-open-admit` ledger record. **Race-safe age-reclaim (round-2 scalability/adversarial finding):** several queued pollers can observe the same >10s lock in one window; a naive unlink-then-retake lets B delete A's FRESH lock and both enter the critical section (a holders row is then silently lost to last-write-wins). Reclaim is therefore by ATOMIC rename of the observed lock (`rename(lock, lock.reclaim-<pid>-<nonce>)`) — only the single winner proceeds to create the new lock; every loser sees its rename fail and re-polls. **But rename targets a PATH, not an inode, so the rename alone is not CAS-safe (round-9 codex): between A observing the stale lock and A's rename, B could reclaim it, create a FRESH lock, and A would then rename B's fresh lock aside and wrongly enter — the exact lost-row race, re-created.** The reclaim therefore captures the observed lock's identity BEFORE the rename (`fstat`/`lstat` → `{dev, ino, mtime}`) and, AFTER the rename, `fstat`s the renamed target and PROCEEDS ONLY IF `dev+ino` (and mtime) match the observed object — i.e. it verifies it moved the SAME file it decided was stale, not a fresh lock some peer created in the gap. On a mismatch (it grabbed a peer's FRESH lock created in the race) it does NOT try to rename the file back (round-10 codex: a rename-back is itself racy — a third actor may have created a new lock at the path, or the peer's critical section is mid-flight): it simply drops its claim, leaves the mis-grabbed file under its private `lock.reclaim-<pid>-<nonce>` name (swept by age like any stale reclaim-temp, §2.4 witness/quarantine sweep), and re-polls treating the lock as live. So the winner is the one who both won the rename AND moved the exact stale object; a fresh lock created in the race is never consumed as a stale reclaim. **The residual (a peer's fresh lock briefly moved aside before that peer completes its sub-ms critical section) resolves in the §1.1-SAFE direction and is bounded (round-10 adversarial confirmed):** the worst outcome is a lost holders ROW (last-write-wins on the atomic holders write), i.e. an UNDER-count → over-admission → one extra concurrent suite (the benign direction for THIS lane, unlike the spawn lane), bounded by the 8-slot `O_EXCL` storm ceiling and made auditable by witnesses — never a wedge, never file corruption (the holders write is atomic temp+rename). The dev+ino verify makes the COMMON case CAS-correct; the design's own fail-open posture makes the residual race safe rather than requiring a perfect POSIX CAS the filesystem cannot give. (Dropping age-reclaim entirely and relying on the wait budget + storm ceiling was considered but leaves an orphaned lock forcing every run to wait the full budget before admitting — a real UX cost — so age-reclaim is kept, made correct in the common case and safe-by-§1.1 in the residual.) An admitting root that never took the lock best-effort appends a lockless **witness record** to a `witness/` subdir (frozen path §4) recording `{ pid, hostname, acquiredAt }` so admitted-open runs remain countable by the soak review. **Witness sweep is liveness-gated (round-2 integration finding):** a witness is swept only when its recorded pid is DEAD or its age exceeds the max-hold TTL — never unconditionally, so a still-running admitted-open suite's witness is not erased mid-run.
- **Honesty on the bound during a persistent lock wedge — with the worst case QUANTIFIED (codex external, rounds 3+4).** Witness records make fail-open admissions *auditable*; they do NOT re-impose the count while the lock stays wedged. If the lock is genuinely wedged for the full wait budget, multiple roots can admit together — the bound is temporarily LOST in the fail-open (safe) direction until the wedged lock ages out and is reclaimed. Worst-case analysis: over-admission is bounded by the number of roots that ATTEMPT within the wedge window — on this host's real actor population that is the handful of agent build sessions plus pre-push runs (single digits), NOT the 29-root storm (which was a queue draining unbounded over hours, exactly what admission normally prevents); each open-admitted root still carries its lane's per-run clamp (targeted ≤4 workers), its TTL, and a witness record, so the load is attributable and self-expiring. The original incident's kill cascade came from the watchdogs, which run in INDEPENDENT long-lived server processes — a wedged test-lane lock file does not impair them, and §2.10 separately ensures waiting runs don't read as hangs. The 10s age-reclaim bounds how long a provably-wedged lock can sustain the loss; only a lock that is continuously RE-contended-but-alive for the full budget yields the multi-admit, and that window equals the wait budget by construction. **And the loss has a lockless FLOOR — the storm ceiling (round-5 codex; made ATOMIC round 6):** a fail-open admit must WIN a **numbered witness slot** — `witness/slot-<n>` for n ∈ 1..`WEDGE_STORM_CEILING` (code constant, **8**), each created with `O_EXCL` — so admission at the ceiling is an atomic lockless reservation, not a count-then-act read (round-6 codex: concurrent waiters could all observe 7 and all admit; with `O_EXCL` slots exactly one winner claims each slot). A run that cannot claim any slot (all 8 held by pid-ALIVE claimants — a dead-pid or over-TTL slot is sweepable and reclaimable, same liveness-gated sweep as ordinary witnesses) REFUSES with the typed error enumerating the held slots' pids + ages (round-6 security: one read distinguishes a genuine storm from planted records) instead of admitting. This is a deliberate, narrow fail-direction flip: §1.1's harm comparison holds at small over-admission (one extra suite is minutes of CPU), but at 8+ simultaneous open-admitted suites the harm ordering inverts — that IS the meltdown this spec exists to prevent, and at that point wedging further pushes is the lesser harm. The ceiling is reachable only during a sustained lock wedge with storm-level attempt volume (never in normal operation, where the lane caps bind first) — specifically it covers the one case the 10s age-reclaim CANNOT (round-6 gemini asked why both exist): age-reclaim fires on a STALE lock; a lock that is being actively re-taken and re-released under heavy contention for the full wait budget never ages past 10s, and THAT is the multi-admit path the slots bound. §5-tested + §6-tabled. This is the correct trade (a wedged-lock wedge must never block the dev loop — until the alternative is the storm itself), stated plainly rather than papered over.

### 2.5 Re-entrancy — ancestry + holders cross-check, NOT a bare inherited env flag

A test can itself spawn a child vitest; without a skip it would deadlock waiting for a slot its parent holds. The spawn-boundary makes a bare inherited `process.env` flag unsafe on two counts: (a) vitest runs the *test* in a WORKER whose env is a snapshot — a `process.env` mutation made in the root's globalSetup is NOT guaranteed to reach the worker (pool/version-dependent; threads copy env unless `SHARE_ENV`), so the child's globalSetup may see no flag and deadlock against its own parent; (b) a leaked/exported flag in a persistent shell silently disables the bound host-wide (fail-open in the *wrong* place — a real second root skips).

The skip is therefore **corroborated, holder-scoped, and cross-checked** — the ancestry+holders check is the AUTHORITY; the env marker is an optimization hint:

- The marker is `INSTAR_TEST_SEMAPHORE_HELD=<root-pid>:<slot-id>`, injected into worker env explicitly via each config's `test.env` (not left to ambient inheritance), so it reliably reaches the worker that spawns the child.
- **The actual skip algorithm (works with OR without the marker) — LANE-SCOPED (round-4 adversarial HIGH):** the child walks its own ancestor pid chain (`ppid` recursion via `ps`), intersects it with the pids in the live holders file, and skips iff a live ancestor holds a current slot **in the SAME LANE the child itself would acquire**. Lane-blindness was a cap-1 bypass: a suite-class child under a TARGETED-lane holder would have skipped the suite lane entirely — sheltering full-suite children under targeted slots that never touch the cap-1 lane (a full-pool amplification the round-8 unconditional ≤4 clamp below now closes on its own, but the lane-scoped skip is still required so a suite-class child correctly QUEUES on the suite lane rather than skipping it). The deadlock rationale for the skip holds only SAME-lane (a child blocking on a slot its own ancestor holds); a suite-class child under a targeted-only ancestor blocks on a slot held by UNRELATED runs — ordinary queuing that resolves — so it acquires the suite lane normally. When the marker is present it only narrows the check (the named pid must be in the ancestor chain AND hold a same-lane slot); a stale/foreign/leaked marker fails the cross-check and does NOT skip. A scrubbed-env child therefore still skips via pure ancestry — the §5 meta-test asserts exactly this.
- A **process-global in-memory flag** ensures one process never takes more than one slot even if multiple prepended globalSetups run in it (workspace/aggregate runs) — and it is **LANE-SCOPED like the ancestry skip (round-5 adversarial):** the flag records the held slot's LANE; a further globalSetup in the same process skips only SAME-lane (a targeted-first process whose later config is suite-class acquires the suite lane, or takes the nested-clamp path) — otherwise a first-evaluated targeted config would shelter a later unclamped suite-class project in the same process, the round-4 lane-blindness shape at the in-process surface. The §5 guard-test additionally FAILS on the introduction of any `vitest.workspace.*` file until the workspace path is reviewed against this flag.
- **ALL nested children run CLAMPED to ≤ 4 workers — no reservation machinery (round-8: both external families + the complexity meta-signal, converging).** The skip is all-or-nothing per process, so a single slot-holding root could otherwise shelter N nested full `vitest run` children (each skipping via ancestry, each fanning a full worker pool = N× amplification under one slot). Rounds 2–5 bounded this with an atomic per-slot reservation (pid-keyed sub-records, an `INSTAR_HOST_TEST_NESTED_MAX` concurrent cap, config-eval reserve-and-observe under the holders lock, a lifecycle teardown/sweep, and sub-count sanity coercion). **Round-8 review retired that machinery.** codex (SERIOUS) and gemini independently — plus the round's shared meta-finding that "the design has crossed into high ceremony … the main risk is implementation complexity creating the next incident" — established that the reservation was itself the dominant risk: a mini-scheduler (ancestry walking, config-time reservations, pid-keyed lifecycle, poisoned-sub-count handling, cross-lane cycles) guarding a rare case, for a benefit (letting the first one or two nested children keep a FULL worker pool) that is trivial. The convergent fix — PRE-AUTHORIZED in round 6 and now ADOPTED as the default: a nested child (detected by the lane-scoped ancestry+holders check above) still SKIPS acquisition — never blocks (blocking a nested child on a slot its ancestor holds is the deadlock the skip exists to prevent) — but ALWAYS at the config-eval maxWorkers ≤ 4 clamp, **unconditionally**. There is no nested count, no reservation, no `NESTED_MAX`, no sub-record lifecycle, and no sub-count-poisoning surface: every nested child is clamped, so N concurrent nested launches are bounded to N × ≤4 workers with ZERO coordination. This deletes an entire class of complexity AND several round-2..5 attack surfaces (reservation races, sub-count poisoning, leaked-reservation sweep) in one move. The cost — a nested child never gets a full pool — is negligible: nested test children are small (the §5 meta-tests are exactly this shape and run fine at ≤4 workers), and the old design already accepted ≤4 for every child past `NESTED_MAX` anyway.
  - **Config-eval is still the clamp point (§2.3):** the clamp can only be applied where a config can reshape the pool before fanout, so the nested child applies the ≤4 clamp during its own config evaluation; globalSetup's skip decision then consumes ONLY the lane-scoped ancestry result — there is no reservation to observe, so the two-point race the old reservation existed to close no longer exists (nothing is counted). Run-shapes that will not execute a suite (`off`/CI, argv-visible `--watch`/list) short-circuit as before.
  - **The nested ≤4 clamp MUST be CLI-proof — reservation deletion re-opened the CLI-precedence bypass on the nested path (round-9 adversarial HIGH regression, the load-bearing round-9 fix).** §2.3 closes the CLI-precedence bypass (`--maxWorkers=32` outranks a config-set `maxWorkers`) for the two LANES by ROUTING a resolved-pool-\>4 run into the cap-1 suite lane "by construction." A nested child SKIPS acquisition (it takes no lane, by design, to avoid self-deadlock), so lane-routing cannot bound it — leaving the config-eval clamp as its SOLE bound, and that clamp is exactly what CLI flags defeat. Under the round-2..7 reservation design the NESTED_MAX count at least bounded HOW MANY such children ran; deleting it (round 8) left a pool-flag nested child bounded by nothing — not the clamp (CLI-defeated), not a slot (skipped), not a count (removed) — i.e. `vitest run x.test.ts --maxWorkers=32` spawned under a slot-holder is unbounded amplification, the fork-storm this spec exists to prevent. So the nested clamp is made CLI-PROOF the same way §2.3 handles pool-shaping argv, applied at the ONE point a config controls: a detected-nested child's config-eval **NEUTRALIZES recognized pool-shaping argv** (`--maxWorkers`/`--minWorkers`/`--pool`/`--poolOptions.*`/`--fileParallelism`/`--no-isolate`) — stripping them from `process.argv` before vitest's CLI parser reads them — AND clamps `maxWorkers`/`poolOptions.*.max*` to `Math.min(resolved, 4)` (round-10 lessons-aware: a clamp, not a set-to-4, so a nested run that legitimately requested FEWER than 4 workers keeps its lower value; the bound is a ceiling of 4, never a floor), so no CLI value survives to raise the pool above the clamp. This rides the SAME "does a config-time action reach before vitest materializes the pool?" validation as the acquire-before-fanout invariant (§2.2 item 5 / §5): it is PROVEN per pinned vitest version by a meta-test that spawns a nested `vitest run --maxWorkers=32` and measures ACTUAL concurrent workers ≤ 4 (parity with the targeted-lane state-check), and if a version cannot guarantee the neutralization reaches before pool fanout, the `guarded-vitest.mjs` wrapper fallback (which strips the flags for spawns that route through it) is wired and the residual (a raw ad-hoc nested `npx vitest --maxWorkers=N` that bypasses both) is documented — mirroring §2.3's own accepted ad-hoc residual. A nested child whose RESOLVED pool STILL exceeds 4 after neutralization (a novel pool flag the list missed) is ledgered a loud `nested-skip` with `clamped:false, poolOverride:true` at WARN — the same soak-visible "unguarded/anomalous nested run" signal — so a miss is loud, not silently unbounded.
  - **A nested skip is ledgered `nested-skip` with the sheltering root pid + slot-id + `clamped:true`** (§2.8) so nested activity stays attributable in the soak data. The ONE remaining edge case: an unguarded config that reaches globalSetup's skip path WITHOUT the config-eval clamp having run still skips (the deadlock rule leaves no alternative) but ledgers `clamped:false` at WARN — the same "an unguarded config shows up loudly rather than passing silently" signal the old `reserved:false` carried, now the only nested edge case. The §5 config-list guard-test covers BOTH per-config seams (below).
  - **Lock trouble at config-eval no longer touches nesting:** there is no reservation to take, so a lock miss/wedge at a nested child's config-eval simply applies the ≤4 clamp (its unconditional default) and proceeds — it can neither skip-unclamped (the round-2 mass-amplification hole is closed by construction, since NOTHING nested is ever unclamped) nor block (self-deadlock).
  - **Cross-lane honesty (round-5 lessons-aware):** the lane-scoped skip can still, in a contrived population (a suite holder spawning a targeted child while every targeted holder spawns suite children), form a bounded hold-and-wait cycle that no run's completion breaks — it resolves by the shortest acquire budget's typed capacity-timeout, the named cycle breaker (unchanged). But with every nested child now clamped, the amplification term is a uniform ≤4 workers per nested child regardless of lane — the cross-lane case carries no full-pool shelter.

  Worst-case nested amplification under one slot is now (number of nested children) × ≤4 workers — each child provably ≤4 by the CLI-proof clamp above, attributable via the `nested-skip` ledger event (sheltering root pid + slot-id). **The remaining exposure is the COUNT, and it is bounded by TEST AUTHORSHIP, not adversarial input (codex round-8/9 — stated as an explicit worst-case assumption, not left as residual prose).** A nested child is spawned by a parent's own test CODE (`execa('vitest', …)` inside a suite this team wrote), never by untrusted external input — so a test that spawns 50 concurrent nested runs is a test-hygiene bug of the author's own making, surfaced loudly by the per-child `nested-skip` attribution (sheltering root + count visible in the soak/route), fixable by that author. The semaphore deliberately does NOT block or count nested children (blocking deadlocks; counting is the reservation machinery round 8 retired as the dominant risk): the design bound is **per-child ≤4, CLI-proof**; the aggregate is a stated assumption (typical nesting is 1–2 children — the §5 meta-tests' shape), not a hard cap. A genuinely pathological nested-spawning test is the author's bug, made loud, not a semaphore-blockable adversarial vector.
- §5 meta-tests: a REAL nested `vitest run` from inside a WORKER (not two roots) asserts the child SKIPS and its ACTUAL concurrent worker count ≤ 4 (measured, not flag-asserted — parity with the targeted-lane meta-test); **a nested `vitest run x.test.ts --maxWorkers=32` asserts its ACTUAL concurrent worker count is STILL ≤ 4 (the round-9 CLI-proof test — the pool-shaping argv is neutralized at config-eval; parity with the targeted-lane CLI-precedence state-check at §2.3)**; a SCRUBBED-env child asserts the ancestry path still skips it (env alone would fail); a nested FULL-SUITE child under a TARGETED-lane holder asserts it ACQUIRES (or waits on) the suite lane — never skips (the lane-scope test); an unguarded-config nested child (reaches globalSetup without the config-eval clamp) asserts it skips with `clamped:false` at WARN.

### 2.6 Exemptions, kill switch, and the recovery lever

- **Kill switch — env only.** `INSTAR_HOST_TEST_SEMAPHORE=off` is the SOLE chokepoint kill switch. Config `enabled:false` is NOT a chokepoint lever (the globalSetup runs in a bare test process that can't know which agent's config to read — the config lever would silently no-op, worse than no lever). The config `intelligence.testRunnerCap` exists only to tune the *route's report* and any server-launched tooling; §6 and this section state env-only rollback honestly. **An `off` skip is never silent beyond its own process:** every skip appends a ledger event (§2.8), the route surfaces a skip-reason histogram, and a sustained 100% `off` rate on a dev host grades the limiter `diverged` in the guards inventory (§4) — the 2026-06-05 lesson (a silently-disabled guard explains more incidents than a broken one) applied to this guard from day one.
- **CI exemption — hardened, and graded like `off` when spoofed (round-2 adversarial finding).** Skip only when `CI === 'true' || CI === '1'` AND a positive CI signal is present (`GITHUB_ACTIONS` / `RUNNER_OS`) — a stray local `CI` export (or `CI=false`, which is truthy) must not disable the dev-host bound. Documented assumption: GitHub-hosted ephemeral single-suite runners; a self-hosted shared runner running matrix jobs concurrently is out-of-scope (stated, not silently unbounded). Because an agent could still export `CI=true GITHUB_ACTIONS=true` on a dev host to buy a labeled-innocent skip, a sustained `CI`-reason skip on a known-non-CI host is graded IDENTICALLY to sustained `off` — `diverged`, with the standard aggregated attention item.
- **Serverless-host honesty (round-3 adversarial finding).** The `diverged` GRADE and its attention item are server-side, but the chokepoint runs on bare builder/worktree checkouts that ARE the incident population and often have NO running agent server — so a server-only surface would be silent on exactly the hosts that matter (the 2026-06-05 "a silently-disabled guard explains more incidents than a broken one" lesson, unmet where it counts). Two out-of-process surfaces cover the serverless host: (a) the globalSetup ESCALATES the severity of its always-printed skip line to a loud stderr WARN specifically for the `off` and `CI-on-a-non-CI-host` reasons (a self-disable is never a quiet one-liner); (b) `instar dev:preflight` and the pre-push hook read the LOCAL ledger and WARN on a sustained self-disable pattern — **WARN-only on the pre-push hook, NEVER a push-blocking fail (round-4 security):** the detection surface for a self-disable must not itself become a new way to wedge the dev loop (§1.1's false-BLOCK inversion), and the ledger is a best-effort, concurrently-appended, same-user-writable file — its content may only ever ADD a warning, never block a push; the parser tolerates torn/malformed lines. `dev:preflight` — an advisory tool a human runs deliberately — MAY fail on the same pattern. The server-side `diverged` grade (which needs the source-of-truth "is this host a CI runner?" — a long-lived agent server knows it is not an ephemeral CI job) is the additional surface WHERE a server exists, stated honestly as such, not the only one.
- **Watch / list exemption — kept, but NEVER quiet (round-4 adversarial HIGH).** Detect via the globalSetup context `config.watch` (vitest v2+ context arg) with an argv fallback; a watch session skips (a long-lived interactive tool must not hold a slot for hours — and having its lingering watcher HOLD a suite slot would wedge the cap-1 lane, the §1.1 false-BLOCK direction, so the exemption stands rather than an initial-run acquire). But round-4 review established two ways this was the cheapest remaining cap-dodge, both now surfaced: (1) **bare `vitest` in a TTY DEFAULTS into watch mode** — agent tmux panes have TTYs, so the incident population reaches the exemption *accidentally*. A run that defaulted into watch (no explicit `--watch`/`-w` in argv) always gets the loud stderr WARN + a "use `vitest run`" hint, never the quiet one-liner. (2) Adversarially, `--watch` is a labeled-innocent full-suite skip — so ANY watch skip in an agent-launched/non-interactive context (the `INSTAR_HOST_TEST_RUN_CLASS` hint, agent env markers) also escalates to the loud WARN, watch skips are a named soak-review metric (§4), and a SUSTAINED watch-skip pattern on a dev host is graded like `off` — `diverged`, same attention item, and the same `dev:preflight`/pre-push (WARN-only) ledger pattern check as (b) above. A deliberate human `--watch` in an interactive shell keeps its quiet skip line. A collect/list invocation (`vitest list`, used by `pre-push-smoke.mjs`) is no-op'd on the same seam so a mere listing doesn't wait for or consume a slot.
- **Every skip prints one deterministic line** naming the reason (`off` / `CI` / `watch` / `list` / `reentrant` / `fail-open-admit`) AND the resolved posture (`dry-run` / `enforcing`), and appends the same to the event ledger — an inert bound is never silent (a fork-storm debugged with the guard off must be observable, and a mixed-posture host must be detectable per-run). **`targeted-run` is NOT a skip reason (round-4: three reviewers, converging):** a targeted run ACQUIRES a targeted-lane slot (§2.3) and is ledgered as an `acquire` event with run-class + file count (§2.8) — its per-run console line is the targeted-lane ACQUIRE announcement (mirroring the suite-lane acquire line), never a `skip` kind. Emitting both would double-count targeted activity in the skip histogram and pollute the §4 soak evidence.
- **Distinct capacity-timeout signal.** The timeout throws a typed error with a DISTINCT exit code and an unmistakable message: "could not START within budget — this is NOT a test failure; N holders: [pid, age]; levers: INSTAR_HOST_TEST_SEMAPHORE=off, INSTAR_HOST_TEST_MAX". The per-minute wait line goes to STDERR and carries an active-work indicator recognized by the silence/load-stall sentinels (§2.10). CLAUDE.md tells agents a rejected push may be contention, not red tests — and that a `git push` run under an outer command timeout needs that timeout ≥ the pre-push acquire budget (a correctly-waiting push must not be killed by its own caller; see §2.9 budgets).
- **Priority/timeout split.** Interactive- and pre-push-launched runs get the longer wait budget (user-blocking, few); background `/build` verification runs get the short fail-loud budget. The class is derived from the launching config (push config, or `INSTAR_HOST_TEST_RUN_CLASS=background` set by the server when it launches build tooling), never from user input. Defaults in §2.9. Admission remains barging (no FIFO claim, §3); the starvation risk this creates for the push class at cap=1 is a named soak-review item — the ledger's would-wait data (§2.8) is collected precisely so a starvation signature (push-class waits trending toward ceiling) is visible BEFORE the enforce flip, and the pre-authorized cheap mitigation is an aging/waiter-ticket refinement or push-class priority admission, buildable post-soak without a new convergence round.
- **Recovery lever (the 2026-07-01 lesson).** `POST /test-runner-limiter/prune` forces a full reclaim pass — the surfaced action replacing hand-edits of JSON. Bearer-auth, POST-only, never on any unauthenticated allowlist, single-flight (concurrent calls coalesce), rate-limited (one forced pass per 5s), and its response enumerates what was reclaimed vs would-be-reclaimed. This route and the globalSetup acquire path are the ONLY two places persistent reclaim (row removal, corroborated signaling) happens — the GET route is pure (§2.7).

### 2.7 Observability route (parity + no-lie constraint)

`GET /test-runner-limiter` (Bearer-auth, GET-only, never on the `/health` allowlist) → `{ cap, targetedCap, posture, clampActive, ttlSignalArmed, liveHolders: [{pid, hostname, acquiredAt, ttlMs, state}], targetedHolders, admittedOpen: [{pid, acquiredAt}], suite: {available, saturated}, targeted: {available, saturated}, recentEvents, skipHistogram }` (`clampActive` reflects the optional §4 clamp-active soak sub-stage — an additively-evolvable detail field, not part of the frozen top-level shape) — availability is **PER-LANE** (round-4 integration: a singular `available`/`saturated` cannot honestly answer "why is my targeted run waiting?" when the targeted lane is full but the suite lane is free — the exact lie class the no-lie constraint below prohibits; §5's integration test asserts the targeted-lane saturation flips when ONLY that lane is full) — matching the `/spawn-limiter` shape decision where applicable (documented divergence: `/spawn-limiter`'s `waiters`/`acquireMs`/`waitersMax` are in-process-singleton stats the out-of-process test globalSetup doesn't share, so they are omitted rather than faked; `liveHolders` is an array here — the honest live-state read). `admittedOpen` is derived from live (pid-alive) witness records so a lock-wedge over-admission is *visible*, not merely ledgered (round-2 integration finding — the §2.4 "countable by the route" claim now has a backing field). `posture` is `'off' | 'dry-run' | 'enforcing'` and `ttlSignalArmed` reflects `INSTAR_HOST_TEST_TTL_SIGNAL`, both resolved through the IDENTICAL resolver the chokepoint uses (§2.9), honestly labeled as the server-process view (a shell whose env diverges is exactly what the per-run ledger posture stamps detect). A CapabilityIndex entry for the `/test-runner-limiter` prefix ships with the route (parity with `/spawn-limiter`; `instar dev:preflight` checks discoverability).

**No-lie constraint (integration H1):** the route resolves `cap` and `posture` through the IDENTICAL resolvers the globalSetup uses (env → flip-file → code-default), reading the holders file only for live state — NEVER from `intelligence.testRunnerCap.maxConcurrent` (which the enforcing process doesn't read; reporting a config cap while runs are bounded at the code-default would make the "why is my run waiting?" answer a lie). The status read is **pure — lock-free, write-free, signal-free** (read + parse + count-with-pid-liveness computed in memory for display; dead/TTL-expired holders are excluded from the reported live count as a VIRTUAL prune, but the file is never written and nothing is ever signaled from a GET). All persistent reclaim is confined to the acquire path and `POST /prune` (§2.6) — this supersedes any reading of "prune-on-read" as a physical write; a GET that could kill a process is an unacceptable surface even Bearer-authed.

**Security on the route (untrusted holder fields):** `hostname`/`id`/`cmd` are untrusted (a poisoned holders file can carry markup). They are clamped to a sane charset + length at the route projection and HTML-escaped on any render; verify whether the existing `/spawn-limiter` dashboard render escapes and fix the sibling if not. `~/.instar` is created `0o700` (owner-only holder metadata). Docs state: do NOT place `~/.instar` on a userspace-synced path (Dropbox/iCloud/Syncthing) — `df -P` reports such a home as local, and a synced holders file cross-counts foreign-hostname holders between machines (the §2.4 foreign-on-local-disk drop + loud surface is the mitigation).

### 2.8 Durable event ledger (the soak's evidence store — and the audit trail for every signal)

Every consequential decision appends one JSON line to `~/.instar/host-test-runner-events.jsonl` (frozen path; `0600`; size-bounded with rotation ~5MB; written OUTSIDE the holders lock; best-effort — a ledger write failure never blocks or fails a run). **Rotation must not eat the soak evidence (round-6 scalability):** at plausible dry-run event rates (~every targeted run emits `would-clamp` + `acquire`/`release`) 5MB covers ~6–10 days — SHORTER than the 14-day soak window every §4 flip criterion is computed over. Rotation therefore moves the live file aside to timestamp-named segments (`host-test-runner-events.<ts>.jsonl`) that are RETAINED until the enforce-flip decision is recorded (thereafter a newest-10-segments floor applies); the §4 soak review reads the SEGMENT SET, not just the live file, and §5 asserts a rotated-out early-window event remains readable by the review. Segments join the §6 inert-artifacts list. Event kinds: `would-block` (dry-run; carries the live holder set that would have blocked), `block` (enforcing), `stale-holder-reclaimed` (the DEFAULT TTL action — slot freed, NO signal), `approaching-ttl` (80%-of-TTL backstop), `signal-term` / `signal-kill` (only under the opt-in TTL-signal arm; each with target pid, holder row, corroboration + group-leadership result), `reclaim-dead`, `reclaim-mismatch` (corroboration failed — slot freed, no signal), `fail-open-admit` (with cause), `skip` (with reason + posture), `nested-skip` (with sheltering root pid + slot + `clamped:true|false` — `false` flags an unguarded config that missed the config-eval clamp, §2.5), `would-clamp` (dry-run; targeted and nested variants with the natural resolved pool bound, §2.11), `schema-unknown` (an unknown-state row excluded from the cap count, §2.9), `tuning-changed` (first observation of a tuning-file content-hash change, with the differing fields old → new, §2.9), `quarantine` (corrupt file), `acquire`/`release` (with timestamps + the resolved `cap`/`targetedCap` + run-class + the resolved targeted file count when run-class is targeted + the tuning-file content hash, so per-class would-wait/realized-wait distributions AND cap/posture/tuning divergence are derivable offline — round-2 integration finding: divergence in the effective cap must be as visible in the soak data as posture divergence).

**Warning-fatigue bound (round-5 codex):** every loud stderr WARN in this design is once-per-condition-per-RUN by construction (each is emitted at one decision point in a short-lived root process — there is no loop that re-prints it), the once-a-minute wait line is the only recurring line and is already rate-fixed, the ledger is rotation-bounded (~5MB), and the route's histogram — not raw stderr — is the aggregation surface for patterns. A busy host's steady state is therefore quiet-by-default with one line per consequential event, not a warning wall.

This single ledger is simultaneously: (a) the **soak evidence** — "zero false would-blocks over the window" is read from here, not from scrolled tmux stderr (bare test processes are ephemeral; without a durable sink the ratified flip criterion is unmeasurable); (b) the **kill audit** — a wrong SIGTERM is forensically visible (which pid, which holder row, which corroboration outcome, fired by whom), answering "loud, audited" with a mechanism the out-of-process chokepoint can actually reach (no server logger, no Bearer token needed — it's a file append); (c) the **abuse surface** — classification gaming and `off`-lever abuse show up as ledger patterns the soak review and the route's histogram surface. The GET route serves a bounded recent tail; the ledger is machine-local like everything else here.

### 2.9 Config surface, env authority, the enforce flip, and deployment

`.instar/config.json` → `intelligence.testRunnerCap`: `{ enabled, maxConcurrent, acquireWaitMs }` (tunes the route report + server-launched tooling only; NOT the chokepoint kill switch — §2.6). Env is a per-process tuning OVERRIDE; the host-uniform authority for EVERY lever that must agree across all actors (the two caps — suite and targeted — plus posture and the signal arm) is the host-local tuning file (below). (The TTL host-max/min ceilings are deliberately NOT among them — both are code constants, §2.4; and there is no nested cap since round 8 — nested is unconditionally clamped, §2.5.)

- `INSTAR_HOST_TEST_MAX` — suite-lane cap, per-process override; host-uniform authority is the tuning file; code-default **1** (ratified §4).
- `INSTAR_HOST_TEST_TARGETED_MAX` — targeted-lane cap, per-process override; host-uniform authority is the tuning file; default **6** (§2.3).
- *(`INSTAR_HOST_TEST_NESTED_MAX` was REMOVED in round 8 — nested children are now unconditionally clamped to ≤4 workers with no per-slot count, §2.5, so there is no nested cap to tune.)*
- `INSTAR_HOST_TEST_ACQUIRE_MS` — background-class SUITE-lane wait budget, default **120000** (2 min, fail-loud). The interactive/pre-push budget resolves as: `INSTAR_HOST_TEST_ACQUIRE_MS_INTERACTIVE` if set, ELSE 5 × the resolved background budget (env or default; note 5×120000 already = the 600000 default — the third clause is a harmless floor). Queue-depth math: at cap=1 the expected wait is (queue depth) × (suite + prepended dist build); the interactive ceiling must cover a depth-1 queue of the p95 suite HOLD duration — an assumption §4 validates against real ledger data before the flip, not an assertion (round-2 scalability finding).
- `INSTAR_HOST_TEST_TARGETED_ACQUIRE_MS` — TARGETED-lane wait budget, default **60000** (1 min, fail-loud/THROW — §2.3; a targeted run whose lane is full throws the typed capacity-timeout, never fail-open-admits).
- `INSTAR_HOST_TEST_TTL_MS` — max-hold ceiling, default 3600000; stamped per-row at acquire (§2.4).
- `INSTAR_HOST_TEST_TTL_SIGNAL` — the signal arm, with an ASYMMETRIC env override (round-6 adversarial MEDIUM — the last quiet twin, closed structurally): **env can only DISARM, never ARM.** `INSTAR_HOST_TEST_TTL_SIGNAL=0` is honored per-process (the weaker direction is always available); `=1` against an authority (tuning file, else code-default) that is UNARMED is **IGNORED + loud stderr WARN + WARN ledger event** — arming the one lever that kills processes is tuning-file-only, so a leaked/inherited env context carrying `TTL_SIGNAL=1 ENFORCE=1` is structurally incapable of running kill-mode reclaim against an unarmed host (no surfacing race to lose — the dangerous direction does not exist per-process). The resolved arm is stamped into every ledger event alongside posture and caps; arm-divergence joins the preflight/pre-push (WARN-only) pattern list and the server-side diverged grading; §5 asserts env-armed-vs-unarmed-authority resolves to UNARMED + WARN; §6 row added. (Default remains OFF — the DEFAULT reclaim never signals; §2.4.)
- `INSTAR_HOST_TEST_SEMAPHORE=off` — kill switch (§2.6).
- `INSTAR_HOST_TEST_ENFORCE` — per-process posture override (§ below).
- `INSTAR_HOST_TEST_RUN_CLASS=background` — the server-set class hint (§2.6).
- `INSTAR_TEST_SEMAPHORE_HELD=<pid>:<slot>` — internal re-entrancy marker (§2.5).

**Host-uniform levers live in a FILE, not a fleet of env exports — for EVERY cross-actor lever, not just the suite cap (round-2 + round-3 integration findings).** Test roots launch from heterogeneous env contexts (interactive shells, husky pre-push inheriting an agent tmux env, launchd-spawned server children that never source shell profiles) — a per-surface env var yields a SPLIT host and corrupts the soak evidence: "multiple agents must agree" is willpower, not structure, if `INSTAR_HOST_TEST_MAX=2` only reaches the shell that exported it (leaving launchd/tmux descendants enforcing cap=1 against the same holders file — the bound splits into per-surface sub-bounds). This applies EQUALLY to every lever that is enforced against a SHARED holders lane: posture, the suite cap, the **targeted cap**, and the signal arm are all cross-actor (round-3 finding: a lone `INSTAR_HOST_TEST_TARGETED_MAX=100` export makes one actor keep admitting targeted runs while others enforce 6 — the effective host cap becomes the MAX across actors, re-uncapping the shard storm). (The nested clamp is no longer a cross-actor lever — every nested child is clamped ≤4 regardless of any count, §2.5, so there is nothing for actors to disagree on.) The host-uniform authority is therefore ONE frozen-path tuning file `~/.instar/host-test-runner-tuning.json` (`{ v, enforcing?, clampActive?, maxConcurrent?, targetedMax?, ttlSignal?, flippedAt, by }` — `clampActive?` is the optional §4 clamp-active soak sub-stage: clamps real, blocks off): one file write sets any of them for the WHOLE host uniformly. The matching env vars remain highest-precedence PER-PROCESS overrides for testing/emergency scoping. Resolution order for each: env override → tuning file → code-default. Every run stamps its resolved posture AND all resolved caps into its ledger events, so a divergent-env process is visible in the soak data.

**Cap values are sanity-clamped on read, and cap-inflation is as LOUD as `off` (round-4 adversarial + security).** A lone `INSTAR_HOST_TEST_MAX=50` export (or an absurd tuning-file value) is the quiet twin of the kill switch — the effective host cap is the max across actors, yet it produces no skip event and dodges every `off`-keyed surface. Three closures: (1) tuning-file cap values are read through **PINNED sanity ceilings — code constants, 4× each cap's code-default (the `HOST_TEST_TTL_MAX_MS` precedent): suite ≤ 4, targeted ≤ 24** (there is no nested cap since round 8) — an out-of-range/non-integer value resolves to the code-default and ledgers a WARN (round-5 adversarial: an unpinned "sane max" would make the ceiling itself the unreviewed number); (2) the chokepoint prints the loud stderr WARN whenever its RESOLVED cap exceeds the host-uniform authority (tuning file, else code-default) by more than 4× — an env override stays legal for emergency scoping but is never silent beyond that factor; (3) `dev:preflight` and the pre-push hook (WARN-only, §2.6) flag a sustained resolved-cap-divergence pattern from the ledger's per-run cap stamps, and beyond-sane resolved caps join the server-side `diverged` grading where a server exists.

**Posture divergence gets the IDENTICAL loudness treatment — in BOTH directions (round-5 adversarial MEDIUM + round-8 adversarial: env can silently ARM enforcement too).** A per-process `INSTAR_HOST_TEST_ENFORCE=0` while the host authority says enforcing admits past the cap with full bookkeeping; conversely a leaked/inherited `INSTAR_HOST_TEST_ENFORCE=1` while the host authority is still dry-run SOAKING silently converts descendants' runs to ENFORCING — producing real capacity-timeout BLOCKS of pushes during the very phase whose purpose is proving zero false blocks, AND contaminating the soak's would-block-vs-block evidence. Since §1.1 declares false-BLOCK the catastrophic direction, env-STRENGTHENING is arguably the MORE dangerous divergence, not the less. So the WARN is bidirectional: the chokepoint prints the loud stderr WARN whenever its resolved POSTURE DIFFERS from the host-uniform authority in EITHER direction — weaker ("admitting past the host cap") or stronger ("enforcing while host authority is dry-run — your blocks are not host policy"). Both join the `dev:preflight`/pre-push (WARN-only) sustained-pattern list AND the server-side `diverged` grading; §5 asserts both directions. (Env `ENFORCE=1` is still HONORED per-process — tests may need it — but never quietly.) And the tuning file — the enforcement AUTHORITY — gets integrity handling at least as strong as the holders file it governs: an unparseable/corrupt tuning file is **quarantined aside + WARN-ledgered + resolved to code-defaults** (never a silent fallback; §6 row), so a deleted or mangled authority cannot silently revert the whole host to dry-run.

**Tuning-file mutations are structurally visible — the announce rule is a mechanism, not prose (round-5 adversarial MEDIUM; baseline semantics PINNED round 6).** Every acquire/skip ledger event carries a short **content hash of the resolved tuning file**, and the last-observed hash is persisted in a tiny **baseline marker file** `~/.instar/host-test-runner-tuning-baseline.json` (frozen path — a marker survives ledger rotation and needs no backwards ledger scan; round-6 integration). Detection semantics: a run whose current hash differs from the MARKER's emits a `tuning-changed` ledger event + the loud stderr WARN (naming which fields differ, old → new) and updates the marker (atomic temp+rename); an ABSENT tuning file hashes to a defined sentinel, so file CREATION (e.g. the enforce flip) is detected as a change; **no-prior-baseline (fresh host, deleted marker) consults the local LEDGER TAIL before establishing silently (round-8 adversarial — pre-seed-by-DELETION is cheaper than forging).** Deleting the marker + editing the tuning file would otherwise silently establish the poisoned state as the baseline (a cap raise, posture weaken, even a `ttlSignal` arm) with no `tuning-changed` event ever — deletion reaches the same outcome as forging, more cheaply, and also covers an accidental state-cleanup wiping markers. The distinguisher is already free: every acquire/skip event stamps the resolved tuning hash. So on no-prior-baseline the run does a bounded, torn-line-tolerant read of the ledger tail — **spanning the newest rotated SEGMENT(S) when the live events file carries no hash-stamped events (round-9 adversarial LOW): a marker-absence coinciding with a just-rotated near-empty live ledger would otherwise read an empty live file and silently establish the just-edited state, missing the very `tuning-changed` WARN the mechanism exists for; the soak review already reads the segment SET (§2.8), so the baseline read reuses it — the most-recent hash-stamped event across the live file PLUS the newest segment**: if any prior event carries a DIFFERENT tuning hash, it emits `tuning-changed` + the loud WARN instead of silently establishing; a genuinely fresh host (no live file AND no segments carrying hash-stamped events) still establishes SILENTLY (preserving §2.8's fresh-host fatigue bound). And a quiet `tuning-baseline-established` event is ledgered in ALL cases, so marker resets are visible to the preflight/soak pattern checks. So ANY edit — enforce flip, cap raise, ttlSignal arm, hand edit, marker-delete-then-edit — is detected by the next run on the host, regardless of who wrote it or whether they self-reported in `by`/`flippedAt`. **Honesty scope (round-6 security):** this catches unannounced, accidental, buggy, and delete-to-reset edits; a same-user adversary who also forges the marker AND rewrites the ledger tail defeats it — as with every same-user-file surface in this design, that is the stated threat floor, not a gap. Where a server exists, `/guards` and the route surface an armed `ttlSignal` and any above-code-default cap as distinct rows (change-detection, not just current-state). Arming `ttlSignal` via the tuning file — the strictly more dangerous lever — is additionally ANNOUNCED to the operator the same way the enforce flip is (§4's flip announcement), with the hash mechanism as the structural backstop when the announce step is skipped. **The arm announcement NAMES the sleep-wake residual (§2.4 gate 5):** on a mobile host where a boot/wake signal is unobtainable, a sleep gap can push a healthy resumed suite past TTL and the four gates cannot distinguish it — an operator arming `ttlSignal` on a laptop accepts that residual explicitly.

**Tuning-file writes are atomic, and quarantine requires CONFIRMED corruption (round-6 integration).** Every tuning-file write goes through atomic temp+rename (the recommended writer is the CLI helper / Bearer route, never a raw shell redirect — a truncate-then-write edit races cold-starting roots). A first bad READ resolves to code-defaults FOR THAT RUN without touching the file; the file is quarantined aside (`host-test-runner-tuning.corrupt-<ts>.json`, keep-newest-5 — §4 frozen path, §6 inert artifact) only after corruption is CONFIRMED by an immediate re-read — so a transient torn-write race can never demote the host authority, while genuinely persistent corruption still fails loud to code-defaults (§5 torn-write fixture). Valid cap values are integers in **[1, ceiling]** (round-6 security: `0` parses fine and would zero host capacity — an indefinite false-BLOCK); 0/negative/non-integer/beyond-ceiling all resolve to the code-default + WARN.

**Rendezvous schema evolution — versioned + tolerant from day one (round-4 integration; supersedes any blanket freeze reading).** The holders file, each holder row, witness records, and the tuning file all carry a schema-version field (`v: 1`) from the first ship, and every reader/writer follows the tolerant rule: (a) unknown top-level and per-row FIELDS are PRESERVED VERBATIM on any rewrite and never treated as corruption; (b) a row whose `state` is unrecognized is EXCLUDED from the cap count (the §1.1 fail-open direction), never signaled, and preserved on rewrite — **but preservation is BOUNDED, and exclusion is LOUD (round-5 adversarial + codex):** each excluded row ledgers a `schema-unknown` WARN (visible on the route histogram and in the preflight checks — a mixed-version host that is silently over-admitting shows up in the soak data), unknown-state rows still COUNT toward §2.4's poison-sanity ceiling (they occupy file bytes, so they must trip the bloat guard), and a row is preserved verbatim only while its pid is ALIVE or it is younger than `HOST_TEST_TTL_MAX_MS` — a dead-pid row older than the pinned ceiling is droppable by ANY reader (WARN-ledgered, never signaled; nothing legitimate outlives pid-death plus the 4h ceiling by this design's own rules, so unbounded preservation buys no correctness); (c) unknown tuning-file fields are ignored without invalidating the known ones. This is what makes mixed-version hosts — a multi-worktree dev host is GUARANTEED to run mixed checkouts for weeks after any change — safe by construction instead of by luck: an older actor can neither strip a newer actor's fields (breaking e.g. a tombstone's fingerprint) nor false-saturate on a state it doesn't know. The §2.6 pre-authorized post-soak refinement (waiter tickets / priority admission) MUST be additive under this rule, or bump `v` with a coordinated same-host cutover. §5 carries the round-trip test (a future-versioned row with unknown fields/state survives a prune pass intact and does not count toward admission).

**Migration wiring (Migration Parity):** type in `src/core/types.ts` under `intelligence`; default in `src/config/ConfigDefaults.ts` (the codebase's only-add-missing mechanism, matching the `spawnCap` treatment — no hand-written `migrateConfig` block); CLAUDE.md awareness in BOTH `src/scaffold/templates.ts` (new agents) AND `PostUpdateMigrator.migrateClaudeMd` (existing agents), including the "a rejected push may be contention" + outer-timeout guidance (§2.6); CapabilityIndex entry (§2.7). **Audience split:** the chokepoint files (5 vitest configs, the globalSetup, the core, the new semaphore module) are instar-repo SOURCE — they reach the incident population (builder/worktree hosts) by `git` pull into a checkout, NOT via `PostUpdateMigrator`; the route + config knob + CLAUDE.md awareness ship to deployed agent servers via the update/migrate path.

**Coverage honesty during rollout:** a worktree on a branch that predates this change has NO globalSetup — its suites run unbounded and invisible to the holders file until the branch rebases onto a base containing the change. On a dev host with many live worktrees this is the common case for a while; merge-to-main + normal rebase cadence is the closure path. The soak review treats old-branch runs as UNCOVERED, never as evidence about the cap.

### 2.10 The wait must not read as a hang (closes the very cascade this spec targets)

An agent `/build` session blocked in globalSetup for the wait budget shows a tmux frame with a once-a-minute line and no spinner — a plausible false-positive for the `ActiveWorkSilenceSentinel` (30-min threshold) and load-stall watchers whose false kills MOTIVATED this spec. Before shipping, the wait's frame signature is validated against the live silence/load-stall predicates; the printed wait line carries an active-work indicator those predicates recognize, OR the waiting run registers as known-blocked so no watcher kills it. The builder picks whichever mechanism the validation proves out, PREFERRING the active-work indicator (zero new coupling); the known-blocked registration is the fallback. Otherwise the fix re-triggers the exact layer-3 kill it targets (adversarial F10).

### 2.11 Dry-run semantics (the ratified soak posture, made precise)

While the resolved posture is dry-run (§2.9), the chokepoint performs **FULL bookkeeping with ZERO enforcement side-effects**:

- Acquire/release runs for real: holders are written, counted, and released; dead-pid reclaim and quarantine run normally (pure bookkeeping, no signals). This is load-bearing for the soak — if dry-run skipped bookkeeping, the holders file would always be empty and every would-block verdict vacuous.
- A run that WOULD block instead logs `would-block` (with the live holder set) to the ledger and **admits**.
- The TTL reclaim frees the stale slot and logs `stale-holder-reclaimed` in BOTH postures — because the DEFAULT reclaim is capacity-only and never signals anyway (§2.4), the dry-run/enforcing distinction for the TTL path is now narrow: it governs only whether a `would-block` ADMITS vs BLOCKS. Process-signaling is doubly gated — it requires BOTH enforcing posture AND the separate `INSTAR_HOST_TEST_TTL_SIGNAL=1` arm — and that arm is never turned on during the soak, so the soak validates TTL classification (via `stale-holder-reclaimed` frequency and any `reclaim-mismatch`) with zero risk of a wrong kill.
- **The config-eval clamps are dry-run-honest too (round-5 lessons-aware MEDIUM):** in dry-run the targeted-class maxWorkers ≤ 4 clamp (§2.3) and the unconditional nested ≤ 4 clamp (§2.5) do NOT reshape the real pool — the run executes with its natural worker count, and the chokepoint ledgers `would-clamp` (targeted and nested variants, carrying the natural resolved pool bound) instead. Lane classification still computes the WOULD-state (targeted argv + would-clamp) so `acquire` events carry meaningful run-class soak data (`wouldClamp: true`). "Log-only" therefore means genuinely zero behavior change on the incident population in Stage 1 dry-run; the clamp MACHINERY is validated pre-flip by the §5 test-time clamp validation plus the §4(g) would-clamp review. **In the OPTIONAL Stage-2 clamp-active sub-stage (tuning `clampActive:true`, §4), the clamps become REAL — the targeted/nested pool IS reshaped to ≤4 — while BLOCKING stays off (a `would-block` still admits); this gathers real clamp-impact soak evidence with no wedge risk, because a clamp can only reduce a run's worker count, never throw (round-9 codex).** A `would-block` blocks only in Stage-3 enforcing.
- All ledger kinds flow in both postures, each stamped with the resolved posture AND resolved cap.

## 3. What this is NOT

- NOT a change to any test's content or vitest version (the §2.3 config-time maxWorkers clamp on targeted-lane runs is the sole pool-settings touch).
- NOT a guarantee that the bound holds through a persistent lock wedge: fail-open admits multiple roots while a lock stays wedged (§2.4), so the bound is temporarily LOST in the safe direction until the 10s age-reclaim fires. Witness records make those admissions auditable (visible in `admittedOpen`), not re-counted. This is the deliberate trade — a wedged lock must never wedge the dev loop.
- NOT an absolute "targeted runs cannot fork-storm": the per-run maxWorkers≤4 clamp bounds ONE run; the aggregate is bounded by the targeted LANE's atomic cap `INSTAR_HOST_TEST_TARGETED_MAX` (§2.3), so worst-case targeted-lane load is ~`TARGETED_MAX × 4` workers, not unlimited. The arithmetic (N slots × 4) is stated so the lane cap's value can be sized against core count.
- NOT a scheduler with priorities/preemption. Admission is barging (whoever wins the lock when a slot frees), bounded by the wait ceiling — **the spec makes NO FIFO fairness claim** (the holder-set primitive has no arrival ordering). Push-class starvation under barging is a named soak-review item with a pre-authorized refinement path (§2.6). A monotonic arrival ticket is a possible future refinement, explicitly out of scope.
- NOT cross-machine: holders file host-local by the `df -P` fail-closed contract. Multi-machine posture: **machine-local BY DESIGN** — a test run consumes THIS host's cores; the route needs no `?scope=pool` (nothing to aggregate). The flip file, df marker, ledger, and witness records are machine-local for the same reason. "Host-wide" means **across actors sharing one OS-user home** (`os.homedir()`); a multi-OS-user host (e.g. an operator's `git push` as `justin` vs an agent's `pnpm test` as `justin_instar_1`) is not bounded across users — stated, with single-user dev host as the covered case. The test cap and the LLM spawn cap are **additive, not jointly bounded** — and the combined worst case is stated as a BUDGET TABLE (round-4 codex) so the sum is a reviewed number, not an implication. Worst legal concurrent load at defaults on a 16-core host:

  | Term | Source | Worst case |
  |---|---|---|
  | Suite lane | cap 1 × full pool | ~16 workers |
  | Targeted lane | `TARGETED_MAX` 6 × ≤4 workers | ~24 workers |
  | Nested shelter (ANY lane) | every nested child clamped ≤4 workers, no per-slot count (§2.5, round 8) — no full-pool shelter under ANY slot | ~4 × (nested children) |
  | LLM spawn cap | 8 subprocesses (mostly IO-bound) | ~8 procs |

  Steady-state worst case WITHOUT nesting ≈ **48 worker threads + 8 procs** (16 suite + 24 targeted + 8 spawn). Nesting adds a uniform **≤4 workers per nested child** — no longer a full-pool amplifier under any slot (the round-8 §2.5 simplification: the old design's nested terms were ~32 + ~48 workers of full-pool shelter; those are gone). A pathological test spawning many nested children still runs them, but each at ≤4 workers, fully `nested-skip`-attributed — the count is bounded by what the tests actually spawn, not by a budget the design must reserve. This is deliberately oversubscribed on 16 cores: the goal is preventing the UNBOUNDED storm (29 roots ≈ ~300 workers + watchdog kill cascade), not fitting inside core count; the load is transient, self-limiting, ledger-attributed, and every lane term has a named trim lever (`INSTAR_HOST_TEST_MAX`, `TARGETED_MAX`, the spawn cap) sized per-host via the tuning file. *Worker-token accounting (a weighted semaphore where a suite costs ~cores tokens and a targeted run ≤4 — round-6 codex) was considered and rejected:* token weights would rest on each root's DECLARED pool size, reopening as a declaration problem the exact bypass class §2.3 closes by OBSERVATION, and it abandons byte-compatibility with the proven spawn-cap core; the lane model plus per-run clamps plus this reviewed table is the CPU-pressure answer, with the token model available as a future refinement if soak data contradicts the table.
- NOT a gate on WHAT code may run: a deterministic resource bound (counting, no content judgment), same class as the shipped spawn cap — Signal-vs-Authority compliant (no brittle blocking authority over behavior).
- NOT zero-friction at cap=1: full suites serialize, and on a busy multi-agent host that is a real, accepted throughput cost (the operator directive priced it deliberately). The ledger's wait distributions make the cost measurable; `INSTAR_HOST_TEST_MAX=2` is the named relief valve.

## 6. Rollback & failure table

**Rollback:** `INSTAR_HOST_TEST_SEMAPHORE=off` (env, immediate, no release — the sole chokepoint switch; ledger-visible, never silent). Posture rollback from "enforcing wedged something": clear `enforcing` in the tuning file (or `INSTAR_HOST_TEST_ENFORCE=0` per-process) — back to dry-run, not off. The TTL signal arm is off by default and independently clearable (`INSTAR_HOST_TEST_TTL_SIGNAL` / tuning file). Full rollback = revert the PR. **On-disk artifacts, all inert and safe to delete after revert:** the holders file, the lock file (`host-test-runner-holders.lock`), the witness dir (`host-test-runner-witness/`, including storm-slot files), the df marker (`host-test-runner-dflocal.json`), the tuning file (`host-test-runner-tuning.json`), the tuning-hash baseline marker (`host-test-runner-tuning-baseline.json`), the event ledger (`host-test-runner-events.jsonl`) and its rotated segments, and up to 5 quarantined corrupt files each for holders and tuning.
