## Problem

Four built-in gate scripts call the local Instar HTTP API (`http://127.0.0.1:4041`)
without an Authorization header. When `authToken` is configured (the default after
`instar init`), `authMiddleware` (src/server/middleware.ts) returns `401` for any
non-allowlisted path. The gate shells then pipe the 401 response body into
`python3 -c 'import sys,json; d=json.load(sys.stdin); ...'`, which raises a
`json.decoder.JSONDecodeError`. `execFileAsync` rejects, `runGateAsync` treats it
as a gate failure, and the job is skipped — every single run, silently, forever.

Affected gates:
- `evolution-proposal-evaluate` — `/evolution/proposals?status=proposed`
- `evolution-proposal-implement` — `/evolution/proposals?status=approved`
- `evolution-overdue-check` — `/evolution/actions/overdue`
- `insight-harvest` — `/evolution/learnings?applied=false`

In the field this manifested as three evolution jobs marked `disabled:false` but
never firing actionable work, plus insight-harvest reporting zero harvests despite
learnings accumulating. No error surface because `authMiddleware` returns 401 as
valid JSON *sometimes* and as text *other times*, and either way the gate's pipe
to `python3` does not print anything — the crash goes to stderr and is swallowed
by the 10-second `timeout` branch in `runGateAsync`.

## Fix

Inject the configured auth token into the gate shell environment as
`$INSTAR_AUTH_TOKEN`, and update the four gates to send
`Authorization: Bearer $INSTAR_AUTH_TOKEN` with their curl calls. When
`authToken` is unset (public-mode deployments), the env var is absent and gates
still work against an unauthenticated server.

Single point of injection, in `JobScheduler.runGateAsync()`:

```typescript
const gateEnv = this.config.authToken
  ? { ...process.env, INSTAR_AUTH_TOKEN: this.config.authToken }
  : process.env;
```

Config plumbing: `JobSchedulerConfig.authToken?: string` (new optional field);
`Config.ts` forwards `fileConfig.authToken` into the scheduler config.

## Scope

- `src/core/types.ts` — add `authToken?: string` to `JobSchedulerConfig`
- `src/core/Config.ts` — forward `fileConfig.authToken` into scheduler config
- `src/scheduler/JobScheduler.ts` — `runGateAsync` builds `gateEnv` per call
- `src/commands/init.ts` — 4 built-in gate scripts add bearer header
- `src/data/builtin-manifest.json` — regenerated by build (manifest-of-builtins)
- `tests/unit/JobScheduler.test.ts` — 2 new tests covering both branches

Out of scope: other gates that call non-API commands (semantic-curation,
molthunt, daily-digest); those don't touch authenticated endpoints.

## Verification

- `npm run build` passes clean (tsc + manifest regeneration; 186 entries).
- `npm test -- JobScheduler.test.ts job-scheduler-edge.test.ts` passes (47 tests)
  including 2 new tests: one asserts gate shell sees `token=<configured-value>`,
  other asserts gate shell sees empty `token=` when `authToken` is unset.
- Semantic: the scheduler change is additive and preserves existing behavior for
  configs without `authToken`, so existing tests continue to pass unchanged.

## Compatibility

No breaking change. Existing `instar.json` files without `authToken` continue to
work (gates see no env var, curl sends no header, unauthenticated server accepts).
Existing `instar.json` files WITH `authToken` — the common case post-0.28.x —
begin routing gate traffic with proper auth, unblocking the four affected jobs
silently and immediately on next scheduler tick.
