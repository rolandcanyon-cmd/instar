# Cartographer Sweep Event-Loop Safety (fix instar#1069)

## Problem statement

Enabling the doc-freshness sweep on echo's real tree (366,757 nodes; `index.json` is **67MB** on disk) put the AgentServer into a supervisor kill-loop: two consecutive server deaths, each ~10–15 min after boot, exactly when the sweep's first tick fired. The SleepWakeDetector logged repeated ~33–38s event-loop starvation bursts; `/health` never answered; the lifeline supervisor (correctly, per its contract) killed the server after **6 consecutive failed checks** (the supervisor's `/internal/telegram-forward` probe cadence × 6 — stated numerically here because the fix's safety bound must be sized against it, see Slice 1). Zero authoring calls ever completed — `feature-metrics.db` has no `CartographerSweep` rows, so the process died in the *detect* phase, before any LLM call.

The original diagnosis named one starver (the sweep's `staleNodes()` walk). Convergence review found that to be **incomplete**: there are **six distinct main-thread, O(nodeCount) or 67MB operations** that can each freeze the event loop on this tree, and a fix that closes only the first leaves the kill-loop reachable through a side door. The complete inventory (the invariant below must close *all* of them, not just #1):

1. **Sweep detect — `CartographerSweepEngine.runPass()` → `tree.staleNodes()`** (`src/core/CartographerTree.ts:546`). A 366k-iteration compare loop that materializes the full non-fresh set. On `authoredCount: 0` every node is `never-authored`, so it materializes all 366,757 entries — yet the engine needs only `maxNodesPerPass` (25) candidates plus counts. (The originally-named starver.)
2. **Sweep revalidation — `runPass()` → `revalidateSample()` → `tree.loadIndex()`** (`CartographerSweepEngine.ts:457`). A synchronous `fs.readFileSync` + `JSON.parse` of the 67MB index, on the main thread, on the no-candidates path **and** the normal-completion path of every pass (`runPass` lines ~196 and ~296). Even after detect moves off-thread, this re-freezes the loop every tick.
3. **Request-path lazy scaffold — `GET /cartographer/{tree,stale,node,health,...}`** call `if (!loadIndex()) scaffold()` synchronously (`src/server/routes.ts:4220, 4236, 4244, 4251, 4300, 4369`). `scaffold()` is the 366k-node structural walk + 366k atomic node-file writes; the problem statement observed it "hangs HTTP routes for ~2 min" during the incident. The **first** hit to any of these routes on a missing/corrupt index runs that walk on the request thread → same starvation, same watchdog kill. `loadIndex()` returning null on a parse error routes into the same `scaffold()` path.
4. **Health route freshness — `GET /cartographer/health` → `tree.freshnessHealth({ graceMs })`** (`routes.ts:4261`). Independently catastrophic: it runs `currentOids()` (full `git ls-tree`) **and** one `readNodeFile` per authorable node (`CartographerTree.ts:646`) = up to 366k synchronous node-file reads on the request thread — heavier than `health()`, which is the only call the first draft named.
5. **Buffered `git ls-tree` — `tree.currentOids()`** (`CartographerTree.ts:185`) runs `SafeGitExecutor.readSync(['ls-tree','-r','-t','-z','HEAD'])`, fully buffered into one string then split into a 366k-entry Map. `SafeGitExecutor.readSync`'s default `maxBuffer` is **10MB** (`SafeGitExecutor.ts:1103,1196`); a 366k-entry `ls-tree` output is ~30–40MB, so this call **throws `ENOBUFS` today** unless a larger buffer is passed — and a throw here that's swallowed makes every node read `path-gone` (a correctness landmine, not just a perf one). This cost is inherited by both detect (1) and freshnessHealth (4).
6. **Author-path index write — `setSummary` / `patchNodeMeta`** (`CartographerTree.ts:443-447, 494-497`) each `loadIndex()` (67MB parse) + `atomicWrite` (67MB serialize) **per node**; authoring up to `maxNodesPerPass` nodes/pass = up to 25× (67MB parse+serialize) on the main thread. Latent only because authoring never ran at scale on the incident tree (detect died first) — but it freezes the loop the moment the sweep authors on a large tree. Closed by the index-write discipline in Slice 1.

**Secondary finding (routing trap):** `cartographer.freshnessSweep.framework` ("codex-cli") is *decorative*. `probeRouting()` resolves via `router.for('CartographerSweep')` (overrides → categories[job] → default) and never reads the config field. With only `categories.sentinel` routed, the sweep resolved to default claude-code and refused every pass — which *masked* starver (1) (the refusal returned before `staleNodes()` ran, so the first enable looked stable). A config field that looks authoritative but is ignored is a trap; the incident workaround on echo was a manual `sessions.componentFrameworks.overrides.CartographerSweep = "codex-cli"`.

**The invariant this spec exists to establish:** *No background freshness job and no `/cartographer/*` request path may run a synchronous, O(nodeCount)-or-67MB operation on the server's main event loop — no matter how large the tree, and no matter which of the six paths above is exercised.* Closing one path is not the fix; closing the class is.

## Proposed design

### Slice 1 — Detect runs off the event loop (worker thread), bounded in time AND memory

A dedicated short-lived `worker_threads` worker performs the entire detect phase off the main loop. Because this is the **first worker in the codebase** (no precedent), the structure is split deliberately for testability and safety:

- **`src/core/cartographerDetect.ts`** — a plain, importable **pure module**: all detect logic (read+parse index, derive per-node staleness, order candidates, compute aggregate counts + the freshness-health aggregate). No `worker_threads` imports. This is what unit tests exercise in-process (resolves Integration Finding 1 — the test suite runs TS source via vitest and a `worker_threads` worker cannot load a `.ts` file, so the *logic* must be testable without spawning a worker).
- **`src/core/cartographerDetect.worker.ts`** — a trivial entrypoint: `parentPort` plumbing only; reads `workerData`, calls the pure module, posts the bounded result, exits. Contains no logic to drift.

**What the worker computes and returns (bounded payload ONLY):**
- Up to `maxNodesPerPass × deferHeadroomFactor` **ordered** candidates (order-then-bound — see below), plus
- Aggregate counts `{ nodeCount, authoredCount, neverAuthored, stale, pathGone, generatedAt, headSha }`, plus
- The **full `freshnessHealth` aggregate** (fresh ratio, un-authored, quarantined, author-failed, grace-clock counts) so the `/health` route never has to call `freshnessHealth()` on the request thread (closes starver 4), plus
- The **revalidation sample** (`revalidateSamplePerPass` oldest-by-`summaryUpdatedAt` fresh node paths) so `runPass` never calls `tree.loadIndex()` on the main thread (closes starver 2).

The full materialized stale set never crosses the worker boundary and never exists on the main thread.

**Ordering is a single bounded pass keeping a top-N heap — never a full sort of a full materialized array (Scalability Finding 4, Adversarial Finding 5; R2 Security Finding 1).** The correct top-N by the ordering rules (deepest-first / dir-defer / anti-starvation) is produced by streaming every candidate through ONE pass that maintains a bounded min-heap of size `maxNodesPerPass × deferHeadroomFactor` — the full non-fresh set is *visited* (so ordering is correct) but never *materialized* into a full array and never fully sorted. Peak memory for the candidate structure is O(N), not O(nodeCount). (Bound-then-order would silently break the invariant; full-materialize-then-sort would reintroduce the memory blowup — the heap pass avoids both.) Ordering derives the parent→child adjacency it needs **from the path strings and the already-parsed index** (a path's parent is computable; the index carries `kind`/`hasChildren`) — it must NOT read per-node files (that would be a fresh 366k synchronous-read starver). The detect logic reads **zero** node files. Test asserts node-file reads during detect == 0 AND peak candidate-structure size ≤ N (not nodeCount).

**The anti-starvation counter must move into the index (R3 Scalability Finding 1 — the load-bearing schema gap).** The dir-defer/anti-starvation front-bias reads `staleSincePass` (`CartographerSweepEngine.ts:347`), an *accumulated* defer counter that today lives ONLY in per-node files (`CartographerNode.staleSincePass`, written by `patchNodeMeta` at `CartographerTree.ts:492`) and is NOT on `CartographerIndexEntry`. It is pure accumulated history, so it cannot be "recomputed from the index" — the zero-node-file-read invariant is only satisfiable if the counter is *in* the parsed index. Therefore: `staleSincePass` is added as an **optional** field on `CartographerIndexEntry` (`staleSincePass?: number`); ALL readers coalesce missing → `0` (`entry.staleSincePass ?? 0`), so an existing on-disk index parses and works **unchanged** — the missing counter simply reads as 0 and anti-starvation re-warms over a few passes (R4 Lessons Finding A: `loadIndex()` performs NO `schemaVersion` check today and MUST NOT gain one that triggers a rewrite — the optional-field coalesce IS the migration; the `schemaVersion` bump is a documentation/observability signal only, never a reset gate). The increment is applied off-thread under the index-write discipline below — the existing main-thread defer-write-back loop at `CartographerSweepEngine.ts:285-290` is **deleted**, not rewritten (R4 Scalability Finding 1: that loop iterates an unbounded `ordered.deferred` set <!-- tracked: instar-1073 --> — ~O(stale-dir count) on a cold tree — and after ordering moves into the worker the main thread no longer even has that set). The Slice 4 golden-ordering test includes a fixture where a starved dir's front-bias actually fires, proving the index-sourced counter drives ordering with zero node-file reads.

**Index-write discipline — the sixth starver (R4 Scalability Finding 1 + Lessons Finding B; the author path).** `setSummary` (`CartographerTree.ts:443-447`) and `patchNodeMeta` (`494-497`) each do `loadIndex()` (67MB parse) + `atomicWrite` (67MB serialize) **per node**. Authoring up to `maxNodesPerPass` nodes/pass = up to 25 × (67MB parse + serialize) on the main thread — a sixth main-thread starver, latent only because authoring never ran at scale on the incident tree (detect died first). The invariant ("no synchronous O(nodeCount)/67MB op on the main event loop") requires closing it. Discipline:
- **The 67MB index is parsed and serialized ONLY off the main thread** (inside the detect/write worker / pure module), never via per-node `loadIndex()+atomicWrite` on the sweep path.
- **Two off-thread index writes per pass, both bounded across the worker boundary by deltas, not full sets:** (a) the detect worker, having parsed the index, applies the pass's defer-counter increments to its in-memory copy and writes the index once **before exiting** (the deferred set <!-- tracked: instar-1073 --> never crosses the boundary); (b) after authoring (which awaits the LLM — already async, never blocks the loop), a single off-thread write-worker pass receives the ≤`maxNodesPerPass` authored-summary deltas, re-reads the current index off-thread, applies them, and writes once. Per-node *node files* (small) are still written on the main thread, bounded to `maxNodesPerPass` — fine.
- **Single-writer (resolves the two-writer race):** the engine is the only index writer on the sweep path; lease-gating + engine single-flight guarantee one pass at a time, so write (a) (defer increments, end of detect) and write (b) (summary deltas, end of authoring) are strictly sequential within a pass and cannot clobber each other. Enforcement is the Slice 4 integration assertion (a pass authoring `maxNodesPerPass` summaries performs exactly the two bounded off-thread writes, never a per-node main-thread `loadIndex()`) — note the lint alone can't catch this, since `setSummary`/`patchNodeMeta`'s `loadIndex()` lives in the allowlisted `CartographerTree.ts`; the refactor moving those off the engine main path is what the integration test compels. The `detectInWorker:false` rollback runs writes (a)/(b) synchronously on the main thread — the documented time-vs-isolation tradeoff of the escape hatch.

**All of the above lives in the `cartographerDetect` pure module — shared verbatim by the worker AND the synchronous rollback path (Slice 5).** The explicit `git ls-tree` `maxBuffer` (≥64MB), the pre-parse byte guard, the heap-pass ordering, and the refusal taxonomy are properties of the *pure module*, not of the worker wrapper — so the `detectInWorker:false` rollback runs the exact same bounded logic on the main thread (bounded, never `staleNodes()`), and no large-tree `currentOids()` callsite inherits the 10MB default (R2 Security Finding 5).

**Memory is bounded, not just relocated (Scalability Findings 1+2, Lessons Finding 3).** Moving the parse off-loop fixes starvation but not memory:
- The worker is spawned with `resourceLimits: { maxOldGenerationSizeMb }` (config `freshnessSweep.detectWorkerHeapMb`). **The heap cap and the byte guard must be co-sized so the *intended* tree succeeds (R2 Security Finding 3):** `JSON.parse` of an N-byte index produces an in-memory graph several× N, so `detectWorkerHeapMb` default must be ≥ `maxIndexBytes × (parse-expansion factor ≈ 4–6×)` + the ls-tree Map + the candidate-heap working set + headroom. A heap default sized off the *file* size rather than the *parsed-graph* size would refuse the real 67MB tree on every pass (safe direction, but the feature would be dead-but-honest on the exact tree it exists for). A Slice 4 test asserts detect **succeeds** (not merely refuses cleanly) on the ≥50k synthetic fixture under the default heap cap, so a too-tight default is caught. An over-large parse beyond the cap triggers a clean worker `error`/`exit` → named refusal → breaker, **never** a process OOM that takes the host down a different way.
- A **pre-parse byte guard**: before `JSON.parse`, the worker stats `index.json`; above a configured ceiling (`freshnessSweep.maxIndexBytes`, default generous) it refuses with `detect-index-too-large` rather than attempting the parse. This is the real bound on the parse, because **`worker.terminate()` cannot interrupt a synchronous `JSON.parse` already in progress** (V8 won't yield) — the spec no longer claims the timeout bounds the parse; the timeout bounds the `git` spawn and ordering, the heap-limit + byte-guard bound the parse.
- `git ls-tree` inside the worker is run with an explicit `maxBuffer` sized for the largest expected tree (≥ 64MB, matching the CI ratchet script), and a git-read failure is treated as a `refused` (`detect-git-error`) — **never** as "every node path-gone" (closes the correctness landmine in starver 5). Streaming `ls-tree` via `spawn` + incremental NUL-delimited parse (peak buffer O(1)) is preferred; if deferred, the explicit maxBuffer + refusal-on-overflow is the floor and the streaming upgrade is filed alongside the index-format follow-up. <!-- tracked: instar-1073 -->

**Detect watchdog.** The await is bounded by `freshnessSweep.detectTimeoutMs` (default 120000). On timeout: `await worker.terminate()` (awaited, so the worker's child `git` process is reaped, not orphaned — Adversarial Finding 7; the worker spawns `git` with a handle it kills on teardown), the pass aborts as a **refusal** (`reason: 'detect-timeout'`, `refused: true`), it feeds the existing breaker, and one bounded degradation is emitted via `reportDegradation`.

**Every failure path sets `refused: true` (Adversarial Finding 9).** `detect-timeout`, `detect-worker-start-failure`, `detect-index-too-large`, `detect-git-error` must ALL set `refused: true` + a `refusalReason`, so they flow through `classifyProgress`'s `r.refused` branch and feed the breaker — otherwise a result with `candidateCount: 0, refused: false` is mistaken for "genuinely nothing to do," resets the breaker accounting, and the sweep silently does nothing forever. Unit test asserts each refusal increments `zeroProgressTicks`, not `consecutiveZeroCandidate`.

**Worker env is an allowlist, not the full process env (Security Finding 1).** The worker is spawned with an explicit minimal `env` (`PATH`, `HOME`, the `GIT_*` it needs) — never the parent `process.env`, which holds the Telegram token, Anthropic keys, Bearer authToken, and dashboard PIN material that the detect worker has no need for. This matches the existing Codex-spawn env-allowlist discipline. Unit test asserts a sentinel secret var is absent in the spawned worker's visible env.

**Worker boundary carries paths/oids/counts ONLY — never leaf file content (Security Finding 5).** Secret-bearing-path exclusion and credential-material tripwires stay on the main-thread author path where they are; the worker never reads blob content. Test: the worker module does not import `committedContent`/credential helpers. Secret-bearing path *names* are filtered out of the snapshot stale sample (Security Finding 2/5) so they aren't published via the Files tab.

**Single-flight at the ENGINE level, not just the poller (Scalability Finding 7).** Detect is guarded by a shared in-flight promise on the engine so any caller — the poller, or a future spec #3 conformance audit that reuses the same engine — coalesces onto ONE worker. Hard invariant + test: at most one detect worker alive at a time, bounding peak host memory to one worker's working set regardless of caller count.

**No silent main-thread fallback.** If the worker fails to start (bundling regression, resource exhaustion), the pass refuses with a named reason. It must never quietly fall back to the synchronous full walk — that would resurrect the kill-loop exactly when the system is least healthy. Grep-assert: no `catch → inline staleNodes()/loadIndex()` branch anywhere on the detect path; any deliberate skip uses the existing `@silent-fallback-ok` annotation.

**Worker path resolution + build (Security Finding 6, Integration Finding 1).** The worker entrypoint is resolved via `new URL('./cartographerDetect.worker.js', import.meta.url)` so it tracks `dist` in prod and is included in the published package (`files: ["dist"]` already covers it; `sign-instar-lockfile.mjs` globs dist — verified, no allowlist edit needed). Exactly **one** integration test runs the worker from **built dist** (the rest unit-test the pure module in-process) so the prod resolution is actually proven, not just asserted.

### Slice 2 — Every `/cartographer/*` request path serves a snapshot or a cheap state, never a walk

Each successful detect persists its aggregate counts + freshness aggregate + a bounded stale sample to a small snapshot file, atomic tmp+rename via `SafeFsExecutor.atomicWriteFileSync` (mode `0o600`; per-host-unique tmp suffix via `crypto.randomBytes` so a shared `.instar` across machines can't collide — Security Finding 3). Sample + counts are one atomic JSON object so they're always mutually consistent (Adversarial Finding 8). The stale sample is capped (`freshnessSweep.snapshotSampleMax`, default 500; target file < 256KB — Scalability Finding 6).

Then, the request paths:

- **`GET /cartographer/health`** serves the **entire** freshness block (both `health()` *and* the former `freshnessHealth()` fields) from the snapshot — closing starver 4. Backward-compat is **additive**: the existing field names/semantics (`nodeCount`, `staleCount`, `authoredCount`, `freshness.*`) are preserved, sourced from the snapshot (Integration Finding 3). New fields: `snapshot: 'present'|'absent'|'detect-failing'`, `generatedAt`, `headSha`, plus a server-computed **`snapshotStale: boolean`** + `ageMs` + `headMoved: boolean` (computed from one cheap `git rev-parse --short HEAD` — never a walk) so consumers don't have to interpret a timestamp (Adversarial Finding 3, Lessons Finding 6). `lastDetectStatus` (`ok|timeout|worker-start-failed|index-too-large|git-error`) + `lastDetectAt` surface detect health so a broken detect is visible on the route, distinct from `snapshot:'absent'` = never-run (Adversarial Finding 1, Integration Finding 8).
- **`GET /cartographer/stale`** serves from the snapshot's bounded sample with `truncated: true` + the `total`. When no snapshot: `{ count: 0, nodes: [], snapshot: 'absent' }` — never a 503, never a missing field.
- **`GET /cartographer/tree` and `/cartographer/node` are brought into compliance too (R2 Scalability Finding 2 — these are documented agent endpoints, not deletable).** Both call `loadIndex()` today (the 67MB sync parse, plus full-index serialization for `/tree`). Post-rewrite: `/cartographer/tree?format=compact` serves `nodeCount` + the tree index summary **from the snapshot** (no parse). The full `/cartographer/tree` (whole index) and single-node `/cartographer/node` are gated behind a node-count ceiling (`freshnessSweep.maxRequestNodes`, default generous-but-bounded): under the ceiling they parse + serve as today (small trees — fine); above it they return `indexState: 'too-large-for-request'` with the snapshot summary + guidance to use the navigator (`/cartographer/navigate`, which is already bounded) instead of the whole tree. No `/cartographer/*` route runs an unbounded parse/serialize on the request thread. The backward-compat integration test (below) covers `/tree` + `/node` in both the under-ceiling and over-ceiling branches.
- **The lazy `scaffold()`/`loadIndex()` preamble is removed from all `/cartographer/*` read routes (closes starver 3).** When no index exists, the routes return the `snapshot: 'absent'` / `indexState: 'not-built'` shape immediately (cheap) — they must NOT call `loadIndex()` (67MB sync parse) or `scaffold()` (366k walk) on the request thread. Index (re)build is triggered off the request path (see Slice 5 boot handling).
- **Shared snapshot state vocabulary across BOTH routes (R2 Security Finding 4).** `/cartographer/health` and `/cartographer/stale` carry the *identical* `snapshot: 'present'|'absent'|'detect-failing'` enum plus `snapshotStale`/`lastDetectStatus`/`lastDetectAt` provenance — `/stale` serves its last-good bounded sample with `snapshot: 'detect-failing'` + `snapshotStale: true` when a prior snapshot exists and detect is currently erroring, and `{ count: 0, nodes: [], snapshot: 'absent' }` when none ever ran. The two routes never disagree on state.
- `CartographerTree.staleNodes()`, `loadIndex()`, `freshnessHealth()`, and `scaffold()` remain for small trees, the pure module, and tests, but each gains a doc-comment contract: **O(nodeCount)/67MB synchronous — forbidden on the server hot path** (enforced by the Slice 5 lint).

### Slice 3 — `freshnessSweep.framework` is honored, observably, without silently outranking `categories.job`

The config field feeds routing instead of being ignored. Precedence: explicit `sessions.componentFrameworks.overrides.CartographerSweep` → `cartographer.freshnessSweep.framework` → `categories[job]` → default. The existing probe semantics are unchanged (still refuses to author on the default/Claude framework unless `allowClaudeFallback`).

- **The field applies only when explicitly set** (not written by scaffold/migration as a default) — so an existing operator who relies on `categories.job` is not silently overridden on update (Adversarial Finding 6, Migration Parity). Confirmed: `ConfigDefaults` carries `framework: 'codex-cli'` today as a *value* but it was never read; this slice makes that value live, so the migration must verify it does not override an operator's pre-existing `categories.job` — if a conflict is possible, the field yields to an explicitly-configured `categories.job`, OR the boot log makes the precedence loud (below).
- **Effective routing is logged at boot**: one line stating the resolved `CartographerSweep` framework and which config source won ("sweep routing: codex-cli (source: freshnessSweep.framework)"), so the precedence is observable, not silent.
- **The Claude-floor invariant gets a test**: `freshnessSweep.framework = <defaultFramework>` with `allowClaudeFallback: false` → pass refuses with `routing-refused` (the field can never become a backdoor around the never-spend-Claude-quota floor). A non-existent/typo framework value degrades to the existing "framework unavailable → refuse" path, never to default-Claude.

### Slice 4 — The invariants get tests, not hopes

- **Unit (pure module):** detect on a generated large-tree fixture (≥50k nodes, synthetic index) returns bounded payloads; **golden-output ordering test** — fixture → committed expected ordered candidate list, so a future edit to ordering that changes behavior fails against the frozen golden (NOT a module-against-itself tautology — Adversarial Finding 5, Lessons Finding 4); node-file-read count during detect == 0; pre-parse byte-guard refuses `detect-index-too-large`; git-read failure refuses `detect-git-error` (not zero-candidates); env-allowlist excludes a sentinel secret; each refusal type increments `zeroProgressTicks`.
- **Integration:** (a) **event-loop-lag harness** — `setInterval` drift sampled on the main thread while a detect AND a full author-write pass run against the large fixture, asserting max lag < 250ms — extended to also hit `GET /cartographer/health`, `/cartographer/stale`, `/cartographer/tree`, `/cartographer/node` AND the off-request **boot-scaffold execution** under the large fixture, so a regression on ANY of the six starvers (sweep detect, the four request routes, the boot scaffold, OR the author-path index write) trips it, not just the sweep tick. A dedicated assertion: a pass authoring `maxNodesPerPass` summaries performs exactly the bounded off-thread index writes (one detect-phase, one author-phase), never a per-node main-thread `loadIndex()`. (b) the **timeout/terminate path** asserts main-thread occupancy stays under the stated watchdog window with margin, and no orphaned `git` process survives a forced-timeout pass. (c) **detect-succeeds-on-large-fixture** under the default heap cap (catches a too-tight `detectWorkerHeapMb`). (d) **`detectInWorker:false` rollback path** runs the pure module synchronously, materializes ≤ N candidates, and never calls `tree.staleNodes()`. (e) **worker-from-dist resolution** test — runs against the actual `pnpm build` / `tsc` dist output (the published package layout), NOT a transpile-on-the-fly runner, so it proves the real prod resolution (R2 Scalability Finding 1, Lessons Finding 3). **Pipeline wiring:** the integration + e2e vitest configs gain a `globalSetup` that runs `npm run build` (idempotent; skips if dist is current) before these tests, so a build actually precedes the dist-backed tests in CI — the current pipeline runs vitest on TS source with no preceding build, which would make these tests find no dist. The dist-backed tests fail-loud (not skip-silently) if dist is absent.
- **E2E:** feature-alive test extended — sweep enabled on the e2e fixture authors through the real worker path end-to-end; `/cartographer/health` and `/cartographer/stale` answer from the snapshot.

### Slice 5 — Structural guards: the lint, the boot path, migration, rollback

- **Lint (resolves Open Question 2 — Lessons Finding 8).** Ship a callsite lint (pattern of the existing `lint-no-unfunneled-topic-creation.js`) forbidding `staleNodes()`, `loadIndex()`, `freshnessHealth()`, and `scaffold()` calls. Because a callsite lint matches **files, not runtime threads** (R2 Lessons Finding 4), it is specified as a path-allowlist: **forbidden** in `src/server/routes.ts` and `src/core/CartographerSweepEngine.ts` (the main-thread surfaces); **permitted** in `src/core/cartographerDetect.ts` (the pure module that legitimately does the parse), `src/core/CartographerTree.ts` (the definitions), and `tests/`. "Enforced by review" is the willpower path the constitution rejects, and a request-path full walk is the exact regression that caused #1069. Because the rollback path lives in the allowlisted pure module (not in the engine), it does not trip the lint — and because it never calls `staleNodes()`, the lint and the rollback no longer contradict each other.
- **Boot-path scaffold (resolves Lessons Finding 1 — do not ship a recurrence-risking deferral unsigned; R2 Adversarial/Scalability/Lessons Finding 2/5).** <!-- tracked: instar-1073 --> The full `scaffold()` writer-safety rework stays deferred <!-- tracked: instar-1073 --> (filed: instar-1073, referenced from #1069), BUT a concretely-specified **interim bound ships in this PR**: `AgentServer.start()` enqueues a one-shot boot task (not inside any GET handler) that builds the index in chunks of `freshnessSweep.scaffoldChunkNodes` (default e.g. 500) node-writes per macrotask, yielding via `await new Promise(r => setImmediate(r))` between chunks, single-flighted. The success criterion is a **per-yield main-thread occupancy ceiling** (the Slice 4 lag harness asserts max lag < 250ms while the boot scaffold runs over the large fixture) — NOT a total-duration bound (a 120s contiguous freeze must never "pass"). P19 brakes: on timeout/failure the partial index is discarded (atomic tmp+rename means a half-written index is never read; routes serve `indexState: 'not-built'`), and the next boot/queued one-shot retries — boot-cadenced, no tight retry loop. This removes the live kill-trigger without the full rework, so no `principal-deferral-approval` is needed for an unguarded deferral. <!-- tracked: instar-1073 -->
- **Migration parity (Integration Finding 5, Lessons Finding 9).** New config fields (`detectTimeoutMs`, `detectWorkerHeapMb`, `maxIndexBytes`, `snapshotSampleMax`, `maxRequestNodes`, `scaffoldChunkNodes`) are added to `ConfigDefaults` freshnessSweep; the existing `applyDefaults` deep-merge backfills them into existing on-disk configs (no bespoke `migrateConfig` needed — the server already reads each field via `num(fsCfg.X, default)` with the same literal). `.gitignore` gains `.instar/cartographer/` via an idempotent migration so the snapshot (and the pre-existing un-ignored 67MB index) stop being committable (Security Finding 2, Integration Finding 4); the false "gitignored" claim in `cartographer-freshness.mjs`'s header is corrected. The CLAUDE.md template's Cartographer Doc-Freshness section gets an idempotent `migrateClaudeMd` update: `/health` + `/stale` now serve a snapshot (may be `absent` before first detect; carries `generatedAt`/`headSha`/`snapshotStale`), `/stale` is truncated with a `total`, and `freshnessSweep.framework` is the supported off-Claude routing knob (manual override no longer required) — without this, agents won't know the field works (Agent-Awareness Standard).
- **Multi-machine lease contract (Scalability Finding 8, Integration Finding 6).** Detect (now cheap on the event loop, but a real per-host memory spike) is gated behind `holdsLease()` — standby machines do NOT spawn a detect worker; they serve `/health` + `/stale` as `snapshot: 'absent'` honestly (the snapshot is per-host and per-Finding-4 never committed). The engine comment "standby machines do cheap detect for read-locality" is corrected (detect is not cheap, and is now lease-gated). This also kills the fleet-wide N× spawn the incident exhibited.
- **Rollback (Integration Finding 9; R2 Security/Lessons Finding 1 — this is the convergence trap, pinned).** The bounded-candidate fix (heap-pass ordering, never materialize the full set) is **separable** from off-threading and is itself the core correctness win. Config escape hatch `freshnessSweep.detectInWorker` (default `true`): if the worker misbehaves in prod, set it `false` to run **the exact same `cartographerDetect` pure module synchronously on the main thread** — byte-guard + bounded heap-pass intact. It is explicitly **NOT** a revert to `tree.staleNodes()` (that would reintroduce #1069 on the event loop and would itself trip the Slice 5 lint). The rollback is "safe" only because it runs the bounded module, not the legacy walk — and a Slice 4 test pins this (rollback path materializes ≤ N, never calls `staleNodes()`). On a pathologically large tree the synchronous bounded pass still occupies the main thread longer than the worker path would, so it is a *deliberate* time-vs-isolation tradeoff for an operator escape hatch, not a free lunch — documented as such. `freshnessSweep.enabled: false` remains the master kill-switch; the snapshot-serving routes degrade to `absent` (not 503) when the sweep is off.

## Foundations audited (one layer below the spec — counter-findings)

- **The poller breaker is signal-only, not authority** (Lessons Finding 10): `openBreaker()` only backs off cadence + emits a degradation + re-escalates; it never permanently disables the feature or blocks a user action, and recovers on the next successful tick. Detect-timeout refusals routing into it is Signal-vs-Authority compliant. *One caveat folded into Slice 2:* a detect that can NEVER succeed (genuinely too-large tree) leaves the breaker backed-off and the snapshot `absent`/stale indefinitely — the accepted safe failure mode, made honest by `lastDetectStatus` + `snapshotStale` on the route and operator-actionable re-escalation (raise `detectTimeoutMs`/`maxIndexBytes` or shard the index). Recovery is operator action, by design, and the route says so.
- **The `allowClaudeFallback` probe is a sound foundation** (Lessons Finding 11): deterministic, fail-closed, signal-only — the structural guarantee that the sweep never spends Anthropic quota. Preserved unchanged.

## Frontloaded Decisions

- **CI ratchet compatibility (was Open Q1):** RESOLVED — `scripts/cartographer-freshness.mjs` re-derives staleness directly from git and never reads the snapshot or the HTTP routes, so the snapshot/response-shape changes cannot affect it. No action needed.

Cheap-to-change-after: the value is a config field with a defensive literal default.
