## Problem statement

Instar routes every internal decision and generation through a *doorway* — a way of reaching a model: the Claude CLI, the Codex CLI, pi-cli, a paid Gemini/Google API key, OpenRouter, the Antigravity CLI. Which doorways exist, which are actually reachable *on this machine*, and what the **top model** behind each one is (with its exact model id) are facts that **rot silently**. The world moves — a new frontier model ships, a door's default model advances, a Homebrew formula gets deprecated, a CLI stops being installed — and nothing forces our knowledge to move with it.

**The crystallizing incident (2026-07-03).** Instar's cross-model reviewers and several routing pins were still pointing at `gemini-2.5-pro` long after the Gemini 3-class models shipped (`gemini-3.1-pro-preview` is the current usable frontier). The pin was stale for weeks. Nobody had re-checked, because nothing *made* anyone re-check. The `lint-model-registry-freshness.mjs` guard (PR #1359) can fail loud when a pin **ages out** or **drifts off a maintained allowlist** — but it is a *tripwire*, not an *engine*: it never re-probes the doorways, never enumerates the top models behind each, and never grows our knowledge. A human still has to manually go find the current frontier and update the allowlist. Between those manual passes, the map degrades invisibly until it degrades *output* (a reviewer silently running on a year-old model).

**The gap, stated precisely.** We have:
- A **human-readable routing doc** (`docs/LLM-ROUTING-REGISTRY.md`) — intentional defaults + a callsite inventory.
- A **machine-checkable freshness tripwire** (the manifest + the lint, #1359) — staleness + drift teeth.
- A **cadenced bench-refresh job pattern** (#1352) — an off-by-default, tier-1-supervised, review-only monthly job that diffs bench results against routing defaults and raises ONE attention item.

We are **missing** the standing process and durable structure that keeps the *doorway/model map itself* current and **growing**: a live re-probe of every doorway on a cadence, an enriched record of the top model(s) + exact ids + pricing behind each door, a way to *notice a brand-new doorway*, and a constitutional standard that says a stale map is a defect.

This spec closes that gap with three components, all of which **extend** (never duplicate) the artifacts above:

1. **Doorway/Model Knowledge Registry** — a durable, *growing* record of every doorway and the top model(s) behind it.
2. **Recurring doorway-scan job** — a cadenced, dark-by-default, cost-aware, fail-safe, secret-safe scan that live-probes every doorway (via a deterministic prober script), updates the registry, diffs against the previous scan, and surfaces ONLY changes.
3. **Constitutional standard** — "Keep the Doorway/Model Map Current": a stale doorway/model map is a defect, enforced structurally by the scan job + the freshness lint (Structure > Willpower).

## §1 — Component 1: The Doorway/Model Knowledge Registry

### §1.1 Two layers (canonical vs live), and why

A doorway's reachability is a **physical fact of one machine's disk**: `codex` is installed on machine A but not on this laptop; the vault key `metered_openrouter_bench` exists here but maybe not on the Mac Mini. So "what can I reach, and as what model?" has two honestly-different answers:

- **Canonical (shipped, reviewed) layer** — the enriched `scripts/model-registry-freshness.manifest.json`, committed to instar source. This is the *reviewed* map: the doors instar knows about, the top models an operator has *confirmed*, and the derived frontier set that gates the routing pins. It is the same on every machine (it is git-tracked source). It is updated **only** through instar-dev + operator confirmation — never auto-edited by a running job (enforced structurally, §2.7).
- **Live (per-agent, scanned) layer** — a new machine-local state file `.instar/state/doorway-scan.json`. This is *this agent's* durable, growing record of what its last live probe actually found on *this* machine: which doors answered, what model ids each enumerated, when. The scan job writes this every run; it is the "durable, growing record" the requirement asks for, and it is **machine-local BY DESIGN** (a door's liveness is a local fact — the identical posture of the Playwright Profile Registry: "a logged-in session lives in cookies on one disk").

The two layers meet at the **diff**: when the live scan finds something the canonical map doesn't know (a new door, a top model that moved, a pin fallen behind, a `flaggedStale` suspicion confirmed by a live probe), the scan raises a **maintainer diff** for the canonical map to be updated via instar-dev. The job never edits canonical source itself.

### §1.2 Canonical registry schema (enriched manifest)

The existing manifest keeps every current field except that the hand-maintained `frontierAllowlist` is **superseded by derivation** (§1.4). Each entry under `doors[<doorId>]` is enriched with the knowledge the requirement names. Illustrative (not exhaustive) shape:

```jsonc
{
  "$comment": "Doorway/Model Knowledge Registry — canonical/reviewed layer. Single human-edit surface.",
  "registrySchemaVersion": 2,
  "lastReviewedAt": "2026-07-03",           // TOOTH 1 (staleness) — unchanged
  "stalenessWindowDays": 45,                 // unchanged
  "enforcement": "report",                   // unchanged (report | strict)
  "doors": {
    "openrouter": {
      "name": "OpenRouter",
      "accessMethod": "direct-api",          // enum: cli | direct-api | subscription
      "status": "alive",                     // enum: alive | referenced-not-installed | deprecated | dead | new-unverified
      "probe": {                             // how the deterministic prober re-verifies this door
        "kind": "http-model-list",           // enum: cli-version | cli-model-list | http-model-list | none
        "endpointRef": "openrouter:GET /api/v1/models",  // a REFERENCE the prober resolves; NEVER a runnable string with a key inlined
        "metered": false,                    // model-listing is free; a liveness completion is metered:true
        "vaultKeyName": "metered_openrouter_bench",      // NAME only, ever — never the value
        "freeDefaultCovered": true           // does the default free-probes cadence actually cover this door? (honesty flag — §2.2)
      },
      "topModels": [                         // the top model(s) reachable through this door, with EXACT ids — the SINGLE source
        { "id": "anthropic/claude-opus-4-8", "role": "capable-anthropic", "frontier": true,
          "pricing": { "inPerMtok": 5, "outPerMtok": 25, "currency": "USD" }, "verifiedAt": "2026-07-03" },
        { "id": "openai/gpt-5.5", "role": "capable-openai", "frontier": true,
          "pricing": { "inPerMtok": 5, "outPerMtok": 30, "currency": "USD" }, "verifiedAt": "2026-07-03" },
        { "id": "google/gemini-3.1-pro-preview", "role": "capable-google", "frontier": true,
          "pricing": { "inPerMtok": 2, "outPerMtok": 12, "currency": "USD" }, "verifiedAt": "2026-07-03" }
      ],
      "lastScannedAt": "2026-07-03",         // when the scan last CONFIRMED this door live
      "changeHistory": [                     // append-only; the "growing knowledge" record
        { "ts": "2026-07-03", "change": "door added; 6 frontier ids enumerated from live /models probe",
          "evidence": "GET /api/v1/models -> 200" }
      ],
      "note": "One door -> all frontier. Pricing is metered per-token."
    }
    // ... claude-code, anthropic-headless, codex-cli, gemini-cli, pi-cli, antigravity-cli, gemini-direct-api ...
  },
  "pins": [ /* unchanged — regex-anchored pin sites */ ],
  "flaggedStale": [ /* unchanged — known-stale-pending-operator-confirmation */ ],
  "candidateDoorways": [                     // the doorways the scan should PROBE FOR (incl. not-yet-present ones)
    "claude-code", "anthropic-headless", "codex-cli", "pi-cli", "gemini-cli",
    "antigravity-cli", "openrouter", "gemini-direct-api", "openai-direct-api", "anthropic-direct-api"
  ]
}
```

**Fields per the requirement, mapped:** name → `name`; access method → `accessMethod` (enum cli/direct-api/subscription); installed/reachable status → `status` + `lastScannedAt`; top model(s) + exact ids → `topModels[].id`; pricing (if metered) → `topModels[].pricing`; last-scanned timestamp → `lastScannedAt`; change history → `changeHistory[]`; noticing NEW doorways → `candidateDoorways[]` + the scan's vault-key auto-notice (§2.5).

**Initial population (Frontloaded Decision D4).** Rollout step 1 seeds each door's `topModels` **1:1 from the already-reviewed values behind today's `frontierAllowlist`**, with `verifiedAt` honestly stamped `"carried-over-from-allowlist"` and `frontier:true`. This requires **no new research/operator round-trip to land the PR** — it is a faithful projection of the *already-reviewed* set. The first *enabled* scan then re-verifies and updates `verifiedAt`/`changeHistory`. (The alternative — shipping `topModels: []` empty — was rejected: it would make the derived frontier set empty and the feature hollow at launch.)

```jsonc
  "lastDiff": {                            // what CHANGED vs last-SURFACED (drives the attention surface)
    "newDoorways": [], "topModelChanged": [], "wentStaleOrDeprecated": [], "wentUnreachable": [],
    "parseDrift": [],                      // L5 canary: door answered but its list no longer parses (§1.3 mapping)
    "frontierSuspicions": []               // e.g. flaggedStale.suspectedFrontier CONFIRMED live
  },
  "surfacedBaseline": { "doorways": [], "at": null },  // last-SURFACED state — advances ONLY on confirmed 2xx delivery (§2.6); the diff is computed against THIS, so a failed/held notice re-surfaces next cadence
  "lastDeliveryFailedAt": null             // a PURE timestamp/boolean marker (NOT stored diff content) — set when the last in-process POST /attention failed/was-held. Retry ALWAYS RECOMPUTES the diff fresh from observed-vs-last-surfaced (never replays stored content), and this marker CLEARS on the next confirmed delivery OR when the recomputed diff is empty (re-convergence). So a resolved blip fires no false notice and cannot wedge.
}
```

`probeStatus` is a **fixed classified enum** — `ok | not-installed | timeout | dns-fail | http-4xx | http-5xx | malformed-response | oversize-response | not-probed-this-scope | not-probed-this-run | not-probed-budget-refused` — never verbatim remote/error text (§Security). `reachable` is a live-probe fact, never read from a config field (*Verify the State, Not Its Symbol*). `pendingFlip` implements the 2-scan debounce (§2.3). This file is the observed truth for *this* machine; it is never authoritative for routing (the canonical layer is).

**`probeStatus` → `reachable` → diff mapping (P20 + L5 — no false alarms, no silent rot).** `reachable` is tri-state (`true` | `false` | `null`=unknown), and the non-`ok` statuses split into three classes that map deliberately (a `probeStatus` must NOT collapse "I couldn't confirm" into "the door is down," and a door that *answers but no longer parses* must NEVER silently stop growing the map):
> - **Transient / no-answer** (`timeout`, `dns-fail`, `http-5xx`) → `reachable: null` (UNKNOWN). Per P20 (Verify the State, Not Its Symbol, mode C), "I couldn't reach it this run" is *unknown*, not the alarming `false` state — so it does **NOT** trigger a `wentUnreachable` diff (even after the 2-scan debounce, two transient blips stay quiet). This is also why the notice fails toward silence.
> - **Definitive unreachable** (`not-installed`, an auth-class `http-4xx`, a `dns-fail` that resolves to a genuinely removed endpoint) → `reachable: false`; this MAY surface `wentUnreachable` (after debounce).
> - **Parse-drift** (`malformed-response` / `oversize-response` **on a door that DID answer**) → `reachable` stays `true`, BUT this is an **L5 drift canary that IS SURFACED as a distinct diff class** ("door answered but its model list no longer parses — the probe parser needs attention"), **never a silent map-non-update**. This is the load-bearing anti-rot guarantee: without it, a drifted CLI/API output format would leave `reachable:true`, silently fail to refresh `topModels`, and the map would stop growing for that door with no diff and no alarm — *exactly* the "nothing made anyone re-check" incident this whole spec exists to prevent. Graceful degradation that never tells the operator maintenance is due is silent rot, not an accepted cost (§Alternatives is corrected accordingly).

**Corrupt / old-schema handling.** On read: a corrupt/unparseable/unreadable file → log ONCE and start a fresh scan-state (never crash the job — matches `PreferencesManager`'s "a corrupt read should never crash a session start" and `TopicIntent`'s "Corrupt file, starting fresh"). An older `schemaVersion` → **additive field backfill** (missing fields default; never a destructive reset). A fresh/backfilled state simply means the next diff has no prior to compare against (it surfaces nothing until it has a baseline, never a false "everything is new" flood).

**Untrusted-on-every-read (not just corrupt-file handling).** Clamping remote values at *write* time (§2.0) is necessary but **not sufficient**: `.instar/state/doorway-scan.json` is a machine-local *plaintext* file (same at-rest posture as the Playwright Profile Registry / SelfKnowledgeTree / relationships store — see §Multi-machine), so anything with filesystem access to this machine can rewrite it *after* the clamped write, planting **well-formed but poisoned** values that a corrupt-file check sails right past (a `topModels` entry carrying shell metacharacters/markup, an out-of-enum `probeStatus`, a non-boolean `reachable`, a forged `pendingFlip` fabricated to fake the 2-scan debounce). Therefore **every field is re-validated on every load through a single `loadScanState()` funnel** — the on-disk file is treated as untrusted data on every read, exactly as if it had just come off the network:
The clamp list is **EXHAUSTIVE — no field is exempt** (an enumerated-but-incomplete clamp list is itself the bug; a unit test asserts the funnel covers every field the scan-state schema defines, so a field added later cannot silently skip validation):
> - `probeStatus` → must be one of the fixed enum above; anything else is coerced to `malformed-response`.
> - door `id` (`doorways[].id`) → charset-clamped to `[A-Za-z0-9._/:-]`, length-bounded, AND cross-checked against the canonical `candidateDoorways`/`doors` set (an id matching no known candidate is clamped-and-flagged, never surfaced raw). This closes the gap where a planted door `id` (markup / a prompt-injection payload / oversized) would otherwise flow through `lastDiff` (`newDoorways`/`wentUnreachable`) into the `/attention` body and the tier-1 supervisor context.
> - `topModels[]` → each id charset-clamped to `[A-Za-z0-9._/:-]`, length-bounded, the array count-capped; a non-conforming id is dropped (never surfaced raw).
> - `changeHistory[]` → the LIVE scan-state's history is a **RECENT-change ring buffer, count-capped to the most-recent N entries (default 20)** — it is deliberately NOT append-only/unbounded (that framing belongs to the CANONICAL manifest's `changeHistory`, §1.2, the reviewed "growing knowledge" record; the "durable, growing" map is the canonical layer + the registry as a whole, not this bounded live window). The prober's writer applies the same cap on write; each entry's `change` free-text (and `machineId`, and any other free-text field) is length-clamped, stripped of control chars, **AND HTML/markdown-escaped** — because it reaches not only the `/attention` body but the **private-view HTML sink** (§2.6). Control-char stripping alone is NOT enough there: `<`, `>`, `"` are printable, so a planted `change="<img src=x onerror=…>"` would survive to render as HTML in the operator's private view (same-origin as the dashboard PIN/token) — a stored-XSS from a local-disk writer. Escaping at the funnel neutralizes it before any sink.
> - `doorways[]` and every array → a blanket per-array length cap, so a bloated local file can't slow the load funnel or `GET /doorways`.
> - `scanScope` → one of `free-probes | +liveness | +web-verify`, else `free-probes`; `machineId` → charset+length-clamped.
> - `reachable` and other booleans → strict `typeof === 'boolean'`, else defaulted (`reachable → false`).
> - timestamps (`lastScanAt`, `lastScannedAt`, `changeHistory[].ts`) → ISO-8601 parse-or-drop.
> - `consecutiveCompleteFailures` → finite non-negative integer, else `0`.
> - `pendingFlip` → shape-validated; and, load-bearing, **a surfaced flip is gated on THIS run's live probe result, never on a stored `pendingFlip` alone** — a planted `pendingFlip` with no corroborating live probe this run surfaces nothing (§2.3/§2.6). This is the anti-forge mechanism for the debounce.
> - `surfacedBaseline` → its `doorways[]` clamped by the SAME per-field/array rules as the primary `doorways[]` (it is compared-only, never itself an emitted sink); `surfacedBaseline.at` is an ISO-8601-or-null timestamp. `lastDeliveryFailedAt` → an ISO-8601-or-null timestamp, NOT text (it carries no diff content — §2.6 retry always recomputes, never replays — so it has no sink-re-entry path). Both are named here explicitly and covered by the exhaustive-funnel coverage test.
>
> **Honest scope of this guarantee.** Read-clamping *bounds the blast radius* of a hostile local writer across **every enumerated downstream sink** — shell injection, a malformed `/attention` body, a raw `GET /doorways` field, **and the private-view HTML render (§2.6)** (the fourth sink the round-2 §2.6 fix added; free text is HTML-escaped for it) — but it does **not** make a machine-local plaintext file tamper-*proof* against someone who already has disk access to this machine (that is impossible for a plaintext file, and it is the same accepted at-rest trade the other machine-local registries make). The read-clamp's job is containment, not unbreakability; the enumerated-sink list is exhaustive and a new sink added later must be added to it (a test asserts every emitted surface runs values through the funnel). **Routing never trusts scan-state at all** (the canonical layer is authoritative — §1.1); that invariant is unchanged.

### §1.4 One source of truth — the frontier set is DERIVED (supersedes a hand-maintained allowlist)

The first draft *checked* `frontierAllowlist ⊆ topModels` (a "TOOTH 3" consistency test). Review (lessons-aware) correctly flagged that a *check* still leaves the allowlist as a **second hand-maintained list** — three places to edit (the pin, the allowlist, `topModels`), which is exactly the rot this spec fights. The fix is **derivation, not a check**:

- The lint computes the effective frontier set for a door as `{ topModels[].id where frontier === true }`. `topModels` is the **only** hand-edited structure; the frontier set is a *view* of it and cannot diverge by construction.
- **TOOTH 2 (drift)** now checks each pin's id against that *derived* set (unchanged behavior, single source).
- **Backward-compat:** if a manifest has a literal `frontierAllowlist` and **no** `topModels` for a door (old shape), the lint reads the literal list exactly as today (no-op change for un-enriched manifests). If a door has **both** a literal `frontierAllowlist[door]` and `topModels`, the lint emits a **transition finding** ("this door migrated to derived frontier — delete the literal `frontierAllowlist[door]`"), so the old hand-list can't silently linger and re-rot.
- The lint stays model-id-agnostic (it asserts internal consistency between pins and the derived frontier set; it never hard-codes a "right" id).

## §2 — Component 2: The recurring doorway-scan job

Modeled directly on `bench-refresh` (#1352): a scaffolded built-in job template + schedule JSON, **off by default**, tier-1-supervised, gated on `/health`, review-only (never auto-applies to canonical source), and — like bench-refresh's `run2.mjs` — the real work lives in a **deterministic script**.

### §2.0 The deterministic prober (`scripts/doorway-scan.mjs`) — the security spine

The tier-1 LLM session does **not** construct probe commands or handle secrets. A deterministic Node script owns 100% of the network I/O, secret handling, timeouts, sanitization, and payload construction; the job session only *invokes* it, *sanity-checks* its already-clamped JSON, and *surfaces* the result — exactly as bench-refresh wraps `run2.mjs`.

`scripts/doorway-scan.mjs` responsibilities:
- **Two kinds of probe, two paths (token-audit distinction).** (a) **Free model-LIST probe** (the default): CLI doors spawn `which <bin>` / `<bin> --version` / a model-list subcommand with an explicit `timeout`/`--max-time` (§2.3); API doors perform the model-*list* HTTP GET **in-process** (Node `fetch`/`https` with the auth header set on the *request object*), so a secret is **never** placed in `argv`/`environ` of a shelled-out `curl` (SEC finding). A model-list is NOT a generation — it spends **zero tokens** — so a raw in-process GET is correct and needs no token accounting. Response bodies are **size-capped** (default 2 MB) before parsing; an oversize body → `probeStatus: oversize-response`. (b) **Metered `+liveness` COMPLETION** (opt-in): a tiny bounded *generation* IS an LLM call, so — per **Token-Audit Completeness** ("an unmetered LLM call is an unaccountable one") — it is issued through instar's `buildIntelligenceProvider` funnel tagged `attribution.component: "doorway-scan"`, NOT a raw fetch. That funnel gives it token attribution in `/metrics/features` + the TokenLedger, the account circuit-breaker, and the host spawn-cap for free (Structure > Willpower — reuse the funnel, don't re-invent metering). The completion carries a **hard per-probe token/`max_tokens` ceiling** (a price-independent structural bound — see D9) so a stale price can never let it overspend.
- **Secret handling.** Retrieves a needed key via the hardened path in-process, uses it only in the request header, and never writes/echoes it. Discovers metered doors by vault key **NAMES** only.
- **Sanitization at the source.** Every remote-derived value is clamped before it leaves the script: model ids to `[A-Za-z0-9._/:-]`, length-bounded, count-capped; `probeStatus` reduced to the fixed enum above (no verbatim remote/error text ever enters scan-state or a payload). A remote model-list is treated as **data, never instruction**, never executed, never interpolated into a shell command. The writer also applies the **`changeHistory` ring-buffer cap (default 20 entries) and the `change` free-text length clamp on write**, so the durable record is bounded at both write and read (§1.3).
- **Payload construction AND delivery are in-process (the session runs no data-carrying curl).** The prober builds the scan-state JSON, the private-view body, and the `/attention` item body as properly JSON-encoded values, and **performs the `POST /view` and `POST /attention` requests itself, in-process** (it already owns all network I/O, the auth header, and JSON encoding). This closes a round-2 contradiction: the tightened §2.7 guard REFUSES pipes/redirects/substitution (`--data-binary @-`, `> file`, `$(…)`), and the prober is write-restrained to scan-state only (no payload temp-file), so a *session-run* data-carrying `curl` is both forbidden and impossible — moving the POSTs in-process removes the need for one entirely. The session's only sanctioned commands become the prober invocation, the `/health` gate GET, and read-only plumbing (§2.7). All scan-state-derived free text is HTML/markdown-escaped before it is composed into the private-view body (§1.3 — the private view is an enumerated containment sink).
- **Metered completions run SEQUENTIALLY (internal concurrency 1).** A manual `+liveness`/`+web-verify` run issues its bounded completions one at a time (never a parallel fan-out), so doorway-scan can never grab several of the shared host spawn-cap slots at once and transiently starve a co-resident safety-gating call (the tone gate) — the lived "spawn-cap saturation → operator unreachable" failure mode. It is manual-only and off the hot path, so serial costs nothing. (Free-probes issues zero completions, so the scheduled cadence has zero such contention.)
- **Scope + budget enforcement (§2.2), all in code, not prose the LLM must remember:** the **free-probes-never-pays invariant** (D8); the **budget fail-closed invariant** (D9 — a metered scope with an absent/zero/invalid `budgetCapUsd` refuses EVERY metered probe, recording `not-probed-budget-refused`, and never falls back to unlimited); the **metered-scope manual-marker gate** (D10 — `+liveness`/`+web-verify` are refused unless the manual marker env `INSTAR_DOORWAY_SCAN_MANUAL=1` is set, which the scheduler never sets); and — load-bearing — the prober's own **SELF-STANDING scheduled-session refusal**: `doorway-scan.mjs` refuses ANY metered scope whenever it detects it is running inside a scheduled doorway-scan job session (`INSTAR_JOB_SLUG==='doorway-scan'` / `state/active-job.json.slug`), **independent of the manual marker AND of the §2.7 guard**. This matters because the §2.7 guard *fails OPEN* on a scope-resolution error, so it cannot be the SOLE thing stopping an LLM-appended `INSTAR_DOORWAY_SCAN_MANUAL=1 … --scope +liveness` during a scheduled run; the prober's own in-session refusal makes "a scheduled run cannot spend" true even when the guard has failed open. (A genuine human manual run is NOT inside a doorway-scan job session — `INSTAR_JOB_SLUG` is unset/different — so this never blocks the intended manual path.)
- **Write-target self-restraint (belt-and-suspenders for §2.7).** The prober itself resolves its output path and **refuses to write anywhere except `.instar/state/doorway-scan.json`** (+ a temp sibling for atomic rename); a malicious/mistaken `--out`-style argument pointing at canonical source is rejected. So even the one sanctioned writer cannot be steered onto source.

The job session's role shrinks to: run the prober and confirm it produced a well-formed result (a crashed/partial run is noted once and the run exits — next cadence retries). The prober itself — NOT the session — performs the `POST /view` and `POST /attention` in-process (§2.6). This is the Structure-over-Willpower form of the security guarantees: they are enforced by the script, not entrusted to a prompt. **The prober advances its diff baseline only on confirmed delivery (no silent notice loss) — see the last-surfaced-vs-last-observed rule in §2.6.**

### §2.2 Cost tiers (cost-aware by construction; the default never pays)

The scan runs in one of three explicit scopes; the default spends **zero metered budget**:

- **`free-probes` (default, and the ONLY scope the scheduled cadence runs).** CLI doors: `which` + `--version` + a model-list subcommand where one exists. API doors: the model-*list* endpoint only (listing does not spend generation tokens). **Structural invariant (D8), enforced in `doorway-scan.mjs`:** under `free-probes`, any door whose only available probe is flagged `metered:true` is **skipped** and recorded `probeStatus: not-probed-this-scope` — it may **never** fall back to a paid probe to satisfy the default scan. A unit test with a metered-only-door fixture asserts it is never charged under `free-probes`.
- **`+liveness` (opt-in, human-manual-only, budget-capped).** Adds a *tiny* bounded completion per metered door to prove the door generates, not just lists. Enforced fail-closed by TWO independent gates in the prober: (i) the **manual-marker gate** (D10) — refused unless `INSTAR_DOORWAY_SCAN_MANUAL=1`, an env the scheduler never sets; and (ii) the **budget gate** (D9) — a hard per-run `budgetCapUsd` that must be present and `> 0`, with an **unknown or missing per-probe price REFUSING to run** that probe and the estimated run cost REFUSING if it would exceed the cap. An absent/zero/invalid cap refuses every metered probe outright (never unlimited).
- **`+web-verify` (opt-in, human-manual-only, bounded).** A bounded web check for *frontier* models no door yet exposes; produces advisory `flaggedStale.suspectedFrontier` suggestions for the maintainer, never auto-written. Same manual-marker + budget gates as `+liveness`.

Scope is a prober argument (`--scope free-probes|+liveness|+web-verify`), **defaulting to `free-probes`**. The **scheduled cadence invokes a FIXED literal `node scripts/doorway-scan.mjs --scope free-probes`** — the scope is a hardcoded string in the job body, never an LLM-substituted or config-derived value, so the tier-1 session cannot select a paid scope (and the template-invariant test asserts the body contains that literal and does NOT contain `+liveness`/`+web-verify`). Because the job session's Bash is ultimately LLM-controlled, the fixed body is backstopped structurally by the D10 manual-marker gate above: even if the session appended `--scope +liveness`, the prober refuses it (the scheduler-spawned env has no `INSTAR_DOORWAY_SCAN_MANUAL`). A **TTY check is deliberately NOT used** for this: a scheduler-spawned tmux session owns a pty just like a human terminal, so a TTY check cannot distinguish them — the env-marker is the robust discriminator. A metered scope is therefore only ever reachable by a human running the prober by hand (marker set + a positive `budgetCapUsd`). The chosen scope is recorded in scan-state and in any attention item so the operator sees exactly what was (and wasn't) checked.

**Honest per-door coverage.** Each door's `probe.kind` (incl. `none`) and `probe.freeDefaultCovered` state plainly whether the free default cadence actually covers it. Some doors have **no enumerable free model-list** (a not-installed CLI like codex; a door whose frontier is known only via an internal known-models list, not a live probe): these are marked `freeDefaultCovered:false` and rely on `+web-verify` or a manual maintainer pass. The spec does **not** imply blanket free coverage — the motivating gemini-2.5-pro case is covered by the gemini door's live model-list, but the honest map states which doors are and aren't.

### §2.3 Fail-safe + the three brakes (No Unbounded Loops)

Every door probe is **independent** and **bounded**. The prober applies:
- **Per-probe timeout (explicit value):** `timeout 10 <bin>` / `--max-time 10` / a 10s fetch abort. No probe can hang indefinitely.
- **Global wall-clock deadline** for the whole scan (default 5 min, well under `expectedDurationMinutes: 10`): any door not started by the deadline is skipped and recorded `not-probed-this-run` (never silently omitted). This bounds total time even as `candidateDoorways[]` grows.
- **Per-probe isolation:** a probe that errors/times-out/returns garbage records that door's classified `probeStatus` and continues — it never aborts sibling probes or poisons their results.

**The three brakes the standard requires:**
1. **Timeout** — above.
2. **Backoff / debounce (2-scan hysteresis) — ASYMMETRIC by direction:** the debounce applies ONLY to **alarming** transitions — `reachable` true→false and a new-doorway appearance — which are **not** surfaced on a single scan: the first such divergent scan records a `pendingFlip`, and only a *second consecutive* scan agreeing flips the status and surfaces it (killing the "one flaky network blip on a weekly cadence → spurious `wentUnreachable`" noise; weekly = 52 chances/year to false-flip). A **recovery** (`false→true`, a door becoming reachable again) is the OPPOSITE — it flips `observed` **immediately, un-debounced** — because a single good scan is trustworthy enough to stop re-surfacing a stale degradation, and a recovery raises no alarm of its own (there is no "recovered" change class in §2.6). **This asymmetry is load-bearing for the §2.6 delivery-robustness flip-back guarantee:** it is exactly what makes a "flip → failed delivery → flip-back-before-next-cadence" recompute to an empty diff (observed recovers to `true` immediately, matches the un-advanced `last-surfaced` `true`) → no false stale notice, marker clears (a symmetric debounce would leave `observed` still `false` on the flip-back scan and wrongly re-surface the stale degradation). Mirrors the codebase's established corroboration pattern (rope-health recovery probe, `ActiveWorkSilenceSentinel`'s re-capture-and-corroborate). A Tier-1 test asserts the direction asymmetry (an alarming flip needs 2 scans; a recovery is immediate).
3. **Breaker + failure-backoff for sustained failure (P19 brakes; P22-shaped escalation):** `consecutiveCompleteFailures` is tracked in scan-state. A *completely* failed scan (zero doors answered — e.g. blocked egress) increments it. The scan does NOT escalate on first detection — it **retries on a widening backoff first** (the weekly cadence IS the base retry interval; after a complete failure the effective next-attempt interval widens — e.g. skip-N-cadences backoff — so a persistent environment fault doesn't re-run pointlessly at full rate), and only after **3 consecutive complete failures** does it raise ONE stable-id attention escalation ("doorway-scan has found zero reachable doors for 3 consecutive runs — check the execution environment") and then keep quietly retrying at the backed-off interval (Eternal-Sentinel shape). This is the **P22 (Self-Heal Before Notify) shape**: the operator-raise sits DOWNSTREAM of retry-and-corroborate (retry attempted → still failing after the threshold → escalate once), not on raw first detection; the escalation is deduped by stable id so it can't flood. **Scope note (P22):** the map-*change* diffs of §2.6 are informational maintainer-*product* (news the operator wants), NOT internal degradations to self-heal — only the zero-doors *breaker* is an internal-health signal, and it is the one that carries the retry-then-escalate gate. This closes the gap the conformance gate + scalability reviewer flagged: `CrashLoopPauser` only catches job *crashes/timeouts*, not a clean exit-0 "zero doors" run, so this feature carries its own breaker + backoff.

### §2.4 Secret-safe (key NAMES only, never values; never in argv)

Enforced in `doorway-scan.mjs` (§2.0): metered doors are discovered by vault key **NAMES** only; a needed key is retrieved and used **in-process in the request header**, never placed in a shelled-out command's argv/environ, never echoed, never written to scan-state, never included in an attention item or maintainer diff. The scan-state stores only the vault key *name* (`vaultKeyName`). A probe that would require printing a secret to satisfy the diff is refused. The `credential-leak-detector` PostToolUse discipline is the backstop.

### §2.5 Candidate / newly-configured doorway detection

**Honest scope (truthful-data):** this is **candidate-and-config-driven detection**, NOT open-ended "discover any doorway in the world" — the prober notices a door only when it is a prelisted `candidateDoorways[]` entry OR is implied by a vault key matching a known metered-door pattern. It cannot invent a doorway type nobody has told it to look for. (The earlier "notice a brand-new doorway" phrasing overclaimed; a truly open-ended provider-registry discovery would be a separate future enhancement.) The prober probes-for every door in `candidateDoorways[]`, even ones **not yet present** in `doors`. Two detection paths:
1. **Candidate list.** A candidate reachable but absent from `doors` is a **newly-configured-doorway** change (subject to the 2-scan debounce; then surfaced; proposed for canonical fold-in).
2. **Vault-key auto-notice.** A vault key matching a metered-door pattern (`metered_*`, `*_api_key`, `*_bench`) that maps to no known door surfaces as *"possible new metered doorway — key `<name>` present, no door defined"* (name only; the key NAME is charset-clamped before it enters the payload). This is how a paid Gemini/OpenAI/Anthropic key the operator dropped gets *noticed* without anyone remembering to register the door.

Adding a candidate is a one-line edit to `candidateDoorways[]` **only when it reuses an already-implemented `probe.kind`** (`cli-version`, `cli-model-list`, `http-model-list`) — that is *candidate activation*, no code change. A candidate that needs a genuinely NEW probe kind (a provider with a different auth flow / listing endpoint / output shape) requires a real probe implementation (parser + auth + listing logic), NOT a one-line edit — the spec does not imply otherwise. Each candidate therefore declares its `probe.kind`, and the prober only activates a candidate whose `probe.kind` it actually supports (an unsupported kind is recorded, not silently treated as reachable/unreachable).

**`frontier:true` is operator judgment, not scan truth (truthful-data).** The scan can *enumerate* which model ids a door exposes and *propose* a newly-seen id as a candidate; it **cannot rank capability** and therefore never asserts `frontier:true` itself. Marking a proposed id as frontier (and folding it into canonical `topModels`) is the operator's confirmation step through instar-dev (the lint's own "operator-confirm the frontier id" rule). The live scan-state records only *observed* ids + reachability; the canonical layer records the operator's *classification*. This keeps the two honestly separated (observed vs classified) and is why the scan never auto-writes canonical (§2.7).

**Classification criteria for `frontier:true` (so the map stays semantically consistent, not just structurally fresh).** "Top / frontier" is **routing-suitability, not marketing tier**: a model qualifies `frontier:true` for a door when it is a current top-*capability*, **GA/usable-today** option for that door's provider role (the pecking order the doorway-research process produces — cross-checked web research + INSTAR-Bench evidence — e.g. the 2026-07-03 research: usable-today Fable 5 → Opus 4.8 → GPT-5.5 → Gemini 3.1 Pro; a preview/partner-gated/not-broadly-usable model like GPT-5.6-Sol is NOT frontier). **Who confirms:** only the operator, via instar-dev (never the scan). **Conflicting provider claims** resolve to the operator's evidence-based judgment (a vendor "strongest yet" that isn't broadly reachable does not qualify). Evidence anchors: capability (bench/research rank), GA/usable availability, and reachability through the door — not context-window or modality alone. **To make the classification reproducible/auditable rather than subjective churn (external-reviewer point), each `frontier:true` entry carries STRUCTURED evidence fields:** `evidence: { kind, sourceUrl?, observedAvailableAt?, benchRef?, operatorConfirmedBy, confirmedAt }` — so a promotion/removal is anchored to a citable source + a named operator + a date, not a bare boolean. **Minimum promotion criteria:** GA/usable-today for the door's role AND a capability anchor (a bench/research ref) AND operator confirmation; **removal** when it is superseded by a higher-capability usable model for that role OR ceases to be usable. This is the human-judgment layer the derived-frontier lint (§1.4) then mechanically enforces consistency against — the fields make the judgment inspectable, the lint keeps the pins consistent with it.

### §2.6 Diff + surface (only changes, never noise, never silent pool-loss)

The prober loads the prior scan-state **through the `loadScanState()` read-validation funnel** (§1.3 — every field re-clamped, the file treated as untrusted on read), computes `lastDiff` over the validated values **against the `last-surfaced` baseline** (not the always-advancing last-observed state — see "Delivery robustness" below), and emits the finished attention payload in-process **only if** the (debounced) diff is non-empty, using a **STABLE, machine-qualified id + sourceContext**. Because the diff (and therefore the `/attention` body) is built only from read-validated values, a planted poisoned value can never reach the attention surface raw, and a forged `pendingFlip` cannot surface a change without a corroborating live probe this run (§1.3):

```jsonc
{ "id": "doorway-scan:changes:<machineId>",
  "sourceContext": "doorway-scan:<machineId>",   // machine-qualified — MUST be set (see below)
  "title": "Doorway map changed: <n> change(s)",
  "body": "<JARGON-SAFE plain-English summary of the change count + kinds, plus a private-view link to the raw diff — NO raw file paths / config keys / CLI commands (see below)>",
  "priority": "medium", "source": "agent" }
```

**The attention body MUST be tone-gate-safe — the raw maintainer-diff goes in a private view, not the `/attention` body (code-confirmed silent-loss path).** `POST /attention` runs the item's text through the outbound tone gate (`checkOutboundMessage`, `src/server/routes.ts` ~L12778, `messageKind:'reply'`), which is *designed* to block raw file paths, config keys, CLI commands, and dev jargon — and a raw doorway maintainer-diff is *made of* exactly those (`scripts/model-registry-freshness.manifest.json`, `topModels`, "update the pin via instar-dev, re-run the freshness lint"). Posting the raw diff would very plausibly be **held/blocked → the item is never created → a silent loss** (and §2.6's machine-qualified `sourceContext` only closes the *coalesce* loss, not this *delivery-path* loss). So the prober emits a **jargon-safe body**: a terse plain-English summary ("your doorway map changed: 1 door went unreachable, 1 top model moved — tap for details") + a **private-view link** (the raw technical diff — paths, exact ids, the instar-dev pin instructions — is rendered as an auth-gated Private View via `POST /view` and linked, never inlined into the tone-gated body). This mirrors the CLAUDE.md outbound-advisory guidance ("replace a raw file path by publishing a private view and sending the link"). The prober performs the `POST /view` and `POST /attention` **in-process** (§2.0), so no tone-gated raw text is ever handed to a session-run curl. **Honest v1 posture (external-reviewer point):** coupling an infra-health notice to a prose-oriented outbound tone gate is admittedly a non-ideal fit; v1 accepts it and delivers through the EXISTING attention path, with the jargon-safe body + the confirmed-delivery / last-surfaced re-surface (the "Delivery robustness" rule below) as the **compensating control** so a tone-gate hold delays rather than drops. A dedicated system-notification lane that bypasses the prose gate for infra notices is the cleaner long-term path — Deferred <!-- tracked: 29723 --> #6. **Precise guarantee (external-reviewer point):** v1 promises **no SILENT loss** (an undelivered change re-surfaces every cadence + `lastDeliveryFailedAt` records the failure — it is never silently dropped), NOT guaranteed delivery; actual delivery is **best-effort, bounded by the tone gate's availability** (a permanently-down gate yields repeated *visible* re-surfacing + the failure marker, which is the honest outcome, not silence).

**Link handling — NEVER a localhost link (round-2 finding).** A `/view/:id` link is only put in the body when a **public tunnel URL is available** (the tunnel is opt-in/often off). When there is no public URL, the private-view link would be `http://localhost:…/view/…` — a machine-local link, which a pre-existing server/outbound guard **refuses regardless** (CLAUDE.md: "a localhost link is the one finding `--ack-advisory` can NOT deliver"), reintroducing the very silent-loss §2.6 exists to close. So with no tunnel the body **degrades to a link-free plain-English summary** (change count + kinds) and points the operator to `GET /doorways` for detail — never a localhost link. A test asserts the emitted body carries no localhost/endpoint link. When a tunnel URL IS used, the link is a **token-free bare `/view/:id`** (the operator authenticates via their own dashboard/tunnel session) — **never a token-bearing tunnel URL**, which would both leak an auth token at-rest into the attention store + Telegram history AND be the endpoint class the tone gate blocks (round-3 finding). The **private view uses a STABLE per-machine view id and is UPSERTED via `PUT /view/<stable-id>` each run** (not a fresh `POST /view` that would orphan the prior view every scan → a slow record leak), so the raw-diff URL is stable across runs. The maintainer-diff (and its private view) is only generated where the instar source tree is present (§2.9); a pure end-user agent gets only the plain-English "your doorway `X` went unreachable" half, which is already jargon-safe.

**Delivery robustness — advance the diff baseline ONLY on confirmed delivery (no silent notice loss).** The scan-state keeps TWO distinct baselines: **`last-observed`** (this machine's latest probe truth, always written) and **`last-surfaced`** (the state as of the last CONFIRMED-delivered notice). **Initial-seed carve-out (no fresh-state flood):** because the debounced diff is now computed against `last-surfaced`, a fresh/never-surfaced state (`surfacedBaseline.at === null` — which arises after a corrupt-file reset per §1.3 AND on any new machine joining the pool, since the file is machine-local + backup-excluded) would otherwise diff the full door list against an EMPTY baseline and emit a spurious full-map notice. So there are TWO advancement paths: (1) **INITIAL SEED** — when `surfacedBaseline.at === null`, set `surfacedBaseline := observed`, surface **NOTHING**, and require no delivery (this preserves §1.3's "surfaces nothing until it has a baseline, never a false 'everything is new' flood" promise, which the last-surfaced fold would otherwise have silently broken); (2) **subsequent non-empty diffs** — advance ONLY on a confirmed 2xx (below). A Tier-1 test asserts a fresh scan-state with a full observed door list surfaces nothing and seeds the baseline. The debounced diff is computed against **`last-surfaced`**, and `last-surfaced` is advanced **only after the in-process `POST /attention` returns a confirmed-created 2xx** — NOT on a tone-gate hold, a non-2xx, or a transient failure (the tone gate fails closed on provider exhaustion, which is content-independent, so a jargon-safe body does not guarantee delivery). If delivery does not confirm, `last-surfaced` stays put and the still-pending change **re-surfaces on the next cadence** — closing the round-3 silent-loss gap where writing the new state advanced the baseline and a failed post then buried the change forever. **Retry is state-based, NOT a replay:** the next cadence **always RECOMPUTES** the diff fresh from read-validated `observed`-vs-`last-surfaced`; it never replays stored diff *content*. A pure marker (`lastDeliveryFailedAt`, a timestamp — never stored diff text) records that the last delivery failed, and it **clears** on the next confirmed 2xx OR when the recomputed diff is empty (re-convergence). Two consequences (round-4 adversarial finding): (a) a door that flipped, failed to deliver, and then flipped BACK before the next cadence recomputes to an empty diff → **no false "went unreachable" notice, and the marker clears — no wedge** (this relies on the §2.3 debounce ASYMMETRY: a recovery `false→true` flips `observed` immediately/un-debounced, so the flip-back scan's `observed` is already back to `true` and matches the un-advanced `last-surfaced` `true` → empty diff; a symmetric debounce would wrongly re-surface the stale degradation); (b) because retry composes only from freshly read-validated observed values (never from stored `lastDeliveryFailedAt`, which carries no text), there is no stored-text re-entry into the escape/sink path. A Tier-1 test asserts: scan-state write succeeds + attention POST fails/held → the change re-surfaces on the next scan (baseline did not advance); AND a flip-then-flip-back-before-next-cadence produces no notice and clears the marker.

**Routing — the ownerless notice lands in the ONE alerts/hub topic, not a per-machine new topic (P23).** A doorway-scan diff (and the zero-doors breaker escalation) is an *ownerless system/housekeeping notice* — it belongs to no user conversation — so per P23 (Notices Route to the Alerts Topic, Never a New One) it routes to the dedicated alerts/hub topic (or the operator-set `maintenance.doorwayScan.digestTopicId`, which **defaults to that alerts surface**, not deferred <!-- tracked: 29723 -->). This reconciles with the P17 machine-qualified `sourceContext` choice: the machine qualification keeps each machine's finding a DISTINCT ROW (no coalesce content-loss) **within that one surface** — it does NOT spawn a forum topic per machine. The `topicCreationBudget` / `AttentionTopicGuard` chokepoint backstops any flood structurally regardless (a pathological N-machine burst coalesces at the ceiling, never a wall of topics).

**Why machine-qualified `sourceContext` is load-bearing (code-confirmed).** In `src/server/routes.ts`, `sourceContext` falls back to the generic `source` (`"agent"`) when unset, and the pool-merge P17 coalesce prefers `sourceContext`; the merge **discards the losing item's body**, keeping only `coalescedFrom`. Two machines' genuinely different doorway diffs (or a collision with an unrelated feature's `source:"agent"` item) would collapse into one row with the other machine's finding invisibly lost — directly defeating "never a silent loss." Each machine's doorway finding is a distinct physical fact, so it must **never** coalesce: a machine-qualified `sourceContext` (and id) guarantees it.

Change classes surfaced: **newly-configured doorway**, **new/changed top model**, **door gone definitively unreachable** (`reachable:false` only — a transient `reachable:null`/unknown does NOT surface, per the §1.3 P20 mapping), **a routing pin fallen behind** (a `flaggedStale` suspicion the live probe now confirms), and — the L5 canary — **"door answered but its model list no longer parses"** (a parse-drift `malformed-response`/`oversize-response` on a door that DID answer; surfaced so a drifted probe parser is never a silent map-non-update, per §1.3). A clean (empty debounced) scan surfaces **nothing**. Priority `medium` by default; a door a *critical gate routes through* going unreachable is `high` (Frontloaded Decision D7 defines "critical door": any door named in `sessions.componentFrameworks` resolution, or the sole doorway for a routing category with no fallback). Remote free text reaching the tier-1 supervisor is wrapped in an explicit untrusted-data envelope (`<untrusted-doorway-probe-output>…</untrusted-doorway-probe-output>`), never treated as instruction.

### §2.7 Never auto-apply to canonical — the REAL mechanism (not SourceTreeGuard)

The first draft claimed this was enforced by `SourceTreeGuard`. Review (security, code-confirmed) correctly flagged that as a **mischaracterization**: `SourceTreeGuard` blocks *other agents' destructive managers* from targeting the instar checkout; it is **not** a PreToolUse hook and does **not** intercept a live job session's `Edit`/`Write`. The guarantee is instead enforced two ways:

1. **toolAllowlist restriction (primary).** The job frontmatter sets `toolAllowlist: ["Bash"]` and `unrestrictedTools: false` — the session has **no `Edit`/`Write` tool at all**, so it cannot modify any file directly. Its only writes are (a) the machine-local scan-state (written by the prober script it invokes — that is the agent's own record, not instar source) and (b) the `/attention` API.
2. **PreToolUse command-allowlist guard (the ACTUAL "never edits source" enforcer) — inverted from a denylist, with fully-specified match semantics.** `toolAllowlist: ["Bash"]` removes the Edit/Write *tools* but does NOT stop Bash from writing files a dozen ways (`cp`, `dd`, `mv`, `install`, `ln`, an interpreter's `open(...,"w")`, `awk … > f`, `perl -i`, a `cat <<EOF > file` heredoc, `git checkout -- <path>`, `patch`, output-redirect `curl -o`), and statically parsing an arbitrary shell command for its "resolved write targets" is undecidable. So this guard — NOT the toolAllowlist — is the real defense, and it is a strict command-shape **ALLOWLIST** with the following **fully-specified, fail-closed match semantics** (round-3 security/adversarial finding: an under-specified matcher is the whole hole — a prefix/substring matcher would wave through `node scripts/doorway-scan.mjs --scope free-probes; cp evil docs/LLM-ROUTING-REGISTRY.md`):
   - **A real shell-grammar parse is the boundary, NOT a regex/byte-scan (the canonicalizer is specified, not just the refuse-rules).** The refuse-rules below presuppose turning the raw command string into structure; that parse IS the security boundary, so it is mandated: the guard **parses the command with a real shell grammar and ACCEPTS only if it is exactly ONE simple-command AST node with NO expansion/operator/redirection nodes** (no pipe, list, subshell, command/process substitution, here-doc, redirection, or assignment-prefix node), then matches `(executable, argv)` against the sanctioned shapes. Anything that fails to parse to that single-simple-command form — or contains any expansion node — is REFUSED. A regex/byte-scan is explicitly disallowed (it is the classic both-ways failure: too strict → false-refuses legit quoting/whitespace → the scan silently breaks; too loose → misses obfuscation). **Tolerated benign normalizations are named** so the legit prober/read invocations don't false-refuse: surrounding/collapsed whitespace, and quoting of an argv token (`--scope "free-probes"` ≡ `--scope free-probes`). `argv` comparison is post-parse (unquoted) exact-token.
   - **Whole-command-anchored, exact executable + argv-shape match.** The guard matches the parsed `(executable, argv)` against a small set of exact shapes — never a command-name prefix/substring.
   - **Any shell compounding/redirection/substitution/env-prefix → REFUSE.** A sanctioned doorway-scan command must be ONE simple command with no shell operators: the presence of any of `;` `&&` `||` `|` `$(…)` `` `…` `` `<(…)` `eval` newline, any redirection (`>` `>>` `2>` `&>` `tee`), OR a leading env-var assignment prefix (`FOO=bar cmd`) is an immediate REFUSE. (This is what defeats the `unset INSTAR_JOB_SLUG; cp …` chain AND the `INSTAR_DOORWAY_SCAN_MANUAL=1 node …` money-gate bypass — see the D10 dependency note below.)
   - **Fails CLOSED on anything undecomposable.** Any command the guard cannot statically prove is exactly one sanctioned simple invocation is REFUSED (never allowed-by-default within the job session).
   - **No interpreters, ever.** `python3 -c`/`node -e`/`perl -e`/`ruby -e`/`sh -c`/`bash -c` are Turing-complete write primitives, so they are **NOT sanctioned shapes** (dropping the `python3 -c` config-read the first draft borrowed from bench-refresh). The prober needs no interpreter: the authToken is read from the already-injected **`$INSTAR_AUTH_TOKEN`** env (verified scheduler-injected), and any config field the body needs is read with a fixed non-Turing-complete reader (`jq -r '<literal path>' .instar/config.json`, no shell interpolation).
   - **`curl` is host-pinned and write-flag-free.** Sanctioned `curl` shapes require the URL host to be `http://localhost:$PORT/…` (the `/attention`, `/view`, `/health` endpoints) and **forbid every output-redirect flag** (`-o` `-O` `--output` `--remote-name` `--create-dirs`) — so `curl` can never write a response body over a source file.

   **The sanctioned shapes (exhaustive) shrank in round 2 — the prober now POSTs `/attention` + `/view` in-process (§2.0), so the session runs NO data-carrying curl at all.** The sanctioned session commands are just: the prober `node scripts/doorway-scan.mjs --scope free-probes` (itself write-restrained to scan-state and the sole network/POST actor, §2.0); the `/health` gate GET (`curl -sf http://localhost:$PORT/health`, host-pinned, no output-redirect flag); and read-only plumbing (`test -f <literal>`, `cat`/`jq -r` of scan-state or a literal config path). Nothing else — and notably no `curl --data`/`--data-binary`/pipe, which the parse-rules refuse anyway. Every write/exec primitive is defeated at once because none is on this list, and the tiny set makes the AST match trivially decidable.

   **Scoped to only this job's session; env-first (zero I/O on the no-op path); fail-OPEN on scope, fail-CLOSED on match.** The guard fires on every Bash call fleet-wide but is a no-op for ~all of them, so it takes an **env-first fast path**: it checks the scheduler-set `INSTAR_JOB_SLUG` first (an env read, ZERO disk I/O); if that is unset or ≠ `doorway-scan` it **ALLOWS immediately without touching disk** (no `active-job.json` read on the hot path — this machine has had event-loop/load meltdowns, so an unconditional per-Bash-call disk read is avoided). It reads/corroborates `state/active-job.json`'s `slug` **only when the env slug already matches** (to confirm before fail-closing). The two fail-modes apply at DIFFERENT stages: (a) **scope resolution fails OPEN** — if the guard errors or cannot resolve whether this is the doorway-scan session, it ALLOWS (a strict no-op), exactly like the sibling `pr-hand-lease-guard.js` fail-open, so a guard bug can NEVER block Bash in an unrelated instar-dev/interactive session; (b) **command matching fails CLOSED** — once the guard has confirmed this IS the doorway-scan session, any command not provably one sanctioned simple invocation is REFUSED. Because a PreToolUse hook sees the entire command string before it runs, a mid-command `unset INSTAR_JOB_SLUG` cannot escape scope (the compound is evaluated under this session's slug, and the `;`/`unset` tokens trip the compounding-refuse rule anyway).

   **D10 money-gate dependency (stated explicitly, per the security finding).** The D10 "scheduled runs can't spend" guarantee is NOT self-standing: an LLM-controlled Bash session could otherwise prefix `INSTAR_DOORWAY_SCAN_MANUAL=1 node scripts/doorway-scan.mjs --scope +liveness`. The ONLY thing preventing that is THIS guard's env-prefix-refuse + `--scope free-probes`-exact-argv rule — so a prompt-injected doorway-scan session cannot self-authorize metered spend. The guard's env-prefix and argv-exact rules are therefore load-bearing for BOTH the source-write and the money guarantees.

   **Dynamic protected set (no rot):** where a defense-in-depth write-target check is also applied, the protected canonical paths are read LIVE from `manifest.pins[].file` ∪ the canonical registry/doc filenames — never hardcoded, so adding a pin automatically extends protection. This guard is a **new built-in hook**, installed via the always-overwrite `instar/` migration path (§Migration Parity).

A change implying the canonical map is stale is surfaced as a **maintainer diff** ("fold into `topModels` + update the pin via instar-dev, then re-run the freshness lint") — never applied. This mirrors bench-refresh's "critical-gate routing is operator-review-gated — never auto-apply."

**Alternative considered — auto-PR (raised by the external reviewer).** A common industry pattern is for the scan to auto-open a PR against the canonical registry. We deliberately choose the **decoupled attention-queue + maintainer-diff** path instead: (a) it keeps the job strictly signal-only (Signal vs. Authority) with zero write authority over source, which the auto-PR pattern erodes (a bot committing to source, even behind review, is a larger blast radius and a new credential/identity surface); (b) canonical model-id changes are *operator-confirmed by design* (the lint's own "operator-confirm the frontier id" rule), and a maintainer folding a surfaced diff through instar-dev already runs the full ship gate + freshness lint; (c) it needs no git-write credential for the job. Auto-PR is a reasonable future enhancement once the scan has soaked, but it is out of scope and not the v1 posture.

### §2.8 Per-machine independence (the scan runs on EVERY machine)

The whole design rests on each machine scanning its own disk. But the job-claim/lease system is keyed globally on `jobSlug` (code-confirmed: `JobLeaseClaimStore.claims` is `Map<jobSlug, …>` with no topic/machine key; a session-spawning job takes a claim before spawning and a losing peer yields). On a **coherent dev-agent pool** (where the `ws43JournalLease` path engages — Echo's own deployment shape, toward which the graduated rollout moves), that would elect **one** machine to run `doorway-scan` per tick; every other machine would silently skip forever, and the "live scan-state is a physical fact of THAT machine's disk, updated every run" guarantee would quietly stop being true everywhere except the lease-winner.

**Fix (new job-def flag `perMachineIndependent: true`).** `JobScheduler` skips claim/lease negotiation for a job carrying this flag — the same way it already skips it for `script`-type jobs — so each machine runs its own scan independently. Concretely, the flag short-circuits the claim seam at `JobScheduler.triggerJob` (code-confirmed: it resolves a claim path via `resolveClaimPath(slug)` then yields when `hasRemoteClaim(slug)` is true, recording `recordSkip(slug,'claimed')`) — a `perMachineIndependent` job never consults that path and so never yields to a peer's claim. This is a small, well-scoped `JobScheduler`/`JobDefinition` addition, and it is the honest structural expression of the design intent. (Today, on a *standard fleet agent*, the lease path is inert by accident — `ws43JournalLease` resolves dark and the legacy `JobClaimManager` is never wired — so the scan would run per-machine anyway; but relying on an accident is exactly the drift this spec fights, so the flag makes it explicit and future-proof.)

**This flag is a durable, cross-cutting `JobScheduler` change, so it is frontloaded as its own decision (D11) with a load-bearing misuse warning:** `perMachineIndependent` defaults `false`, and its type/JSDoc must warn that **setting it `true` means EVERY machine runs the job on every fire, untracked by the lease — so the job MUST be per-machine-idempotent and cheap** (a scan that only reads local disk + writes local scan-state qualifies; a job that mutates shared/canonical state, spends metered budget on the cadence, or spawns duplicated cross-machine side-effects must NOT set it). A regression test asserts a job **without** the flag keeps today's exact behavior (still yields on `hasRemoteClaim===true`), and a job **with** it does not consult the claim path (§Testing).

### §2.9 Presence / scope gate

The canonical registry (`scripts/model-registry-freshness.manifest.json`), the lint, and the deterministic prober (`scripts/doorway-scan.mjs`) are all **instar source** — present on an agent that carries the instar source tree (a maintainer/dev agent like Echo, and the fixture repos), NOT scaffolded into a pure end-user agent home (`instar init` installs `.claude/`, `.instar/`, CLAUDE.md, and built-in job templates — not `scripts/`). So, mirroring bench-refresh's harness-presence gate exactly:

- The scan job template **is** installed on every agent (it's a built-in job), but its first step is a **prober-presence gate** — `test -f scripts/doorway-scan.mjs || exit-silently`. On a non-instar-source agent the prober is absent, so the job **EXITS SILENTLY** (the honest no-op path; it does not message anyone). This is why the earlier round-1 phrasing "meaningful on any agent" was too loose — the useful scan only runs where the prober is present.
- Where the source tree IS present, the job writes its local scan-state, surfaces local changes, and — since the canonical manifest is also present there — the **maintainer-diff** portion of the attention body is always relevant (canonical fold-in matters on exactly the agents that can act on it).
- `GET /doorways` follows the same honesty: on an agent carrying the canonical manifest it returns 200 (never-run until a scan runs, then merged); on a non-instar-source agent (no manifest) it returns the D5 `503` rather than a fabricated empty. This is consistent — the doorway *knowledge registry* is an instar-source artifact, so an install that never had it has no registry to read.

**Per-machine scan-liveness — the accepted backstop (P18, Observation Needs Structure).** The spec's founding failure is "nothing *made* anyone re-check," so it must answer: what watches that the per-machine scan itself keeps *firing*? The breaker (§2.3) only fires when the scan *runs* and finds zero doors — it is silent if the scan silently *stops running* (disabled, or a silently-stopped job) on one machine, which would let that machine's live scan-state go stale unnoticed. **The accepted backstop is the freshness lint's independent `lastReviewedAt` staleness tooth (#1359):** it fails loud on canonical-pin rot *regardless of whether any scan runs*, so the *original incident* (a stale routing pin) cannot recur even if a machine's scan lapses. What a lapsed per-machine scan can still silently lose is only the *live/observed* half's currency on that one machine (not routing correctness). Making per-machine scan-liveness itself a monitored signal (e.g. a lightweight check that each machine's `lastScanAt` is within an expected multiple of the cadence, surfaced via the existing guard-posture inventory) is a **clean additive follow-up <!-- tracked: 29723 -->, explicitly scoped OUT of v1** — the canonical staleness tooth is the load-bearing anti-rot guarantee and it is independent of the scan.

## §3 — Component 3: The constitutional standard

Proposed entry for `docs/STANDARDS-REGISTRY.md` (the standard is *proposed* here; adding it to the live registry is the operator-ratified amendment step — this spec does not edit the registry). Format matches existing entries.

> ### Keep the Doorway/Model Map Current
> **Rule.** The set of *doorways* the agent can reach a model through — and the top model(s) (with exact ids) behind each — is knowledge that **rots**, so it must be kept current by a standing *process*, never by anyone remembering to re-check. A stale doorway/model map is a **defect**, not a chore that's overdue.
> **In practice.** One machine-readable registry is the single source of truth for what doors exist and what the top model behind each is (canonical/reviewed layer) plus what each machine last actually reached (live/scanned layer); the frontier set that gates routing pins is *derived* from that record, never re-typed. A recurring, dark-by-default, cost-aware scan job re-probes every doorway on a cadence (via a deterministic prober) and surfaces **only changes** — never noise, and never a silent loss *on the notification path* (each machine's finding is a distinct machine-qualified row that cannot silently coalesce away). Per-machine *scan-liveness* monitoring (that the scan keeps firing everywhere) is a named additive follow-up <!-- tracked: 29723 --> whose anti-rot backstop is the independent freshness lint, not this standard's guarantee (§2.9). A freshness lint fails loud when a pin ages out of its review window or drifts off the derived frontier set. Adding an LLM callsite or a new door without updating the registry is caught structurally, not by review vigilance.
> **Earned from.** The stale-pin incident (2026-07-03): Instar's cross-model reviewers and several routing pins were still pointing at `gemini-2.5-pro` weeks after the Gemini 3-class models shipped, silently degrading review quality. The freshness lint (#1359) could fail loud on staleness/drift, but it was a tripwire, not an engine — nothing re-probed the doorways or grew the map, so a human had to remember to go find the frontier and update the allowlist, and between manual passes the knowledge rotted invisibly until it degraded output. (Lived corroboration during this very spec's review: the cross-model external reviewer itself ran on `gemini-2.5-pro`, the exact stale pin.)
> **Traces to the goal.** A coherent agent routes each decision to the best model it can actually reach; a self-evolving agent's *map of what it can reach* must itself evolve, or its coherence quietly degrades as the world moves underneath it. This is *Structure beats Willpower* applied to the frontier: the map stays true because a job re-probes it and a ratchet gates it — not because someone remembers to look.
> **Applied through.** The recurring **doorway-scan job** (`src/scaffold/templates/jobs/instar/doorway-scan.md` + the deterministic `scripts/doorway-scan.mjs`) + the **freshness lint** (`scripts/lint-model-registry-freshness.mjs`, #1359) + the enriched **Doorway/Model Knowledge Registry** (`scripts/model-registry-freshness.manifest.json`) + the human routing narrative (`docs/LLM-ROUTING-REGISTRY.md`). **Parent principle:** *Structure beats Willpower*.

**Ratification precondition (do not let the standard outrun its teeth).** The standard says "a stale map is a defect," but the lint ships in `report` (non-gating) mode with known-stale `flaggedStale` rows. Ratifying the standard while the lint can't *fail* on staleness would be a paper rule. **Therefore rollout step 5 (add the standard) has an explicit companion precondition:** the current `flaggedStale` rows are operator-confirmed + reconciled into `topModels`, and the lint's staleness+drift teeth are flipped to `enforcement: "strict"` (gating). The standard is ratified *with* live teeth, not ahead of them.

## Decision points touched

1. **The doorway-scan job's surface-or-stay-silent decision** — the debounced diff-emptiness check (§2.3/§2.6) gates whether an attention item is raised. Deterministic; the tier-1 supervisor only sanity-checks the prober's output.
2. **The freshness lint's derived-frontier drift check** (§1.4) — TOOTH 2 now reads a *derived* set; non-gating under `report`, gating only when an operator flips `strict`. Backward-compatible for un-enriched manifests.
3. **The never-auto-apply boundary** (§2.7) — enforced by toolAllowlist (no Edit/Write tool) + a job-slug-scoped PreToolUse **command-shape allowlist** (not a new block on user action).
4. **New-doorway / metered-key auto-notice** (§2.5) — an advisory attention item; never acts on the key.
5. **New job-def flag `perMachineIndependent`** (§2.8, D11) — makes `JobScheduler` skip the claim seam for this job; defaults `false`; affects only jobs that opt in.
6. **`free-probes`-never-pays invariant** (§2.2, D8) — a structural gate in the prober; the default scan can never trigger a metered probe.
7. **Budget fail-closed invariant** (§2.2, D9) — a metered scope with an absent/zero/invalid `budgetCapUsd` refuses EVERY metered probe (never unlimited).
8. **Metered-scope manual-marker gate** (§2.2, D10) — `+liveness`/`+web-verify` require a manual env marker the scheduler never sets; the scheduled cadence physically cannot spend money.
9. **`GET /doorways` status contract** (D5) — exactly two states: 200 (canonical present; `scanState:"never-run"` until a scan runs) vs 503 (canonical registry genuinely unavailable); no "disabled" state, because the route ships always-mounted.
10. **Scan-state read-validation** (§1.3/§2.6) — the machine-local file is re-clamped field-by-field on every load; the diff and `GET /doorways` are built only from validated values.

No decision surface here blocks a *user* action, spends money without an explicit opt-in scope AND a positive budget AND a manual marker, or mutates instar source autonomously.

## Frontloaded Decisions

The whole feature **ships dark (`enabled: false`)** and the cost-bearing scopes are **opt-in, manual-only, and fail-closed on budget** (D9/D10), so cadence/channel/config choices are cheap-to-change-after. The money decision (D2) is scoped precisely below, with its fail-closed default (`budgetCapUsd: 0`, D6) and structural invariants (D8/D9) making zero-spend the only reachable default.

- **D1 — Scan cadence.** Default **weekly** (`0 4 * * 1`). Free-probes spends zero metered budget, so weekly is cheap. *Cheap-to-change-after:* job frontmatter; job is off by default. (Operator may prefer monthly to match bench-refresh — surfaced for confirmation, non-blocking.)
- **D2 — May the scan spend metered API $ to probe?** Default **no on the scheduled cadence** — only `free-probes` runs automatically; `+liveness`/`+web-verify` are opt-in, budget-capped, refuse-on-unknown-price, and **per-invocation manual**. *Cheap-to-change-after — SCOPED:* the cheap/reversible part is **leaving it off / invoking a metered scope manually per-run**. **Flipping the *scheduled* default to auto-run a metered scope is a NEW, non-cheap money decision requiring its own fresh operator sign-off at that time** — it is NOT covered by this tag (a "ships dark" label never makes recurring automatic real-dollar spend cheap). This build commits only to the safe default.
- **D3 — Alert channel.** Default the **attention queue** with a stable, machine-qualified id/`sourceContext` (§2.6). *Cheap-to-change-after:* a `maintenance.doorwayScan.digestTopicId` knob can redirect to a digest topic later (rope-health-digest pattern) without changing diff logic.
- **D4 — Initial `topModels` population.** Seed 1:1 from the already-reviewed `frontierAllowlist` values (`verifiedAt: "carried-over-from-allowlist"`, `frontier:true`); the first enabled scan re-verifies. **No new research/operator round-trip to land the PR.** `pricing` is **omitted/null on carry-over** (the allowlist carries no price, and pricing is consumed only by the dark, opt-in, manual metered scopes — a null price simply means a metered probe of that model refuses-on-unknown-price per D9 until an operator fills it in). **Carry-over seeds are also exempt from the §2.5 structured-`evidence` requirement** — their honest provenance IS the `verifiedAt:"carried-over-from-allowlist"` marker (the values were already operator-reviewed as the allowlist); the `evidence` block (§2.5) is required for POST-seed operator *promotions* going forward, not retroactively on the carry-over set. (The §1.4 lint validates pin-vs-derived-frontier consistency, not evidence-block presence, so this is a documentation/curation convention, not a gating check.) (§1.2)
- **D5 — Ship `GET /doorways` in v1, with ONE precise status contract.** **Yes — committed.** A read-only, Bearer-authed, **always-mounted** route (there is NO "enabled/disabled" flag for it) returning the **merged view**: canonical `topModels` (with pricing) overlaid with this machine's read-validated live `reachable`/`probeStatus`/`lastScannedAt`, keyed by `doorId`. **Exactly two states → two codes** (this replaces the earlier internally-inconsistent "200 vs 503 when absent/disabled" wording):
  - **(a) Canonical registry present, but no scan has run yet → `200`** with `topModels` populated from the canonical layer and a top-level `scanState: "never-run"`; each door's live fields are honestly `reachable: null` / `probeStatus: "never-scanned"` / `lastScannedAt: null`. This is **honest-empty, NOT 503** — it makes the ELI16 promise ("a real, fresh answer to read") true from day one, before the first scan.
  - **(b) The canonical registry is genuinely unavailable → `503`.** The only 503 case: the canonical manifest is **missing or corrupt/unparseable** — e.g. a **non-instar-source install** (a pure end-user agent scaffolded via `instar init` carries no `scripts/model-registry-freshness.manifest.json`), or a manifest that won't parse. There is no registry to serve, so the route says so honestly rather than fabricating an empty payload. The 503 body carries a **stable machine-readable `code`** — `registry-unavailable-no-instar-source` (manifest absent) or `registry-corrupt` (present but unparseable) — so a client distinguishes "this feature isn't present on this install" from a transient server failure (the external-reviewer concern that a bare 503 reads like server failure); 503 is kept (per the round-2 status-contract mandate) rather than 404/501, with the code carrying the precise meaning.
  - `401` without a Bearer token (unchanged). A Tier-2 test asserts **never-run → 200** (not 503); a Tier-2/Tier-3 test asserts **no-manifest → 503**. A route shape is a published interface (never "cheap" under the closed taxonomy), so it is committed now with its shape named.
- **D6 — Config knob (with a stated money default) + enable-surface precedence.** `maintenance.doorwayScan` = `{ enabled, scope, cadence, digestTopicId, budgetCapUsd }` — named definitively so nothing dangles. **Defaults:** `scope:"free-probes"`, `cadence:"0 4 * * 1"` (weekly), `digestTopicId:null` (→ the default alerts/hub attention surface, per P23 §2.6 — NOT a deferred <!-- tracked: 29723 --> choice), **`budgetCapUsd: 0`** — a fail-closed money default (zero metered spend until an operator explicitly sets a positive cap AND opts into a metered scope manually; see D9). **`enabled`: absent/undefined by default (NOT seeded `false`).** **Enable-surface precedence (no "flipped a flag, nothing ran" surprise):** the **job-manifest `enabled` flag (§2.1, seeded `false`) is authoritative for whether the scan RUNS** — the scheduler gates on it, exactly like bench-refresh — so the feature is off-by-default via the job manifest. The `maintenance.doorwayScan.enabled` config field is a **master kill-switch with deny-wins** semantics: the scan runs iff the job-manifest flag is enabled **AND `config.enabled !== false`**. **This is why the config field must default absent/undefined, not `false`** (round-2 fix): were it seeded `false`, `false !== false` would be `false` and the scan would NEVER run even after the operator enables the job manifest — the exact "flipped it on, nothing ran" bug. So: enabling ONLY the config flag does NOT start the scan (the job manifest is the enable); an *explicit* `config.enabled:false` reliably stops it even if the job is enabled; and an absent config field never blocks. `migrateConfig` seeds every field EXCEPT `enabled` (which stays absent unless the operator sets it).
- **D7 — "Critical door" for `high` priority.** Any door named in `sessions.componentFrameworks` resolution, OR the sole doorway for a routing category with no fallback.
- **D8 — `free-probes` never pays.** Structural invariant in the prober (§2.2) + a metered-only-door unit test asserting no charge under `free-probes`.
- **D9 — Metered spend is bounded by a hard token ceiling FIRST, budget-dollars second; fail-closed when the cap is absent/zero/invalid.** Two layered structural invariants in the prober, sibling to D8:
  - **Primary (price-independent) bound — a hard per-probe token/`max_tokens` ceiling.** Because this spec exists *because pricing rots*, the dollar estimate alone is not a trustworthy bound (a risen price could let a metered run silently exceed the intended dollars). So every metered `+liveness` completion is issued through `buildIntelligenceProvider` (§2.0) with a **hard `max_tokens`/completion ceiling** — a true structural cap that holds even if `pricing` is stale. This is the load-bearing spend bound.
  - **Secondary — dollar budget, fail-closed, and price-staleness-aware.** If a metered scope is requested while `budgetCapUsd` is absent, unset, `0`, negative, or non-finite, the prober **refuses EVERY metered probe** (records `probeStatus: not-probed-budget-refused`) — it **never** falls back to unlimited spend. A metered probe runs only when `budgetCapUsd > 0`, the per-probe price is **known AND fresh**, and the estimated run cost (token ceiling × the known price) stays under the cap. **Price staleness is treated exactly like an unknown price → refuse** (the whole spec exists because facts rot, so a stale price is untrusted): each `topModels[].pricing` carries its own `verifiedAt` (or reuses the model's `verifiedAt`), and a price older than a configured window (default aligned with the manifest's `stalenessWindowDays`) is refused just like a null/unknown price (per D2/§2.2 and the D4 carry-over-null-pricing behavior). The hard token ceiling (primary bound) holds regardless.
  - **`budgetCapUsd` is an ESTIMATE guard, not a billing guarantee** (it can lag currency / tiered pricing / cached-token discounts / provider markups) — which is exactly why the **hard per-probe token ceiling is the real, price-independent bound and applies to EVERY metered probe regardless of price**; the dollar cap is the secondary estimate layered on top.
  - Unit tests assert: `+liveness` with `budgetCapUsd` absent/`0` charges nothing and records `not-probed-budget-refused`; and the completion's token count never exceeds the hard ceiling regardless of the configured/stale price. (D8 covers the default scope; D9 covers "a metered scope was explicitly selected" — together they make it structurally impossible to spend without an explicit positive cap AND keep spend bounded even when the price is wrong.)
- **D10 — Metered scope is human-manual-only (the scheduled cadence cannot spend).** The scheduled job body invokes a **fixed literal `--scope free-probes`** (asserted by the template-invariant test, which also asserts the body contains no `+liveness`/`+web-verify`). Because the job session's Bash is LLM-controlled, this is backstopped structurally: `doorway-scan.mjs` **refuses a metered scope unless the manual marker env `INSTAR_DOORWAY_SCAN_MANUAL=1` is set**, which the scheduler never sets on a spawned job session — so a scheduled run physically cannot select a paid scope even if the session appends one. A TTY check is deliberately not used (a tmux-spawned job session owns a pty like a human terminal; the env-marker is the robust discriminator). A metered scope is reachable only by a human running the prober by hand (marker + positive `budgetCapUsd`). A unit test asserts a scheduler-env invocation (no marker) refuses `+liveness`.
- **D11 — New job-def flag `perMachineIndependent` (a durable, cross-cutting `JobScheduler` change).** Default **`false`**; the doorway-scan template sets it `true` deliberately (each machine must scan its own disk). When `true`, `JobScheduler` skips the claim seam (§2.8) so every machine runs the job. **Misuse warning (load-bearing, lives in the type/JSDoc):** setting this `true` means EVERY machine runs the job on every fire, untracked by the lease — so the job **MUST be per-machine-idempotent and cheap** (local-read + local-write only; never shared/canonical mutation, never metered spend on the cadence, never duplicated cross-machine side-effects). A regression test asserts a job **without** the flag keeps today's exact claim/lease behavior (still yields on a remote claim), and a job **with** it does not consult the claim path.

## Multi-machine posture (Cross-Machine Coherence)

- **Canonical registry** (`scripts/model-registry-freshness.manifest.json`) — **git-tracked source**, identical on every machine by construction. No replication concern.
- **Live scan-state** (`.instar/state/doorway-scan.json`) — **machine-local BY DESIGN**, because doorway reachability is a physical fact of *that machine's* disk. The default posture is `unified`; this surface claims the machine-local exception, so per Standard A (P21 — An Instar Agent Is Always a Multi-Machine Entity) it carries the required taxonomy-jailed justification marker (a bare "machine-local BY DESIGN" declaration is itself a material finding):

  `machine-local-justification: physical-credential-locality`

  The key is substantively correct: a door's reachability rests on **per-disk logins / keys / service-bindings** — the claude-code OAuth login, the codex ChatGPT login, the `metered_openrouter_bench` / `metered_gemini_bench` vault keys, and which CLIs are installed on *this* disk — the exact "a logged-in session lives in cookies on one disk" category the taxonomy names, and the identical deliberate posture of the Playwright Profile Registry (a machine-local resource registry). `hardware-bound-resource` would be WRONG (installed CLIs + network reachability are not bound to specific physical hardware), and `unified` would be *infeasible* (replicating one machine's disk-liveness onto another is actively false). A future *pool-scope read* (merge each machine's scan-state, tagged by machine) is a clean additive follow-up <!-- tracked: 29723 -->, explicitly **DECLINED here** (more conservative than the precedent, which surfaces nothing pool-scope at all).
- **Attention items** — use the existing attention queue; **machine-qualified `sourceContext`** (§2.6) prevents cross-machine coalesce/content-loss. A stable id makes a re-run update one row per machine.
- **Job execution** — **`perMachineIndependent: true`** (§2.8) so the scan runs on every machine, not just the lease-holder.
- **Maintainer diff** — surfaced only where the instar source tree is present (the maintainer/dev machine). No cross-machine action.

No feature here silently assumes a single machine; the one durable state surface is machine-local on purpose, with the reason stated, and both the notification and execution multi-machine hazards are addressed.

## Security posture

- **Deterministic prober owns the danger** (§2.0): all network I/O, secret handling, timeouts, size-caps, sanitization, and payload construction are in code, not a prompt.
- **Secrets:** key NAMES only; a value is used in-process in a request header, **never** in shelled-out argv/environ, never echoed/stored/surfaced (§2.4).
- **Untrusted remote data (write-time):** model ids charset-clamped (`[A-Za-z0-9._/:-]`), length/count-bounded; `probeStatus` is a fixed classified enum (no verbatim remote/error text); any remote free text reaching the supervisor is wrapped in an untrusted-data envelope; a model-list is never executed or interpolated into a shell command (§2.0/§2.6).
- **Untrusted-on-every-read (scan-state):** the machine-local scan-state file is re-validated field-by-field on every load through `loadScanState()` (§1.3) — write-time clamping is not enough because a local writer can plant well-formed poisoned values (out-of-enum `probeStatus`, injection-laden `topModels`, a forged `pendingFlip`) after the write. The diff and `GET /doorways` are built only from read-validated values, never raw pass-through; a surfaced flip requires a live corroborating probe this run, so a planted `pendingFlip` cannot fake the debounce. Honest scope: containment (no downstream injection/malformed payload), not tamper-proofing a plaintext file (same accepted at-rest posture as the other machine-local registries).
- **Response-size cap** (default 2 MB) before parsing, in addition to timeouts (§2.0).
- **No source mutation:** enforced by `toolAllowlist: ["Bash"]` (no Edit/Write tool) + a job-slug-scoped PreToolUse **command-shape allowlist with fully-specified match semantics** (§2.7): exact-executable+argv, whole-command-anchored, **no interpreters** (`python3 -c`/`node -e`/`sh -c` dropped — authToken read from `$INSTAR_AUTH_TOKEN` env), `curl` host-pinned to localhost with output-redirect flags forbidden, and **REFUSE on any shell compounding/redirection/substitution/env-prefix**. It **fails OPEN on scope resolution** (a guard bug never blocks unrelated sessions) but **fails CLOSED on command match within a confirmed doorway-scan session**. This — not SourceTreeGuard (which does not intercept a job's writes) and not a 3-idiom denylist (which misses `cp`/`dd`/interpreters/`git checkout`/`patch`/heredoc) — is the real enforcer. The prober is additionally write-restrained to scan-state only (§2.0).
- **Money is quadruple-gated:** the scheduled cadence runs a fixed `--scope free-probes`; the §2.7 guard refuses any env-prefix (`INSTAR_DOORWAY_SCAN_MANUAL=1 …`) or non-`free-probes` argv (so a prompt-injected session can't self-authorize spend); D8/D9 make metered spend structurally impossible without a positive `budgetCapUsd`; and the metered completion carries a hard `max_tokens` ceiling that bounds spend even if the price is stale. No spend without an explicit scope AND a manual-marker env (blocked in the scheduled session by the guard) AND a positive cap AND under the token ceiling.
- **Metered LLM call is token-attributed** (Token-Audit Completeness): the `+liveness` completion goes through `buildIntelligenceProvider` tagged `attribution.component:"doorway-scan"` (so it appears in `/metrics/features` + TokenLedger, is circuit-broken and spawn-capped), never a raw off-funnel fetch (§2.0). The free model-LIST GET is not a generation (zero tokens).
- **`/attention` + `/view` POSTs are performed IN-PROCESS by the prober** (§2.0), with proper JSON encoding and no shell interpolation of remote text — the session runs no data-carrying curl at all (which also resolves the round-2 contradiction: the tightened §2.7 refuses the `--data-binary @-` pipe such a curl would need). The body is **tone-gate-safe** (jargon-safe summary; raw diff in a private view) because `POST /attention` runs the body through `checkOutboundMessage`, which would otherwise block a raw path/config-key/CLI maintainer-diff → a silent loss (§2.6). **No localhost link** ever goes in the body: a `/view/` link only when a public tunnel URL exists, else a link-free summary + "read `GET /doorways`" (a localhost link is refused by a pre-existing guard → silent loss).
- **Private-view render is an enumerated injection sink:** all scan-state-derived free text is HTML/markdown-escaped at the read funnel (§1.3) before composing the private-view body, so a local-disk writer's planted `<script>`/`onerror` in `changeHistory[].change` cannot become stored XSS in the operator's (dashboard-PIN/token-scoped, possibly tunnel-shared) view.
- **Self-standing scheduled-spend refusal:** the prober refuses any metered scope when it detects a scheduled doorway-scan session (§2.0/D10), independent of the §2.7 guard — so "a scheduled run cannot spend" holds even though the §2.7 guard fails open on a scope-resolution error.
- **The §2.7 guard parses with a real shell grammar** (exactly one simple-command AST node, no expansion/operator/redirection), not a regex/byte-scan — the parse IS the boundary, and the sanctioned set is tiny now that POSTs are in-process.
- **No new network trust:** probes hit only endpoints already reachable via existing operator-provided credentials.
- **`GET /doorways`** requires the Bearer token (a `401`-without-Bearer test is required, §Testing) and follows the D5 two-state status contract (200 never-run vs 503 no-registry).

  - **changeHistory ring-buffer on write:** the prober's writer caps `changeHistory[]` to the most-recent N (default 20) and length-clamps `change`, so the durable record is bounded at write as well as read.
  - **`probeStatus` → `reachable` → diff mapping (round-2 P20/L5):** a `timeout`/`dns-fail`/`http-5xx` sets `reachable:null` and — even repeated across two scans — surfaces NO `wentUnreachable` (P20 unknown≠down); a `not-installed`/auth-`http-4xx` sets `reachable:false` and MAY surface; a `malformed-response`/`oversize-response` on a door that DID answer keeps `reachable:true` AND surfaces the distinct "parser needs attention" diff class (the L5 canary — never a silent map-non-update).
  - **Private-view XSS neutralization (round-2 adv#3):** a scan-state with a planted `changeHistory[].change` of `<img src=x onerror=alert(1)>` → the composed private-view body is HTML-escaped (the payload renders inert), asserted at the funnel; the private view is an enumerated containment sink.
  - **In-process POST + no session data-curl (round-2 adv#1):** the prober issues `POST /view` and `POST /attention` in-process (test the prober's own request path); the job body contains no `curl --data`/`--data-binary`/pipe (template-invariant assertion).
  - **§2.7 guard AST-parse (round-2 adv#5):** the guard accepts the sanctioned prober invocation with benign normalizations (`--scope "free-probes"`, extra whitespace) and the `/health` GET; it REFUSES anything that parses to more than one simple command or contains an expansion/operator/redirection node (`;`, `|`, `$(…)`, `>`, backticks, env-prefix); a regex/substring matcher is not an acceptable implementation (the test drives the parser contract).
  - **Self-standing scheduled-spend refusal (round-2 security):** with `INSTAR_JOB_SLUG=doorway-scan` set, the prober refuses `--scope +liveness` EVEN WITH `INSTAR_DOORWAY_SCAN_MANUAL=1` set and the §2.7 guard simulated failed-open — charges nothing; a non-doorway-scan-session invocation (manual path) with marker + positive cap is permitted.
  - **Sequential metered concurrency (round-2 scal#1):** no more than one metered completion is in flight at once during a `+liveness` run.
  - **Pricing staleness (round-2 codex):** a metered door whose `pricing.verifiedAt` is older than the staleness window is refused exactly like an unknown price (charges nothing), even under a positive `budgetCapUsd`.
  - **Delivery robustness — no silent notice loss (round-3/4 adv):** with a change detected, the scan-state write succeeds but the in-process `POST /attention` fails/is-held → `last-surfaced` does NOT advance and the change **re-surfaces on the next scan** (retry RECOMPUTES from observed-vs-last-surfaced, never replays; `lastDeliveryFailedAt` is set); a confirmed 2xx advances `last-surfaced` and clears the marker. The diff is computed against `last-surfaced`, never the always-advancing last-observed state.
  - **Initial-seed / no fresh-state flood (round-4 integration):** a fresh scan-state (`surfacedBaseline.at === null`) with a full observed door list surfaces **NOTHING** and seeds `surfacedBaseline := observed` (no delivery required) — a new pool machine / post-corrupt-reset does not emit a full-map notice. Distinct from the delivery-failure test.
  - **Flip-then-flip-back clears cleanly (round-4 adv):** a door that flips, fails delivery, then flips BACK before the next cadence → the recomputed diff is empty → NO false "went unreachable" notice AND `lastDeliveryFailedAt` clears (no wedge).
  - **View/link handling (round-3 sec/int):** the private view is upserted via `PUT /view/<stable-per-machine-id>` (no orphan `POST /view` per run); the emitted body carries a token-free bare `/view/:id` link only when a tunnel exists, never a token-bearing tunnel URL, and a link-free summary otherwise.

The sanctioned shapes (prober with exact `--scope free-probes`, host-pinned redirect-free `/attention` + `/view` + `/health` curls, `test -f`/`jq -r` reads) asserted **PERMITTED**. Fail-mode tests: an unresolvable/errored slug resolution → guard **fails OPEN** (allows — never blocks an unrelated session); a confirmed doorway-scan session with an undecomposable command → guard **fails CLOSED** (refuses). ## Migration Parity

- **New built-in job** (`doorway-scan.md` + schedule JSON) → **no bespoke migration needed**: `InstallBuiltinJobs` (invoked from the update path, non-destructive install-if-missing — the same mechanism that shipped `bench-refresh`) installs it on every update. New agents get it via `init`; existing agents via update. A template-invariant test pins the safety frontmatter (as `bench-refresh-job-template.test.ts` does).
- **New deterministic script** (`scripts/doorway-scan.mjs`) + **lint change** → ship as source; every source-carrying agent gets them on update. The derived-frontier lint change is backward-compatible (un-enriched manifest → today's behavior). (On a pure end-user agent these are absent → the prober-presence gate makes the job a silent no-op, §2.9.)
- **New PreToolUse guard hook** (the §2.7 command-shape allowlist) → a **built-in `instar/` hook**, installed via `migrateHooks()` on the **always-overwrite** path (never install-if-missing — the `hook-event-reporter.js` lesson: a stale broken hook must be replaceable). It is **registered fleet-wide** (runs on every Bash call in every session) but is scoped by `INSTAR_JOB_SLUG`/`active-job.json` to activate only for the `doorway-scan` session, and it **FAILS OPEN on its own error / an unresolvable slug** (any exception in scope resolution or guard logic → ALLOW) — exactly like the sibling `pr-hand-lease-guard.js` — so a guard bug can never block Bash in an unrelated instar-dev/interactive session. Its settings wiring rides `migrateSettings()`. Idempotent: re-running the migration re-writes the same hook. New agents get it via `init`; existing agents via update. A hook-migration regression test asserts it installs, is scoped to the doorway-scan slug (a strict no-op elsewhere), and fails open on a resolution error.
- **New state file** (`.instar/state/doorway-scan.json`) → created lazily on first scan; corrupt/old-schema handling (§1.3) means an update can never brick on a stale/corrupt state file (log-once-and-fresh; additive backfill — matches `PreferencesManager`/`TopicIntent`). **Deliberately EXCLUDED from the backup manifest.** `BackupManager` selects files via an explicit `includeFiles` allowlist (not a glob over `.instar/state/`), so the scan-state is correctly not backed up by default — the right posture, since it is machine-local + regenerable and a fresh state surfaces nothing until it has a baseline. This MUST stay so: it must NEVER be added to `includeFiles`, because restoring one machine's disk-liveness onto a *different* machine is precisely the replication §Multi-machine calls "actively false." It belongs in the same excluded per-machine-runtime-state category as `state/pr-hand-leases.json` / `state/pending-inbound.*` — noted here so the exclusion is structural intent, not an accident a future maintainer "helpfully" reverses.
- **Enriched manifest** → ships as source; `topModels` seeded from the reviewed allowlist (D4).
- **CLAUDE.md template** → `migrateClaudeMd()` with a content-sniff guard adds the awareness block so existing agents learn the registry + job exist.
- **Config knob** (`maintenance.doorwayScan`) → `migrateConfig()` existence-checked additions only, seeding the fail-closed defaults (`scope:"free-probes"`, `budgetCapUsd:0`, `digestTopicId:null`, D6). **It seeds every field EXCEPT `enabled`** — `enabled` stays absent/undefined (NEVER seeded `false`), because the D6 deny-wins predicate (`config.enabled !== false`) would treat a seeded `false` as a permanent block and the scan would never run even after the job manifest is enabled (the round-2/round-5 bug — the feature is off-by-default via the *job-manifest* `enabled:false`, not the config field). Only missing fields are added; an operator's existing values are never overwritten. The Tier-2 enable-precedence test guards this.

## Maturation path / dark-gate exception (documented)

- The **`GET /doorways` route handler** is compiled into the server `dist`, so it is **always mounted on every agent**; it returns 200 honest-empty (`scanState:"never-run"`) where the canonical manifest is present, and the D5 `503` on a non-instar-source install (no manifest to serve). The **instar-source pieces** (enriched manifest, derived-frontier lint in `report` mode, the deterministic prober script) live on **source-carrying agents** — inert-but-live there (the lint runs in non-gating report mode; the prober is dormant until the job is enabled). They are simply absent on a pure end-user agent, which the route and the job's prober-presence gate both handle honestly (§2.9).
- The **scan job itself** ships `enabled:false` for **all** agents (including developer agents). This is a **documented, structurally-registered exception** to the "every feature ships enabled on developer agents" maturation-path standard — recorded the **same way bench-refresh is** (the standard's own `DARK_GATE_EXCLUSIONS` allowlist / the maturation-path checker's cadenced-job carve-out), so the exception is a named entry the conformance checker can see, not a prose claim. The **justification is identical to bench-refresh's**: a cadenced job that spawns a session on a schedule (and can opt into metered probes) is a deliberate opt-in by norm, not a silent default — an always-on scheduled session-spawner would itself be the anti-pattern. The maturation is: land the machinery live → operator enables the job on the maintainer agent (free-probes) → soak → optionally opt into metered scopes → flip the lint to `strict` + ratify the standard (rollout step 5). (Adding the `doorway-scan` slug to that exclusion allowlist alongside `bench-refresh` is part of the build; the conformance gate's Maturation-Path flag closes once it is listed there.)

## Alternatives considered

- **Auto-PR instead of the attention queue** (external reviewer) — rejected for v1; see §2.7 (signal-only / no source-write authority / no git credential for the job). A reasonable post-soak enhancement.
- **A consistency *check* (allowlist ⊆ topModels) instead of derivation** — rejected; a check leaves a second hand-maintained list that re-rots. Derivation (§1.4) is the actual one-source-of-truth mechanism.
- **Shipping `topModels: []` empty** — rejected; hollow at launch. Seeded from the reviewed allowlist instead (D4).
- **Acknowledged cost:** probe parsers are inherently brittle (a CLI changing its `--version` format, an API changing its response schema breaks a probe). The design is *fail-safe per-probe* (a broken probe degrades that door, never the scan) and the prober applies response schema validation to fail fast with a classified `probeStatus` rather than a cryptic crash — but the standing maintenance cost of fixing probe parsers as upstreams drift is real and accepted, not "fire-and-forget." **Crucially, that drift is SURFACED, not silently absorbed** (§1.3/§2.6, L5): a door that answers but no longer parses raises the "probe parser needs attention" diff class, so the operator learns maintenance is due rather than the map silently ceasing to grow — an accepted maintenance cost that notifies is not silent rot.
- **Acknowledged cost — the §2.7 shell-matcher:** putting a security/spend boundary behind parsing a shell command is inherently fragile at the edges. The design mitigates this two ways: it *fails closed on match* (an edge case an over-strict parser rejects breaks the scan loudly/safely, never widens scope) and it *fails open on scope* (a parser bug can't block unrelated sessions), and it mandates a real shell-grammar AST parse (not a regex) over a tiny sanctioned set (just the prober invocation + `/health` GET + reads, since POSTs moved in-process). A fully brittleness-proof alternative — a dedicated non-shell runner entrypoint the scheduler invokes directly (no Bash surface at all) — is noted as a clean future hardening (Deferred <!-- tracked: 29723 --> #5), out of scope for v1. **Why v1 keeps the supervised Bash-job shape (answering the external reviewer's "why not a scheduler-native fixed-argv invocation now?"):** this is a `prompt`-type, tier-1-**supervised** job by deliberate design, mirroring bench-refresh — an LLM session runs the deterministic prober AND sanity-checks its output before anything surfaces (the tier-1 wrap the maturation standard wants for a cadenced job). A pure `script`-type scheduler-native invocation would run the prober fine but would DROP that tier-1 supervision layer. So v1 keeps the supervised job (hence a Bash surface, hence the guard), and the job body's only meaningful command is ALREADY the single fixed wrapper invocation (`node scripts/doorway-scan.mjs --scope free-probes`) — the external reviewer's "at minimum a single fixed wrapper" bar is met today. The fully-non-shell runner is the deferred stronger form that would re-house the supervision differently.
