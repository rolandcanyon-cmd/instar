# Correction Class Review + Verify Before Done — ELI16

Two different kinds of promises were falling through cracks.

First, when a person corrected an agent, Instar saved the correction but waited for repeated occurrences before deciding whether to improve a standard or the development process. That is useful for noisy preference learning, but wrong for direct feedback: one important correction could remain an orphan forever. The new path creates a durable review receipt immediately. A bounded intelligence pass asks two separate questions—“is a standard missing?” and “what process allowed this class?”—then records each arm until it is explicitly shipped, ratified, rejected, or marked as needing follow-up. The model can suggest; it cannot ratify its own rules.

Second, an agent could say “I pushed it” or “it is live” without checking the evidence produced in that turn. Claude Code now builds a tiny structural receipt from only the tail of its local transcript: which safe kind of action ran, a bounded non-secret target when available, and whether the tool succeeded. It never sends the command, tool output, transcript path, or raw inputs. Instar compares completion claims with that receipt and records an advisory signal.

Neither mechanism is a new blocking gate in this release. Development agents start in dry-run; the fleet remains dark. Corrections still use their existing recurrence pipeline, and future-action promises still use the existing Action-Claim sentinel. This release adds the missing durable review and evidence surfaces so their accuracy and operational cost can be measured before any stronger posture is considered.
